# 110+ Strings, Two Cache Zones, One System Prompt

> `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` is a literal string that splits the prompt into a globally-cached prefix and a per-session suffix. Sections registered with `DANGEROUS_uncachedSystemPromptSection` break the cache on every value change -- the `DANGEROUS_` prefix exists so developers feel the cost before they pay it.

## Overview

The system prompt Claude Code sends to the model is not a static document. It is assembled at runtime from over 110 individual strings produced by roughly 20 section functions, merged in a specific order, split by a cache boundary marker, and filtered for nulls. The final result is an array of strings -- not one big concatenated blob -- that the API receives as separate system message blocks.

The sophistication is in two mechanisms most agent builders never implement:

1. **A two-zone cache architecture** that makes the first ~40,000 tokens of every request free (served from Anthropic's prompt cache at 90% discount) while allowing the remaining ~5,000-15,000 tokens to change freely per session.

2. **A section registration system** with two functions -- one safe, one intentionally scary -- that forces developers to declare whether their section can be cached or whether it must break the cache on every change.

This chapter dissects `systemPromptSections.ts` (the registration and caching layer) and `prompts.ts` (the assembly pipeline) to show exactly how 110+ strings become the instruction document the model follows.

## Two Registration Functions

Every section of the system prompt is registered through one of exactly two functions. This is the core design decision of the entire system:

```typescript
// systemPromptSections.ts

// Safe: cached across turns until /clear or /compact
function systemPromptSection(
  name: string,
  compute: () => Promise<string | null> | string | null,
): SystemPromptSection {
  return { name, compute, cacheBreak: false }
}

// Volatile: breaks prompt cache on every value change
function DANGEROUS_uncachedSystemPromptSection(
  name: string,
  compute: () => Promise<string | null> | string | null,
  reason: string,
): SystemPromptSection {
  return { name, compute, cacheBreak: true }
}
```

The `DANGEROUS_` prefix is not a joke. It is a forcing function -- a deliberate API design choice that makes developers stop and justify why their section cannot be cached. The `reason` parameter is mandatory: you must explain in a string literal why this section needs to break the cache. Real examples from the codebase:

```typescript
DANGEROUS_uncachedSystemPromptSection(
  'mcp_instructions',
  () => isMcpInstructionsDeltaEnabled() ? null : getMcpInstructionsSection(),
  'MCP servers connect/disconnect between turns',
)
```

The reason string is not logged or displayed -- it exists purely as documentation at the call site. But requiring it changes developer behavior. Without it, every new section would default to uncached because "it might change." With it, developers must articulate *why* it changes, which often reveals that it does not.

## Section Resolution with Caching

When the prompt is assembled, each section goes through a resolution step that checks a name-keyed cache:

```typescript
// systemPromptSections.ts (lines 42-58)
export async function resolveSystemPromptSections(
  sections: SystemPromptSection[],
): Promise<(string | null)[]> {
  const cache = getSystemPromptSectionCache()
  return Promise.all(
    sections.map(async (s) => {
      if (!s.cacheBreak && cache.has(s.name)) {
        return cache.get(s.name) ?? null
      }
      const value = await s.compute()
      setSystemPromptSectionCacheEntry(s.name, value)
      return value
    }),
  )
}
```

Three things to notice:

1. **Sections resolve in parallel** via `Promise.all`. The compute functions are independent -- memory loading does not block environment info, which does not block MCP instructions.

2. **Cache lookup is by name, not by content hash**. A section named `'memory'` returns its cached value until the cache is explicitly cleared. This means the cache is a session-level memoization, not a content-addressable store.

3. **`cacheBreak: true` sections skip the cache entirely**. They always call `compute()`, and they always store the result (so the next non-cacheBreak section that happens to share a name would get the latest value -- though in practice names are unique).

## Cache Invalidation

The cache is cleared in exactly two places:

```typescript
// Called by /clear and /compact commands
export function clearSystemPromptSections(): void {
  systemPromptSectionCache.clear()
  // Also resets beta header latches
}
```

This means the section cache lives for the entire conversation unless the user explicitly runs `/clear` or `/compact`. A CLAUDE.md file edited mid-conversation will not be re-read until one of these commands fires. This is intentional: re-reading all CLAUDE.md files on every turn would add disk I/O to every request.

## The Static Sections (Cached, Stable)

The first half of the prompt -- everything before the boundary -- consists of sections that rarely change within a session. These are registered with `systemPromptSection` (no `DANGEROUS_` prefix):

```
Section Name              Source Function                What It Contains
─────────────────────────────────────────────────────────────────────────
intro                     getSimpleIntroSection()        "You are Claude Code..."
                                                         Core identity paragraph

system                    getSimpleSystemSection()       Tool execution rules,
                                                         output format, hooks behavior

doing_tasks               getSimpleDoingTasksSection()   Code style guidance,
                                                         task completion patterns

actions                   getActionsSection()            Reversibility assessment,
                                                         risk classification rules

using_tools               getUsingYourToolsSection()     When to use which tool,
                                                         tool selection heuristics

tone_and_style            getSimpleToneAndStyleSection()  No emojis, no filler,
                                                         reference style rules
```

These six sections form the identity core. They define *what Claude Code is* and *how it behaves*. Because they are compiled into the binary and do not depend on any runtime state, they are identical across every session, every user, every project. This is what makes them perfect candidates for prompt caching -- Anthropic's cache operates on prefix matching, and these sections form a stable prefix.

The cache scope for everything above the boundary is `'global'` -- meaning it can be shared across organizations, not just across requests from the same user. A hash of the static content determines cache hits. If even one character changes (say, a typo fix in the intro section), every user's cache misses until the new version warms up.

## The Dynamic Boundary

The boundary itself is a constant string:

```typescript
const SYSTEM_PROMPT_DYNAMIC_BOUNDARY = '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'
```

It appears as a literal element in the assembled array. The API layer (or the prompt cache implementation) uses this marker to know where the cacheable prefix ends. Everything before it can match a globally-cached prefix. Everything after it is session-specific and processed fresh.

```
┌─────────────────────────────────────────────────────────┐
│  intro                    "You are Claude Code..."      │
│  system                   Tool rules, output format     │
│  doing_tasks              Code style, task guidance     │  scope: 'global'
│  actions                  Risk assessment rules         │  (cross-org cache)
│  using_tools              Tool selection heuristics     │
│  tone_and_style           Style rules                   │
│                                                         │
├── '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__' ─────────────────┤
│                                                         │
│  session_guidance         Tools + skills available now   │
│  memory                   ~/.claude/CLAUDE.md content   │
│  mcp_instructions  [!]    MCP server instructions       │  scope: per-session
│  claude_md                Project CLAUDE.md files        │  (recomputed each turn
│  environment              OS, shell, cwd, git status    │   for [!] sections,
│  active_context           Currently open files          │   cached for others)
│  deferred_tools           Tool names without schemas    │
│  ... (12-15 sections)                                   │
└─────────────────────────────────────────────────────────┘
```

## The Dynamic Sections (Per-Session)

Below the boundary, sections vary by project, by session, and sometimes by turn. Here the `DANGEROUS_` function earns its name:

```typescript
// Cached per-session (safe -- changes rarely within a conversation)
systemPromptSection('session_guidance', () =>
  getSessionSpecificGuidanceSection(enabledTools, skillToolCommands),
)

systemPromptSection('memory', () => loadMemoryPrompt())

// DANGEROUS -- recomputed every turn because MCP servers can
// connect or disconnect between any two messages
DANGEROUS_uncachedSystemPromptSection(
  'mcp_instructions',
  () => isMcpInstructionsDeltaEnabled() ? null : getMcpInstructionsSection(),
  'MCP servers connect/disconnect between turns',
)
```

The `session_guidance` section is a good example of the caching tradeoff. It depends on which tools are enabled and which skills are available -- both determined at session start and stable thereafter. So it uses the safe registration function: compute once, cache for the session.

The `mcp_instructions` section is different. MCP servers can connect or disconnect at any time (the user might start a new server in another terminal). If this section were cached, the model would not know about a newly available server until `/clear`. So it pays the cache-break cost on every turn.

## Deferred Tools -- Keeping Under 200K Tokens

The system prompt has a hard budget. With 40+ tools, each carrying a full JSON schema, the tool descriptions alone can consume 50,000+ tokens. Claude Code solves this with tool deferral:

```
Full tool registration (in system prompt):
  Read, Write, Edit, Bash, Glob, Grep, Skill  -- 7 core tools
  Each includes: name, description, full parameter schema

Deferred tool registration (in system prompt):
  NotebookEdit, WebFetch, WebSearch, CronCreate, ...  -- 18+ tools
  Each includes: name only (no schema, no description)
```

The `ToolSearchTool` (itself a fully-registered tool) acts as the schema loader. When the model needs a deferred tool, it calls ToolSearch first:

```
Model wants to use WebSearch:
  1. Sees "WebSearch" in deferred tool list (name only)
  2. Calls ToolSearchTool with query "select:WebSearch"
  3. Receives full schema definition
  4. Now can call WebSearch with proper parameters
```

This is a two-step invocation pattern that trades one extra API round-trip for ~30,000 fewer tokens in every system prompt. At scale, this saves far more than it costs.

## The Assembly Pipeline

The final assembly in `prompts.ts` follows this structure:

```typescript
// prompts.ts (lines 444-577, simplified)
async function assembleSystemPrompt(session: Session): Promise<(string | null)[]> {
  const staticSections = [
    systemPromptSection('intro', getSimpleIntroSection),
    systemPromptSection('system', getSimpleSystemSection),
    systemPromptSection('doing_tasks', getSimpleDoingTasksSection),
    systemPromptSection('actions', getActionsSection),
    systemPromptSection('using_tools', getUsingYourToolsSection),
    systemPromptSection('tone_and_style', getSimpleToneAndStyleSection),
    // ... possibly more
  ]

  const dynamicSections = [
    systemPromptSection('session_guidance', ...),
    systemPromptSection('memory', ...),
    DANGEROUS_uncachedSystemPromptSection('mcp_instructions', ..., '...'),
    // ... 12-15 more sections
  ]

  const resolvedStatic = await resolveSystemPromptSections(staticSections)
  const resolvedDynamic = await resolveSystemPromptSections(dynamicSections)

  return [
    ...resolvedStatic,
    SYSTEM_PROMPT_DYNAMIC_BOUNDARY,
    ...resolvedDynamic,
  ].filter(s => s !== null)
}
```

The return type is an array of strings, not a single concatenated string. The API layer handles joining them. This matters because Anthropic's API accepts system prompts as arrays of content blocks, and keeping them separate preserves the cache boundary semantics.

The `filter(s => s !== null)` at the end is important: any section whose `compute()` returns `null` is silently omitted. This is how conditional sections work -- if no MCP servers are connected, `getMcpInstructionsSection()` returns `null` and the MCP block simply does not appear in the prompt.

## Proactive/KAIROS Mode -- A Different Prompt Entirely

When Claude Code operates in proactive mode (also called KAIROS), the entire assembly pipeline is replaced:

```
Normal mode:
  20 section functions → 110+ strings → full assistant identity

Proactive/KAIROS mode:
  Lean autonomous agent prompt → minimal tool set → goal-directed identity
```

This is not a flag that tweaks a few sections. It is an entirely separate prompt assembly path that produces a fundamentally different agent identity. The normal prompt says "you are an assistant that helps developers." The KAIROS prompt says something closer to "you are an autonomous agent pursuing a specific goal." Different tools are enabled, different behavioral rules apply, and the cache boundary may be positioned differently.

## Cost Arithmetic

To understand why this architecture matters, consider the token economics:

```
Prompt cache pricing (Anthropic, approximate):
  Uncached input:    $15.00 / 1M tokens  (Opus)
  Cached input:       $1.50 / 1M tokens  (90% discount)

Typical system prompt:
  Static sections:   ~40,000 tokens
  Dynamic sections:  ~10,000 tokens
  Total:             ~50,000 tokens

Cost per request (cold, no cache):
  50,000 tokens x $15.00/1M = $0.75

Cost per request (warm, static cached):
  40,000 tokens x $1.50/1M  = $0.06  (cached)
  10,000 tokens x $15.00/1M = $0.15  (fresh)
  Total:                       $0.21

Savings per request: $0.54
Over 100 requests/day: $54/day saved
```

The boundary marker is not a cosmetic separator. It is a $54/day architectural decision.

## Section Lifecycle

```
Session starts
    │
    ├─ Static sections computed, cached globally
    ├─ Dynamic sections computed, cached per-session
    │
    ▼
Turn 1: all sections resolved (cache cold for dynamic)
    │
    ▼
Turn 2: cached sections return instantly
         DANGEROUS_ sections recompute
    │
    ▼
Turn N: same pattern continues
    │
    ▼
User runs /clear or /compact
    │
    ├─ clearSystemPromptSections() called
    ├─ All caches flushed
    ├─ Beta header latches reset
    │
    ▼
Next turn: everything recomputes fresh
```

## Key Takeaways

- The system prompt is an array of 110+ strings assembled from ~20 section functions, split by `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` into a globally-cached prefix and a per-session suffix. The boundary is a literal string: `'__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'`.

- Two registration functions enforce cache discipline: `systemPromptSection()` (safe, cached) and `DANGEROUS_uncachedSystemPromptSection()` (volatile, recomputes every turn). The `DANGEROUS_` prefix and mandatory `reason` parameter are intentional API friction that prevents careless cache-breaking.

- Section resolution runs in parallel via `Promise.all`, with a name-keyed cache that persists until `/clear` or `/compact`. Non-cacheBreak sections compute once and return cached values for the rest of the conversation.

- Six static sections (intro, system, doing_tasks, actions, using_tools, tone_and_style) form the stable identity core. They use `scope: 'global'` caching -- shared across organizations, invalidated only when source code changes.

- Dynamic sections (session_guidance, memory, mcp_instructions, claude_md, environment, deferred_tools, and others) live below the boundary. Most are cached per-session; only `DANGEROUS_` sections recompute per-turn.

- Tool deferral keeps the prompt under budget: 7 core tools include full schemas, 18+ tools include only their names. The `ToolSearchTool` loads full schemas on demand, trading one extra round-trip for ~30,000 fewer tokens per request.

- Proactive/KAIROS mode bypasses the normal assembly entirely, using a lean autonomous agent prompt with different behavioral rules and a different tool set.

- The cache boundary is not an abstraction -- it is a direct cost optimization. With 40,000 tokens cached at 90% discount, each request saves roughly $0.54 compared to uncached processing.
