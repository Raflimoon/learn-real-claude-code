# One Claude Commanding Many

> The coordinator directs workers via system prompt injection, worker results are injected as `<task-notification>` XML in user-role messages, and SendMessageTool uses file mailboxes for inter-agent communication.

## Overview

Claude Code's multi-agent mode is not an abstract orchestration framework -- it is a combination of three concrete mechanisms:

1. **Coordinator Mode** -- Activated via an environment variable, replacing the main Claude's system prompt with a commander role.
2. **AgentTool / Fork Subagent** -- Spawns child agent processes that can inherit the parent's full conversation context.
3. **SendMessageTool + Mailbox** -- File-based IPC using JSONL inboxes for inter-agent messaging.

The entire system has no dependency on LangGraph, CrewAI, or any orchestration framework. Let's look at how it works through the source code.

## Coordinator Mode: Environment-Variable-Driven Role Switching

Coordinator mode is not a code path branch -- it is a simple environment variable check:

```typescript
// coordinator/coordinatorMode.ts:36
export function isCoordinatorMode(): boolean {
  if (feature('COORDINATOR_MODE')) {
    return isEnvTruthy(process.env.CLAUDE_CODE_COORDINATOR_MODE)
  }
  return false
}
```

Once activated, Claude's system prompt is completely replaced. The source code defines the coordinator's full behavioral specification in a 370-line prompt string:

```typescript
// coordinator/coordinatorMode.ts:111
export function getCoordinatorSystemPrompt(): string {
  return `You are Claude Code, an AI assistant that orchestrates
software engineering tasks across multiple workers.

## 1. Your Role
You are a **coordinator**. Your job is to:
- Help the user achieve their goal
- Direct workers to research, implement and verify code changes
- Synthesize results and communicate with the user
- Answer questions directly when possible

Every message you send is to the user. Worker results and system
notifications are internal signals, not conversation partners --
never thank or acknowledge them.

## 2. Your Tools
- **${AGENT_TOOL_NAME}** - Spawn a new worker
- **${SEND_MESSAGE_TOOL_NAME}** - Continue an existing worker
- **${TASK_STOP_TOOL_NAME}** - Stop a running worker

...`
}
```

Note a key design decision in the system prompt: worker results arrive at the coordinator as **user-role messages** formatted as `<task-notification>` XML. These are not system messages but user messages -- so the coordinator must learn to distinguish between user messages and worker notifications.

## Worker Tool Visibility: Dynamic Injection

The coordinator also tells the model which tools workers can use, including MCP servers:

```typescript
// coordinator/coordinatorMode.ts:80
export function getCoordinatorUserContext(
  mcpClients: ReadonlyArray<{ name: string }>,
  scratchpadDir?: string,
): { [k: string]: string } {
  if (!isCoordinatorMode()) return {}

  const workerTools = isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE)
    ? [BASH_TOOL_NAME, FILE_READ_TOOL_NAME, FILE_EDIT_TOOL_NAME]
        .sort().join(', ')
    : Array.from(ASYNC_AGENT_ALLOWED_TOOLS)
        .filter(name => !INTERNAL_WORKER_TOOLS.has(name))
        .sort().join(', ')

  let content = `Workers spawned via the ${AGENT_TOOL_NAME} tool ` +
    `have access to these tools: ${workerTools}`

  if (mcpClients.length > 0) {
    const serverNames = mcpClients.map(c => c.name).join(', ')
    content += `\n\nWorkers also have access to MCP tools from ` +
      `connected MCP servers: ${serverNames}`
  }

  if (scratchpadDir && isScratchpadGateEnabled()) {
    content += `\n\nScratchpad directory: ${scratchpadDir}\n` +
      `Workers can read and write here without permission prompts.`
  }

  return { workerToolsContext: content }
}
```

`INTERNAL_WORKER_TOOLS` are filtered out -- workers cannot see coordination-layer-only tools like `TeamCreate`, `TeamDelete`, `SendMessage`, and `SyntheticOutput`.

## Fork Subagent: The Key to Prompt Cache Sharing

Fork subagent is a special agent launch method: the child agent inherits the parent's full conversation history and system prompt. This is not an optimization -- it is the engineering foundation for prompt cache sharing:

```typescript
// tools/AgentTool/forkSubagent.ts:60
export const FORK_AGENT = {
  agentType: FORK_SUBAGENT_TYPE,
  tools: ['*'],
  maxTurns: 200,
  model: 'inherit',          // Inherit parent's model
  permissionMode: 'bubble',  // Permission requests bubble up to parent terminal
  source: 'built-in',
  getSystemPrompt: () => '',  // Not used -- fork passes parent's rendered bytes directly
} satisfies BuiltInAgentDefinition
```

Why does `getSystemPrompt` return an empty string? The comment explains:

```typescript
// tools/AgentTool/forkSubagent.ts:56 (comment)
// The fork path passes override.systemPrompt with the parent's
// already-rendered system prompt bytes, threaded via
// toolUseContext.renderedSystemPrompt. Reconstructing by re-calling
// getSystemPrompt() can diverge (GrowthBook cold->warm) and bust
// the prompt cache; threading the rendered bytes is byte-exact.
```

This is the core trick for prompt cache sharing: instead of re-rendering the system prompt (which might diverge due to feature flag state changes), the parent's already-rendered **raw bytes** are passed directly. Byte-level identity is what guarantees a cache hit.

## Fork Message Construction: Unified Placeholders

To maximize cache hit rates, all fork children's message prefixes must be byte-identical:

```typescript

const FORK_PLACEHOLDER_RESULT = 'Fork started -- processing in background'

// tools/AgentTool/forkSubagent.ts:107
export function buildForkedMessages(
  directive: string,
  assistantMessage: AssistantMessage,
): MessageType[] {
  // ...
  // Build tool_result blocks for every tool_use, all with identical
  // placeholder text
  const toolResultBlocks = toolUseBlocks.map(block => ({
    type: 'tool_result' as const,
    tool_use_id: block.id,
    content: [{ type: 'text' as const, text: FORK_PLACEHOLDER_RESULT }],

  // Only the final text block differs per child, maximizing cache hits

```

The structure is: `[...shared history, assistant(all tool_use), user(unified placeholders..., unique directive)]`. Only the final text block differs because of the directive.

## Fork Child Directive Injection

Each fork child receives a strict behavioral specification:

```typescript
// tools/AgentTool/forkSubagent.ts:172
export function buildChildMessage(directive: string): string {
  return `<${FORK_BOILERPLATE_TAG}>
STOP. READ THIS FIRST.

You are a forked worker process. You are NOT the main agent.

RULES (non-negotiable):
1. Your system prompt says "default to forking." IGNORE IT -- that's
   for the parent. You ARE the fork. Do NOT spawn sub-agents.
2. Do NOT converse, ask questions, or suggest next steps
3. USE your tools directly: Bash, Read, Write, etc.
4. If you modify files, commit your changes before reporting.
5. Keep your report under 500 words.
9. Your response MUST begin with "Scope:". No preamble.

Output format:
  Scope: <echo back your assigned scope>
  Result: <key findings>
  Key files: <relevant file paths>
  Files changed: <list with commit hash>
  Issues: <list if any>
</${FORK_BOILERPLATE_TAG}>

${FORK_DIRECTIVE_PREFIX}${directive}`
}
```

Rule 1 is an anti-recursion guard: the fork child's system prompt still contains the "default to forking" instruction (since it inherits the parent's full prompt), so it must be explicitly overridden.

Anti-recursion is also enforced at the code level:

```typescript
// tools/AgentTool/forkSubagent.ts:78
export function isInForkChild(messages: MessageType[]): boolean {
  return messages.some(m => {
    if (m.type !== 'user') return false
    const content = m.message.content
    if (!Array.isArray(content)) return false
    return content.some(
      block => block.type === 'text' &&
        block.text.includes(`<${FORK_BOILERPLATE_TAG}>`),
    )
  })
}
```

It checks the conversation history for the fork boilerplate tag -- if present, further forking is rejected.

## SendMessageTool + Mailbox: File-Based IPC

Inter-agent communication does not go through memory but through the filesystem. Each teammate has a JSON inbox:

```typescript
// utils/teammateMailbox.ts:1 (module doc)
/**
 * Teammate Mailbox - File-based messaging system for agent swarms
 *
 * Each teammate has an inbox file at
 * .claude/teams/{team_name}/inboxes/{agent_name}.json
 */

// utils/teammateMailbox.ts:56
export function getInboxPath(agentName: string, teamName?: string): string {
  const team = teamName || getTeamName() || 'default'
  const safeTeam = sanitizePathComponent(team)
  const safeAgentName = sanitizePathComponent(agentName)
  const inboxDir = join(getTeamsDir(), safeTeam, 'inboxes')
  return join(inboxDir, `${safeAgentName}.json`)
}
```

Writes use file locks for concurrency safety:

```typescript
// utils/teammateMailbox.ts:35
const LOCK_OPTIONS = {
  retries: {
    retries: 10,
    minTimeout: 5,
    maxTimeout: 100,
  },
}
```

10 retries with 5ms-100ms backoff -- handling race conditions when multiple Claude instances write to the same inbox simultaneously.

## SendMessageTool Routing Logic

SendMessageTool does more than write to the mailbox -- it has a complete routing logic:

```typescript
// tools/SendMessageTool/SendMessageTool.ts:802
// Route to in-process subagent by name or raw agentId
if (typeof input.message === 'string' && input.to !== '*') {
  const appState = context.getAppState()
  const registered = appState.agentNameRegistry.get(input.to)
  const agentId = registered ?? toAgentId(input.to)
  if (agentId) {
    const task = appState.tasks[agentId]
    if (isLocalAgentTask(task) && !isMainSessionTask(task)) {
      if (task.status === 'running') {
        queuePendingMessage(agentId, input.message, ...)
        return { data: { success: true,
          message: `Message queued for delivery to ${input.to} at its next tool round.`
        }}
      }
      // task exists but stopped -- auto-resume
      const result = await resumeAgentBackground({ agentId, prompt: input.message, ... })
      return { data: { success: true,
        message: `Agent "${input.to}" was stopped; resumed it in the background.`
      }}
    }
  }
}
```

Routing priority:
1. **In-process subagent** (running) -- queue directly, delivered at the next tool round.
2. **In-process subagent** (stopped) -- auto-resume and pass the message.
3. **Broadcast** (`to: "*"`) -- write to all teammates' mailboxes.
4. **Named teammate** -- write to the corresponding file mailbox.
5. **Bridge/UDS** -- cross-process/cross-machine messaging (remote collaboration mode).

## Session Recovery Mode Matching

An easily overlooked detail: coordinator mode is bound to the session. When recovering a session, the code checks and switches modes:

```typescript
// coordinator/coordinatorMode.ts:49
export function matchSessionMode(
  sessionMode: 'coordinator' | 'normal' | undefined,
): string | undefined {
  if (!sessionMode) return undefined

  const currentIsCoordinator = isCoordinatorMode()
  const sessionIsCoordinator = sessionMode === 'coordinator'

  if (currentIsCoordinator === sessionIsCoordinator) return undefined

  // Flip the env var
  if (sessionIsCoordinator) {
    process.env.CLAUDE_CODE_COORDINATOR_MODE = '1'
  } else {
    delete process.env.CLAUDE_CODE_COORDINATOR_MODE
  }

  return sessionIsCoordinator
    ? 'Entered coordinator mode to match resumed session.'
    : 'Exited coordinator mode to match resumed session.'
}
```

This guarantees consistent behavior after session recovery -- a coordinator session will not accidentally continue in normal mode due to a different current environment variable.

## Why Not Message Queues?

Redis, RabbitMQ, and other message queues are the standard choice for distributed systems. Why does Claude Code use files?

1. **Zero dependencies**: No extra services need to be installed on the user's machine. The filesystem is universally available "infrastructure."
2. **Crash-recoverable**: Inbox files survive process crashes and can be read after restart. Message queues' ACK semantics can actually lead to message loss.
3. **Auditable**: The `~/.claude/teams/{team}/inboxes/` directory can be directly `cat`-ed for inspection; debugging requires no broker connection.
4. **Concurrency model match**: Each agent writes to its own inbox (many-write, single-read), and file locks are sufficient for handling contention. Message queues' pub/sub model is over-engineering here.

## Why Not gRPC/WebSocket?

Long-lived connection protocols seem more efficient, but they do not fit Claude Code's agent lifecycle:

1. **Short agent lifespan**: Workers exit after completing their task; maintaining a persistent connection is pointless.
2. **Clear process boundaries**: Forked children are independent processes; files are the most natural cross-process communication mechanism.
3. **Prompt cache sharing does not depend on communication speed**: Inter-agent communication latency (millisecond-level file I/O) is negligible compared to API call latency (seconds).

## Key Takeaways

- Coordinator Mode is activated via the `CLAUDE_CODE_COORDINATOR_MODE` environment variable, replacing the system prompt with a 370-line commander role definition that includes tool usage specifications and worker management strategies.
- Fork subagent passes the parent's already-rendered system prompt **raw bytes** directly (rather than re-rendering), ensuring byte-level identity to maximize prompt cache hits.
- All fork children's message prefixes use a unified placeholder `"Fork started -- processing in background"`, with only the final text block differing per directive, maximizing the shared cache region.
- Inter-agent communication uses `~/.claude/teams/{team}/inboxes/{agent}.json` file mailboxes, with file locks ensuring concurrency safety (10 retries, 5-100ms backoff).
- SendMessageTool has 5 routing layers: in-process running agent direct queue, stopped agent auto-resume, broadcast to all inboxes, named teammate to corresponding inbox, bridge/UDS for cross-process communication.
- On session recovery, `matchSessionMode` automatically toggles the environment variable, ensuring a coordinator session does not accidentally continue in normal mode.
