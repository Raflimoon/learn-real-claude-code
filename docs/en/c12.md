# Chapter 12 -- React in the Terminal: Custom Reconciler and Three Object Pools

> CharPool uses an ASCII fast path (`Int32Array[128]`) to bypass Map lookups; StylePool's bit 0 encodes "whether a space is visible," letting the renderer skip invisible cells with a single bitmask check.

---

## Overview

Claude Code's terminal interface is not ncurses, not hand-written ANSI escapes -- it is a full React 19 application running on a forked version of Ink. The core consists of four layers:

1. **Custom Reconciler**: `react-reconciler` adapted for the terminal DOM (`ink-box`, `ink-text`).
2. **Yoga Layout**: Facebook's Flexbox engine where each character = one "pixel."
3. **Three Object Pools**: `CharPool`, `StylePool`, `HyperlinkPool` -- turning O(n) string comparisons into O(1) integer comparisons.
4. **Double Buffering + Blit Optimization**: Front/back Screen swapping; `Output` operation queue for batched writes and blits, sending only diffs.

All code blocks in this chapter come from the actual source, annotated with file names and line numbers.

`createInstance` transforms JSX elements into terminal DOM nodes. The `isInsideText` check prevents `<Box>` from being nested inside `<Text>` -- terminals have no inline-block, and such nesting would break layout:

```typescript
// ink/reconciler.ts:331-358
createInstance(
  originalType: ElementNames,
  newProps: Props,
  _root: DOMElement,
  hostContext: HostContext,
  internalHandle?: unknown,
): DOMElement {
  if (hostContext.isInsideText && originalType === 'ink-box') {
    throw new Error(`<Box> can't be nested inside <Text> component`)
  }

  const type =
    originalType === 'ink-text' && hostContext.isInsideText
      ? 'ink-virtual-text'
      : originalType

  const node = createNode(type)
```

`commitUpdate` performs a shallow diff of old vs. new props, calling `setAttribute` or `applyStyles` only for changed properties:

```typescript
// ink/reconciler.ts:426-458
commitUpdate(
  node: DOMElement,
  _type: ElementNames,
  oldProps: Props,
  newProps: Props,
): void {
  const props = diff(oldProps, newProps)
  const style = diff(oldProps['style'] as Styles, newProps['style'] as Styles)

  if (props) {
    for (const [key, value] of Object.entries(props)) {
      if (key === 'style') {
        setStyle(node, value as Styles)
        continue
      }
      // ...
    }
  }

  if (style && node.yogaNode) {
    applyStyles(node.yogaNode, style, newProps['style'] as Styles)
  }
},
```

### Why Not Browser DOM Emulation?

The terminal "DOM" has only 7 node types (`ink-root`, `ink-box`, `ink-text`, `ink-virtual-text`, `ink-link`, `ink-progress`, `ink-raw-ansi`). A browser DOM's `document.createElement` carries the massive overhead of event bubbling, CSS cascade, CSSOM, and more. In the custom DOM, node creation is just a plain object allocation plus a Yoga node:

```typescript
// ink/dom.ts:110-132
export const createNode = (nodeName: ElementNames): DOMElement => {
  const needsYogaNode =
    nodeName !== 'ink-virtual-text' &&
    nodeName !== 'ink-link' &&
    nodeName !== 'ink-progress'
  const node: DOMElement = {
    nodeName,
    style: {},
    attributes: {},
    childNodes: [],
    parentNode: undefined,
    yogaNode: needsYogaNode ? createLayoutNode() : undefined,
    dirty: false,
  }

  if (nodeName === 'ink-text') {
    node.yogaNode?.setMeasureFunc(measureTextNode.bind(null, node))
  }
  // ...
}
```

## 2. Yoga Layout and Dirty Marking

Every DOM mutation (attribute changes, child additions/removals) propagates upward via `markDirty`, marking the entire ancestor chain as dirty. For text/raw-ansi leaf nodes, it also calls `yogaNode.markDirty()` to trigger remeasurement:

```typescript
// ink/dom.ts:393-413
export const markDirty = (node?: DOMNode): void => {
  let current: DOMNode | undefined = node
  let markedYoga = false

  while (current) {
    if (current.nodeName !== '#text') {
      ;(current as DOMElement).dirty = true
      if (
        !markedYoga &&
        (current.nodeName === 'ink-text' ||
          current.nodeName === 'ink-raw-ansi') &&
        current.yogaNode
      ) {
        current.yogaNode.markDirty()
        markedYoga = true
      }
    }
    current = current.parentNode
  }
}
```

`setStyle` and `setTextStyles` perform shallow equal checks internally -- React creates new style objects on every render even when values haven't changed. Without this check, every frame would trigger a Yoga recalculation:

```typescript
// ink/dom.ts:266-274
export const setStyle = (node: DOMNode, style: Styles): void => {
  if (stylesEqual(node.style, style)) {
    return
  }
  node.style = style
  markDirty(node)
}
```

### Why Not CSS Grid?

Yoga only implements Flexbox, not Grid. But terminal UIs almost never need Grid -- content flows linearly (message lists, tool output), and Flexbox's `flexDirection: 'column'` is a natural fit. Yoga's WASM-compiled version needs only 1-5ms per frame for layout calculation; a general-purpose CSS Grid engine would be an order of magnitude slower.

## 3. Three Object Pools: CharPool, StylePool, HyperlinkPool

This is performance-critical. Double-buffer diffing needs to compare 10,000+ cells (200x50 terminal) per frame. Object pools convert string comparisons to integer comparisons.

**CharPool** provides an `Int32Array` fast path for ASCII characters, skipping `Map.get`'s hash computation:

```typescript
// ink/screen.ts:21-53
export class CharPool {
  private strings: string[] = [' ', ''] // Index 0 = space, 1 = empty (spacer)
  private stringMap = new Map<string, number>([
    [' ', 0],
    ['', 1],
  ])
  private ascii: Int32Array = initCharAscii() // charCode -> index, -1 = not interned

  intern(char: string): number {
    // ASCII fast-path: direct array lookup instead of Map.get
    if (char.length === 1) {
      const code = char.charCodeAt(0)
      if (code < 128) {
        const cached = this.ascii[code]!
        if (cached !== -1) return cached
        const index = this.strings.length
        this.strings.push(char)
        this.ascii[code] = index
        return index
      }
    }
    const existing = this.stringMap.get(char)
    if (existing !== undefined) return existing
    // ...
  }
}
```

**StylePool** is more clever -- bit 0 encodes "whether this style is visible on a space character" (background color, inverse, underline). The renderer skips unstyled spaces by checking `id & 1`, without needing to parse the AnsiCode array:

```typescript
// ink/screen.ts:129-140
intern(styles: AnsiCode[]): number {
  const key = styles.length === 0 ? '' : styles.map(s => s.code).join('\0')
  let id = this.ids.get(key)
  if (id === undefined) {
    const rawId = this.styles.length
    this.styles.push(styles.length === 0 ? [] : styles)
    id =
      (rawId << 1) |
      (styles.length > 0 && hasVisibleSpaceEffect(styles) ? 1 : 0)
    this.ids.set(key, id)
  }
  return id
}
```

StylePool also caches transition sequences between styles. The ANSI string from `(fromId, toId)` is computed only once:

```typescript
// ink/screen.ts:153-163
transition(fromId: number, toId: number): string {
  if (fromId === toId) return ''
  const key = fromId * 0x100000 + toId
  let str = this.transitionCache.get(key)
  if (str === undefined) {
    str = ansiCodesToString(diffAnsiCodes(this.get(fromId), this.get(toId)))
    this.transitionCache.set(key, str)
  }
  return str
}
```

**HyperlinkPool** has the simplest structure -- index 0 = no link, the rest are interned on demand:

```typescript
// ink/screen.ts:57-75
export class HyperlinkPool {
  private strings: string[] = [''] // Index 0 = no hyperlink
  private stringMap = new Map<string, number>()

  intern(hyperlink: string | undefined): number {
    if (!hyperlink) return 0
    let id = this.stringMap.get(hyperlink)
    if (id === undefined) {
      id = this.strings.length
      this.strings.push(hyperlink)
      this.stringMap.set(hyperlink, id)
    }
    return id
  }
}
```

### Why Not WeakMap or LRU Instead of Immortal Pools?

The three pools are never reset during the entire session (except HyperlinkPool, which resets every 5 minutes). The combination space of characters and styles is bounded -- terminals have only 128 ASCII characters and a few dozen SGR combinations. LRU eviction logic would add branching overhead to every intern call. Immortal pools give O(1) `Map.get` with zero GC pressure.

## 4. Double Buffering and the Output Operation Queue

`ink.tsx` holds `frontFrame` and `backFrame`. Each frame renders to the backFrame's Screen, diffs against frontFrame, then swaps:

```typescript
// ink/ink.tsx:99-100
private frontFrame: Frame;
private backFrame: Frame;
```

The `Output` class collects `write`, `blit`, `clip`, `clear`, and `shift` operations, batch-processing writes to the Screen buffer in `get()`. `charCache` is reused across frames -- most lines do not change, so tokenize + grapheme clustering becomes a cache hit:

```typescript
// ink/output.ts:196-205
reset(width: number, height: number, screen: Screen): void {
  this.width = width
  this.height = height
  this.screen = screen
  this.operations.length = 0
  resetScreen(screen, width, height)
  if (this.charCache.size > 16384) this.charCache.clear()
}
```

The `blit` operation copies unchanged regions in bulk from prevScreen, avoiding re-tokenizing and re-interning lines that have not changed:

```typescript
// ink/output.ts:209-211
blit(src: Screen, x: number, y: number, width: number, height: number): void {
  this.operations.push({ type: 'blit', src, x, y, width, height })
}
```

## 5. The Full Rendering Pipeline

```
JSX Component Tree
    | React 19 reconciler (ConcurrentRoot)
    v
ink DOM (ink-box, ink-text, ...)
    | dirty marking propagates upward
    v
Yoga calculateLayout (WASM)
    | each character = 1px
    v
renderNodeToOutput -> Output operation queue
    | write/blit/clip/clear/shift
    v
Output.get() -> Screen (charId[], styleId[], hyperlinkId[])
    | three object pools intern
    v
LogUpdate.diff(frontScreen, backScreen)
    | per-cell integer comparison
    v
ANSI escape sequences -> stdout
```

## Key Takeaways

- React 19's custom Reconciler lets the terminal UI be developed with standard JSX and hooks; `commitUpdate` directly receives old and new props, eliminating the need for `prepareUpdate` to return a payload.
- Yoga's Flexbox engine treats each terminal character as 1px, using `markDirty` upward propagation + `setStyle` shallow equal checks to avoid unnecessary layout recalculations.
- `CharPool` uses `Int32Array[128]` to provide an array-direct-lookup fast path for ASCII characters, bypassing `Map.get` hash overhead; `StylePool`'s bit 0 encodes space visibility, letting the renderer skip invisible cells with a bitmask check.
- The `Output` operation queue batch-processes write/blit/clip/clear to the Screen buffer; `charCache` reuses across frames to avoid repeated tokenization -- in steady-state frames, most cells take the blit path, making rendering cost proportional to changed area.
- The design philosophy of all three pools: **the combination space of terminal characters/styles/links is bounded; immortal pools with O(1) intern are superior to LRU eviction overhead**.
