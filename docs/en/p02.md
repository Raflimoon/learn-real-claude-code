# Agent Scaffolding: The Framework Matters More Than the Prompt

> Of query.ts's 1,729 lines, the LLM call itself occupies just 1 line. The other 1,728 lines determine whether that 1 line can execute reliably.

## Overview

The gap between a "build an AI agent in 5 minutes" tutorial and Claude Code is not prompt quality -- it is **scaffolding**: the framework code wrapped around the LLM call. This claim sounds abstract, but Claude Code's source provides precise quantitative evidence.

This chapter uses no pseudocode analogies. We directly dissect `query.ts` (1,729 lines) and `cli.tsx` (302 lines), using line numbers and line counts to prove the real scale of scaffolding.

## Source Insight 1: What query.ts's 1,729 Lines Are Doing

`query.ts` is Claude Code's core query loop -- the complete cycle of model generating a response, calling tools, and generating again after each user message. It has only 3 top-level functions (`yieldMissingToolResultBlocks`, `query`, `queryLoop`), but covers the full complexity of the agent runtime.

**Where does the actual LLM call happen?**

`query.ts:659` -- the `deps.callModel()` call. This is the data source for a `for await` loop, receiving messages from the streaming API. This is "the LLM call itself."

**So what are the other 1,728 lines doing?** Here is the functional breakdown by line number range:

### State Management (~80 lines)

```
query.ts:181-199   QueryParams type definition -- 18 parameter fields
query.ts:202-218   State type definition -- 12 mutable state fields
query.ts:252-280   Initial state construction + budgetTracker
```

The State type (`query.ts:202-218`) contains: messages, toolUseContext, autoCompactTracking, maxOutputTokensRecoveryCount, hasAttemptedReactiveCompact, maxOutputTokensOverride, pendingToolUseSummary, stopHookActive, turnCount, transition. Every field corresponds to an edge case you would not handle in a 50-line demo.

### Context Preprocessing Pipeline (~150 lines)

```
query.ts:369-394   applyToolResultBudget -- per-message tool result size limits
query.ts:396-410   snipCompact -- history snip preprocessing
query.ts:412-427   microcompact -- API-based tool result clearing
query.ts:440-447   context collapse -- collapse projection
query.ts:449-468   autocompact -- conversation-level compression
query.ts:470-543   compaction result handling and task budget tracking
```

Before the model is called, the message array passes through at least 5 preprocessing stages. Note the ordering: snip -> microcompact -> context collapse -> autocompact. This is not arbitrary -- the comment at `query.ts:396` explains that snip runs before microcompact because "both may run -- they are not mutually exclusive."

### Streaming Message Processing (~200 lines)

```
query.ts:654-863   for await streaming message loop
query.ts:748-787   backfillObservableInput -- tool input field backfill
query.ts:799-825   withheld error judgment (prompt_too_long, max_output_tokens, media_size)
query.ts:826-862   streaming tool executor integration
```

Streaming messages are not simple `stream.on('text', console.log)`. Every message requires judgment on whether it should be "withheld" -- held back from the user, waiting for a recovery mechanism to attempt processing. `query.ts:799-822` shows four independent withholding checks that can all be active simultaneously.

### Error Recovery Mechanisms (~300 lines)

```
query.ts:893-953    FallbackTriggeredError -- model degradation retry
query.ts:955-997    General error handling (image errors, API errors)
query.ts:1062-1183  prompt_too_long recovery chain:
                    1. context collapse drain (1090-1117)
                    2. reactive compact (1119-1183)
query.ts:1188-1256  max_output_tokens recovery chain:
                    1. 8K -> 64K upgrade (1199-1221)
                    2. recovery message injection (1223-1252)
                    3. max 3 retries (MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3)
```

This is the most sophisticated part of query.ts. The `prompt_too_long` error recovery is an ordered fallback chain: first try context collapse drain (lightweight), then try reactive compact (full compression) on failure, and only then report the error. The `max_output_tokens` recovery has three layers: first try upgrading from 8K to 64K (one-time, `query.ts:1199`), then inject a "Resume directly -- no apology, no recap" message to let the model continue, up to 3 times.

### Tool Execution Orchestration (~350 lines)

```
query.ts:560-568   StreamingToolExecutor initialization
query.ts:838-862   collecting completed tool results during streaming
query.ts:1012-1052 abort handling (streaming abort and tool abort, two paths)
query.ts:1360-1408 non-streaming tool execution fallback
query.ts:1411-1482 tool use summary async generation (Haiku model)
```

Tool execution has two modes: StreamingToolExecutor (starts executing tools while the model is still generating) and traditional mode (waits for model completion before batch execution). In streaming mode, when the user interrupts, synthetic tool_result blocks must be generated for all in-progress tools (`query.ts:1016-1029`), otherwise the API will error because tool_use blocks lack corresponding tool_result.

### Telemetry and Analytics (~100 lines)

Spread throughout the file, `logEvent('tengu_*', {...})` calls record every critical decision point. These event names are prefixed with `tengu_` -- the internal codename for Claude Code.

## Source Insight 3: 50-Line Demo Agent vs 1,729-Line query.ts

A typical demo agent looks like this:

```typescript
// 50-line demo agent
const response = await anthropic.messages.create({
  model: 'claude-sonnet-4-6', messages, tools: TOOLS,
});
for (const block of response.content) {
  if (block.type === 'tool_use') {
    const result = await executeTool(block.name, block.input);
    messages.push(/* tool_result */);
  }
}
```

**What are query.ts's 1,679 "extra" lines doing?** Precise breakdown:

| Category | Lines (est.) | Present in demo? |
|---------|-----------|---------------|
| State management (12 mutable fields) | ~80 | No |
| Context preprocessing pipeline (5 layers) | ~150 | No |
| Streaming message processing | ~200 | No (demo uses await) |
| Error recovery (3 fallback chains) | ~300 | No (demo throws) |
| Tool execution orchestration (streaming + abort) | ~350 | No (demo runs synchronously) |

| Turn management and loop control | ~200 | No (demo loops infinitely) |

| **Total** | **~1,630** | **~50** |

The demo agent is a `while(true)` loop with no exit condition, no error classification, no context management. query.ts's `while(true)` (`query.ts:307`) has the same form, but each iteration passes through 5 layers of preprocessing, 4 classes of error recovery, and 3 kinds of termination checks.

### Why Not Use LangChain's Off-the-Shelf Agent Loop?

`query.ts` is custom-built with no framework dependency. The reasons can be read from the source:

1. **State granularity**. query.ts's State type (`query.ts:202-218`) has 12 fields, including `transition` (`query.ts:216`) which records why the last iteration continued -- letting subsequent iterations decide next steps based on the previous recovery path. Framework abstraction layers do not provide this granularity.

2. **Diversity of continue sites**. query.ts has 7 `state = next; continue` sites -- each represents a different recovery scenario (collapse drain retry, reactive compact retry, max output tokens escalate, max output tokens recovery, stop hook blocking, token budget continuation, next turn). A framework's `onError` hook cannot distinguish these scenarios.

3. **Generator architecture**. `query()` is an `AsyncGenerator` (`query.ts:219-238`) that `yield`s messages to callers, enabling the UI layer to render in real time. This backpressure control is difficult to achieve in frameworks' callback-based architectures.

### Why Not Split Error Recovery Into Separate Modules?

Recovery logic does reference external modules (`reactiveCompact.ts`, `contextCollapse/index.ts`), but the orchestration logic ("try A first, then B on failure") must live inside the query loop -- because recovery involves modifying `state` and `continue`ing to the next iteration. Extracting it externally would require passing State between recovery modules and the query loop, adding unnecessary coupling.

## Source Insight 4: The transition Field -- Scaffolding's "Memory"

One field in query.ts's State type deserves special attention (`query.ts:216`):

```typescript
// query.ts:216
transition: Continue | undefined
```

`transition` records "why the last loop iteration continued." This field does not affect the LLM call -- the model never sees it. Its sole consumer is the query loop's own recovery logic.

Concrete example: `query.ts:1092-1093` checks whether the last transition was `collapse_drain_retry`:

```typescript
// query.ts:1092-1093
if (state.transition?.reason !== 'collapse_drain_retry') {
  // ... attempt collapse drain
}
```

If collapse drain was already attempted last time but the API still returned 413, it skips collapse drain and jumps directly to reactive compact. This kind of "recovery path awareness" is completely absent from demo agents -- which either retry infinitely or crash outright.

**The complete list of 7 transition reasons:**

1. `collapse_drain_retry` -- retry after context collapse drain
2. `reactive_compact_retry` -- retry after full compaction
3. `max_output_tokens_escalate` -- upgrade from 8K to 64K
4. `max_output_tokens_recovery` -- inject recovery message to continue
5. `stop_hook_blocking` -- retry after stop hook blocks
6. `token_budget_continuation` -- token budget auto-continue
7. `next_turn` -- normal next tool execution turn

Each reason encodes a specific recovery semantic. This is scaffolding's "memory" -- it lets the query loop know where it came from and why after each `continue`.

### Why Not Use Retry Count Instead of Transition Reason?

Because retry count only knows "how many times it retried," not "why it retried." Jumping from collapse drain failure to reactive compact versus reporting an error after reactive compact failure are two completely different semantics. The transition reason encodes position in the recovery chain, not just count.

## Key Takeaways

- Of query.ts's 1,729 lines, the LLM call itself is just 1 line (`deps.callModel()`, line 659); the other 1,728 lines are state management, context preprocessing, streaming processing, error recovery, tool orchestration, telemetry, and turn management -- the most precise quantitative evidence that "scaffolding matters more than prompts."
- Before each LLM call, the message array passes through a 5-layer preprocessing pipeline (tool result budget -> snip -> microcompact -> context collapse -> autocompact); the order is non-interchangeable, with each layer solving context management problems at different granularities.
- Error recovery is not simple try/catch: prompt_too_long has a 3-level fallback chain (collapse drain -> reactive compact -> error), max_output_tokens has a 3-level recovery mechanism (8K->64K upgrade -> recovery message injection -> max 3 retries), and model unavailability has model degradation (FallbackTriggeredError).
- cli.tsx's 302 lines contain 15 fast-path branches, demonstrating the real complexity of a production CLI: a single binary serves daemon worker, bridge, background session, template job, environment runner, self-hosted runner, and more runtime modes.
- query.ts does not use a framework because of its 7 different continue sites, each corresponding to a different recovery scenario -- this "recovery context-aware" loop control is something framework abstraction layers cannot provide.
