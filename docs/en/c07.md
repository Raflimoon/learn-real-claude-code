# Memory System: File-Based Persistence & Auto-Dream

> MEMORY.md is an index, not a database -- 200-line cap, truncated with a warning injected for the model.

## Overview

Every LLM session is inherently amnesiac. Claude Code solves cross-session persistence through a three-layer mechanism: (1) a `MEMORY.md` index file, loaded automatically at every session start; (2) topic files (standalone .md files), recalled on-demand via an AI selector; (3) Auto-Dream background consolidation, periodically merging scattered information into structured memory.

There is no "Dream four-phase sub-agent" abstraction in the source code -- the real implementation is a forked agent executing a carefully designed prompt, using a lock file for concurrency control and `mtime` for time gating.

## Memory Directory Structure

Memory is stored under `~/.claude/projects/<sanitized-path>/memory/`. The path computation logic lives in `memdir/paths.ts`:

```typescript
// memdir/paths.ts:223
export const getAutoMemPath = memoize(
  (): string => {
    const override = getAutoMemPathOverride() ?? getAutoMemPathSetting()
    if (override) {
      return override
    }
    const projectsDir = join(getMemoryBaseDir(), 'projects')
    return (
      join(projectsDir, sanitizePath(getAutoMemBase()), AUTO_MEM_DIRNAME) + sep
    ).normalize('NFC')
  },
  () => getProjectRoot(),
)
```

`getAutoMemBase()` uses the git repository's canonical root (not cwd), so all worktrees of the same repo share a single memory store. The result is `memoize`-cached to avoid stat-ing the filesystem on every render.

## MEMORY.md: An Index with Hard Caps

MEMORY.md is not where all memories are stored -- it is an **index**, with each line pointing to a topic file. It has strict dual caps:

```typescript
// memdir/memdir.ts:37
export const ENTRYPOINT_NAME = 'MEMORY.md'
export const MAX_ENTRYPOINT_LINES = 200
export const MAX_ENTRYPOINT_BYTES = 25_000
```

The truncation logic in `truncateEntrypointContent()` applies line truncation first, then byte truncation:

```typescript
// memdir/memdir.ts:57
export function truncateEntrypointContent(raw: string): EntrypointTruncation {
  const trimmed = raw.trim()
  const contentLines = trimmed.split('\n')
  const lineCount = contentLines.length
  const byteCount = trimmed.length

  const wasLineTruncated = lineCount > MAX_ENTRYPOINT_LINES
  const wasByteTruncated = byteCount > MAX_ENTRYPOINT_BYTES

  if (!wasLineTruncated && !wasByteTruncated) {
    return { content: trimmed, lineCount, byteCount, wasLineTruncated, wasByteTruncated }
  }

  let truncated = wasLineTruncated
    ? contentLines.slice(0, MAX_ENTRYPOINT_LINES).join('\n')
    : trimmed

  if (truncated.length > MAX_ENTRYPOINT_BYTES) {
    const cutAt = truncated.lastIndexOf('\n', MAX_ENTRYPOINT_BYTES)
    truncated = truncated.slice(0, cutAt > 0 ? cutAt : MAX_ENTRYPOINT_BYTES)
  }

  // Append warning on truncation, visible to the model
  return {
    content: truncated +
      `\n\n> WARNING: ${ENTRYPOINT_NAME} is ${reason}. Only part of it was loaded...`,
    // ...
  }
}
```

The byte cap (25KB) guards against the "fewer than 200 lines but each line is enormous" case -- p97 observations have shown a single MEMORY.md reaching 197KB within 200 lines.

## Four Memory Types

Memory files use frontmatter to tag their type:

```typescript

```

The prompt explicitly excludes code-derivable information:

```typescript

  '- Git history, recent changes, or who-changed-what',
  '- Debugging solutions or fix recipes',

  '- Ephemeral task details: in-progress work, temporary state',
]
```

### Why Not a Vector Database

The memory system is entirely file-based -- no embeddings, no vector retrieval. The reasoning: (1) depending on an external database adds runtime complexity and failure points; (2) with fewer than 200 .md files, the AI selector is already precise enough; (3) the filesystem natively supports git version control and manual editing.

## findRelevantMemories: AI-Driven On-Demand Recall

When a session starts or the user asks a question, the system does not load all memory files -- it uses a Sonnet side-query to select up to 5 relevant files:

```typescript
// memdir/findRelevantMemories.ts:39
export async function findRelevantMemories(
  query: string,
  memoryDir: string,
  signal: AbortSignal,
  recentTools: readonly string[] = [],
  alreadySurfaced: ReadonlySet<string> = new Set(),
): Promise<RelevantMemory[]> {
  const memories = (await scanMemoryFiles(memoryDir, signal)).filter(
    m => !alreadySurfaced.has(m.filePath),
  )
  if (memories.length === 0) return []

  const selectedFilenames = await selectRelevantMemories(
    query, memories, signal, recentTools,
  )
  // ...
}
```

The selector uses a Sonnet model, receiving the memory file manifest (filenames + frontmatter descriptions) and returning up to 5 filenames:

```typescript
// memdir/findRelevantMemories.ts:18
const SELECT_MEMORIES_SYSTEM_PROMPT = `You are selecting memories that will
be useful to Claude Code as it processes a user's query. You will be given
the user's query and a list of available memory files with their filenames
and descriptions.

Return a list of filenames for the memories that will clearly be useful
(up to 5). Only include memories that you are certain will be helpful...`
```

The `recentTools` parameter is an elegant denoising mechanism: if the model is currently using a tool (e.g., `mcp__X__spawn`), the selector will not recall that tool's reference docs -- the model is already using it and does not need the documentation. However, gotchas/known-issues memories are still selected.

The scan phase (`memoryScan.ts`) reads only the first 30 lines of frontmatter from each .md file (`FRONTMATTER_MAX_LINES = 30`), not the body. Results are sorted by mtime descending, capped at the first 200 (`MAX_MEMORY_FILES = 200`). `Promise.allSettled` ensures a single file read failure does not block the entire scan.

## Auto-Dream: Lock-File-Based Background Consolidation

Auto-Dream is not a user-triggered "dream" -- it is a background task that runs automatically when conditions are met.

### Triple Gating

Gates are ordered from cheapest to most expensive (`autoDream.ts:8` comment): (1) Time: >= 24 hours since last consolidation (a single stat call); (2) Session count: >= 5 new sessions since last consolidation; (3) Lock: no other process is currently consolidating. Default thresholds are `minHours: 24, minSessions: 5` (adjustable remotely via GrowthBook).

The time gate requires only a single stat call -- reading the lock file's mtime:

```typescript
// services/autoDream/consolidationLock.ts:29
export async function readLastConsolidatedAt(): Promise<number> {
  try {
    const s = await stat(lockPath())
    return s.mtimeMs
  } catch {
    return 0  // File doesn't exist = never consolidated
  }
}
```

The lock file's mtime *is* `lastConsolidatedAt` -- no additional state file needed. The file content holds the owner's PID, used to detect stale locks after crashes.

### Why Not Database Timestamps

Using mtime instead of a database record has several advantages: (1) zero dependencies, no extra process needed; (2) natural cross-process coordination (the filesystem is shared); (3) lock acquisition and timestamp update are a single atomic operation (write automatically updates mtime).

### Consolidation Prompt: Four-Phase Instructions

The prompt built in `consolidationPrompt.ts` contains four phases: Phase 1 Orient (ls the memory directory, read MEMORY.md and existing topic files), Phase 2 Gather (collect new signals from logs and transcripts), Phase 3 Consolidate (merge into existing topic files, convert relative dates, delete contradicted facts), Phase 4 Prune (update the MEMORY.md index, keeping it within 200 lines / 25KB).

The critical constraint: the Dream agent's Bash is restricted to read-only commands:

```typescript
// services/autoDream/autoDream.ts:216
const extra = `
**Tool constraints for this run:** Bash is restricted to read-only commands
(ls, find, grep, cat, stat, wc, head, tail, and similar). Anything that
writes, redirects to a file, or modifies state will be denied.`
```

This constraint is not enforced by the prompt alone -- it is enforced at the code level through `createAutoMemCanUseTool`:

```typescript
// services/extractMemories/extractMemories.ts:171
export function createAutoMemCanUseTool(memoryDir: string): CanUseToolFn {
  return async (tool, input) => {
    // Read/Grep/Glob unrestricted
    if (tool.name === FILE_READ_TOOL_NAME || tool.name === GREP_TOOL_NAME
        || tool.name === GLOB_TOOL_NAME) {
      return { behavior: 'allow', updatedInput: input }
    }
    // Bash only allows isReadOnly commands
    if (tool.name === BASH_TOOL_NAME) {
      const parsed = tool.inputSchema.safeParse(input)
      if (parsed.success && tool.isReadOnly(parsed.data)) {
        return { behavior: 'allow', updatedInput: input }
      }
      return denyAutoMemTool(tool, 'only read-only bash allowed')
    }
    // Edit/Write only allowed within the memory directory
    if (tool.name === FILE_EDIT_TOOL_NAME || tool.name === FILE_WRITE_TOOL_NAME) {
      const filePath = (input as any).file_path
      if (typeof filePath === 'string' && isAutoMemPath(expandPath(filePath))) {
        return { behavior: 'allow', updatedInput: input }
      }
      return denyAutoMemTool(tool, 'writes only allowed in memory dir')
    }
  }
}
```

This is the real "read-only bash sub-agent" -- not a separate agent type, but a `canUseTool` callback function that applies fine-grained permission control on every tool invocation.

### Extract Memories: Mutually Exclusive with Dream

In addition to periodic Auto-Dream, there is an `extractMemories` process that runs after each query, pulling worth-remembering information from the current session. It is mutually exclusive with Dream:

```typescript
// services/extractMemories/extractMemories.ts:121
function hasMemoryWritesSince(
  messages: Message[],
  sinceUuid: string | undefined,
): boolean {
  // If the main agent already wrote memory files, skip extraction
  for (const block of content) {
    const filePath = getWrittenFilePath(block)
    if (filePath !== undefined && isAutoMemPath(filePath)) {
      return true
    }
  }
  return false
}
```

If the main agent already wrote memory during the conversation (user said "remember this"), background extraction is skipped -- preventing duplicate writes.

1. **Verify MEMORY.md caps**: In `memdir/memdir.ts`, confirm `MAX_ENTRYPOINT_LINES = 200` and `MAX_ENTRYPOINT_BYTES = 25_000`. Then examine how `truncateEntrypointContent` appends a `> WARNING:` message upon truncation.

2. **Verify memory types**: In `memdir/memoryTypes.ts`, confirm `MEMORY_TYPES = ['user', 'feedback', 'project', 'reference']`. In `WHAT_NOT_TO_SAVE_SECTION`, confirm that code patterns and git history are excluded.

3. **Verify Dream gating thresholds**: In `services/autoDream/autoDream.ts`, confirm `DEFAULTS = { minHours: 24, minSessions: 5 }`. In `consolidationLock.ts`, confirm that the lock file's mtime serves as lastConsolidatedAt.

4. **Verify read-only constraint**: In `services/extractMemories/extractMemories.ts`, find `createAutoMemCanUseTool` and confirm that Bash only allows `isReadOnly` commands, and Edit/Write only allow writes to the memory directory.

5. **Inspect your own memory directory**: Run `ls ~/.claude/projects/` to see memory directories organized by project path, each containing a `MEMORY.md` and various topic files.

## Key Takeaways

- MEMORY.md is an index, not a database, with dual caps of 200 lines + 25KB; on truncation, a WARNING message is appended so the model knows the index is incomplete.
- Memory types are limited to four (user/feedback/project/reference), and code-derivable information is explicitly excluded -- grep and git log are more reliable than memory files.
- `findRelevantMemories` uses a Sonnet side-query to select up to 5 relevant files from a frontmatter manifest, without reading file bodies or using a vector database.
- Auto-Dream uses triple gating (time > session count > lock file) to control trigger frequency, with the lock file's mtime doubling as the lastConsolidatedAt timestamp.
- The Dream agent's "read-only Bash" is not enforced by prompt instructions alone -- it is enforced at the code level through the `createAutoMemCanUseTool` callback: Bash only allows `isReadOnly` commands, Write/Edit only allow writes to the memory directory.
