# Hook Lifecycle: Aspects of Tool Execution

> 27 hook events, 4 hook backends (command/http/prompt/agent), async registry -- the real aspect system is far more complex than the documentation suggests.

## Overview

The hook system is Claude Code's most powerful extension point. It inserts custom logic at every critical node of tool execution, covering the complete lifecycle from session startup to file changes.

But the real source code diverges significantly from the simplified "three hook types" description: there are actually **27 hook events**, **4 persistent backends** plus one runtime function hook, a complete **async hook registry**, and a sophisticated JSON output validation pipeline.

## 4 Persistent Backends + 1 Runtime Backend

## Hook Output JSON Validation Pipeline

Hooks are not simple pass/fail. The source code has a complete set of Zod schemas validating each hook's JSON output, supporting per-event specific output:

```typescript

```

Key detail: `PreToolUse`'s `hookSpecificOutput` can return both `permissionDecision` (allow/deny/ask) and `updatedInput` (modify tool input) simultaneously, meaning a single hook can approve execution while also modifying the parameters.

## Async Hook Registry

Hooks can return `{ async: true }` to declare themselves asynchronous, and Claude Code will continue polling for results in the background:

```typescript
// utils/hooks/AsyncHookRegistry.ts:28
const pendingHooks = new Map<string, PendingAsyncHook>()

// utils/hooks/AsyncHookRegistry.ts:30
export function registerPendingAsyncHook({
  processId, hookId, asyncResponse, hookName, hookEvent,
  command, shellCommand, toolName, pluginId,
}: { ... }): void {
  const timeout = asyncResponse.asyncTimeout || 15000 // Default 15s
  const stopProgressInterval = startHookProgressInterval({
    hookId, hookName, hookEvent,
    getOutput: async () => { ... },
  })
  pendingHooks.set(processId, {
    processId, hookId, hookName, hookEvent,
    startTime: Date.now(), timeout,
    responseAttachmentSent: false, shellCommand, stopProgressInterval,
  })
}
```

The polling logic in `checkForAsyncHookResponses` iterates over all pending hooks, checks if the shell command has completed, and parses JSON lines from stdout:

```typescript
// utils/hooks/AsyncHookRegistry.ts:193
const lines = stdout.split('\n')
for (const line of lines) {
  if (line.trim().startsWith('{')) {
    try {
      const parsed = jsonParse(line.trim())
      if (!('async' in parsed)) {
        response = parsed
        break
      }
    } catch { /* ... */ }
  }
}
```

This means the async hook output protocol is: first output `{"async":true}` to start background mode, then output a regular JSON object as the final result when complete.

## PreToolUse Permission Decision Chain

The `permissionDecision` returned by a PreToolUse hook is not the final verdict. The source code has a precise decision chain: a hook's allow cannot bypass settings.json deny/ask rules:

```typescript
// services/tools/toolHooks.ts:347
if (hookPermissionResult?.behavior === 'allow') {
  const hookInput = hookPermissionResult.updatedInput ?? input

  // Hook allow skips the interactive prompt, but deny/ask rules still apply.
  const ruleCheck = await checkRuleBasedPermissions(tool, hookInput, toolUseContext)
  if (ruleCheck === null) {
    return { decision: hookPermissionResult, input: hookInput }
  }
  if (ruleCheck.behavior === 'deny') {
    // deny rule overrides hook approval
    return { decision: ruleCheck, input: hookInput }
  }
  // ask rule -- dialog required despite hook approval
  return {
    decision: await canUseTool(tool, hookInput, toolUseContext, assistantMessage, toolUseID),
    input: hookInput,
  }
}
```

This is a safety invariant: hooks can skip the interactive prompt, but they cannot override admin-configured deny rules.

## Event System: Buffering + Selective Emission

Hook events are not unconditionally emitted. The source code has a buffering mechanism and an allowlist:

```typescript
// utils/hooks/hookEvents.ts:18
const ALWAYS_EMITTED_HOOK_EVENTS = ['SessionStart', 'Setup'] as const

const MAX_PENDING_EVENTS = 100

// utils/hooks/hookEvents.ts:72
function emit(event: HookExecutionEvent): void {
  if (eventHandler) {
    eventHandler(event)
  } else {
    pendingEvents.push(event)
    if (pendingEvents.length > MAX_PENDING_EVENTS) {
      pendingEvents.shift()  // Ring buffer -- drop the oldest
    }
  }
}
```

Before a handler is registered, events are buffered into a ring queue with a 100-event cap. Only `SessionStart` and `Setup` are emitted by default; other events require the `includeHookEvents` option or `CLAUDE_CODE_REMOTE` mode to be emitted.

## Session Hooks: Map, Not Record

Session hooks use Map instead of Record for storage, and this is not an arbitrary choice:

```typescript
// utils/hooks/sessionHooks.ts:48
/**
 * Map (not Record) so .set/.delete don't change the container's identity.
 * Mutator functions mutate the Map and return prev unchanged, letting
 * store.ts's Object.is(next, prev) check short-circuit and skip listener
 * notification. ...
 * This matters under high-concurrency workflows: parallel() with N
 * schema-mode agents fires N addFunctionHook calls in one synchronous
 * tick. With a Record + spread, each call cost O(N) to copy the growing
 * map (O(N^2) total) plus fired all ~30 store listeners.
 * With Map: .set() is O(1), return prev means zero listener fires.
 */
export type SessionHooksState = Map<string, SessionStore>
```

When N concurrent agents register hooks simultaneously, the Record approach has O(N^2) time complexity, while Map is O(N).

## Why Not EventEmitter?

Node.js's built-in EventEmitter seems like a natural fit for a hook system, but the source code uses a custom event pipeline. The reasons are evident in `hookEvents.ts`:

1. **Buffering requirement**: EventEmitter drops events emitted before handlers are registered, but Claude Code needs to buffer events that arrive before session startup.
2. **Selective emission**: The `shouldEmit()` function decides whether to emit based on event type and configuration -- EventEmitter has no such filtering layer.
3. **Ring queue overflow**: `MAX_PENDING_EVENTS = 100` prevents memory leaks; EventEmitter's listener queue has no capacity control.

## Why Not Middleware Pattern?

Express/Koa-style middleware chains are the classic pattern for request processing, but the hook system does not adopt it:

1. **Async lifecycle**: Hooks can return `{async:true}` and run in the background; middleware's `next()` semantics cannot express "I'm returning now but will have results later."
2. **Multi-source aggregation**: Hooks come from three sources (settings.json, session runtime, plugins) and need to be merged before execution -- middleware chains assume a single registration source.
3. **Permission layering**: PreToolUse results must be layered with settings.json deny/ask rules, not a simple pass/block -- this exceeds middleware's req/res model.

## Key Takeaways

- Claude Code actually has **27 hook events**, covering the complete lifecycle from `SessionStart` to `FileChanged`, far beyond the 3 documented types.
- **4 persistent backends** (command/http/prompt/agent) plus 1 runtime function hook; the agent hook spawns an independent Claude instance for agentic verification.
- Async hooks are managed through `AsyncHookRegistry`, using a two-phase protocol of "first output `{async:true}`, then output the result JSON when complete."
- PreToolUse hook `allow` is not the final verdict -- `resolveHookPermissionDecision` in the source code ensures that settings.json deny rules always take priority over hook approval.
- The event system uses a ring buffer capped at 100 events; only `SessionStart` and `Setup` are emitted by default, and other events require explicit enablement.
- Session hooks use Map instead of Record for storage, avoiding O(N^2) copy overhead and unnecessary listener notifications when N concurrent agents register simultaneously.
