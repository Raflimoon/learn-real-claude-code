# Chapter 14 -- Observability: OTel Metrics, Per-Model Cost Tracking, and Error Recovery Pipeline

> `withRetry` is an AsyncGenerator -- each retry wait yields a `SystemAPIErrorMessage`, letting the UI display a real-time countdown without blocking control flow.

---

## Overview

A production-grade agent needs engineering guarantees on three dimensions: knowing what is happening (telemetry), knowing how much it costs (cost tracking), and automatically recovering from failures (retry pipeline). Claude Code's implementation is far more complex than "retry on error":

1. **OTel metrics**: Session/cost/token/LOC counters reported via OpenTelemetry `Meter`.
2. **Per-model cost tracking**: `cost-tracker.ts` accumulates input/output/cache-read/cache-write tokens by model, calculating USD cost in real time.
3. **Retry pipeline**: `withRetry` is an AsyncGenerator that distinguishes 529 (overloaded) from 429 (rate limited), supports fast mode degradation, model fallback, and unattended infinite retry.

All code blocks in this chapter come from the actual source, annotated with file names and line numbers.

### Why Not StatsD?

Legacy Claude Code used Datadog StatsD. OpenTelemetry is a vendor-neutral standard -- when enterprise users use Bedrock/Vertex, OTEL data can route to their own Grafana/Datadog/New Relic. StatsD is a Datadog-proprietary protocol that locks you into one backend.

## 2. Per-Model Cost Tracking

`cost-tracker.ts` is not just a total counter -- it aggregates all token categories by model name:

```typescript
// cost-tracker.ts:71-80
type StoredCostState = {
  totalCostUSD: number
  totalAPIDuration: number
  totalAPIDurationWithoutRetries: number
  totalToolDuration: number
  totalLinesAdded: number
  totalLinesRemoved: number
  lastDuration: number | undefined
  modelUsage: { [modelName: string]: ModelUsage } | undefined
}
```

After each API call completes, `addToTotalSessionCost` accumulates to the corresponding model:

```typescript
// cost-tracker.ts:250-276
function addToTotalModelUsage(
  cost: number,
  usage: Usage,
  model: string,
): ModelUsage {
  const modelUsage = getUsageForModel(model) ?? {
    inputTokens: 0,
    outputTokens: 0,
    cacheReadInputTokens: 0,
    cacheCreationInputTokens: 0,
    webSearchRequests: 0,
    costUSD: 0,
    contextWindow: 0,
    maxOutputTokens: 0,
  }

  modelUsage.inputTokens += usage.input_tokens
  modelUsage.outputTokens += usage.output_tokens
  modelUsage.cacheReadInputTokens += usage.cache_read_input_tokens ?? 0
  modelUsage.cacheCreationInputTokens += usage.cache_creation_input_tokens ?? 0
  modelUsage.webSearchRequests +=
    usage.server_tool_use?.web_search_requests ?? 0
  modelUsage.costUSD += cost
  modelUsage.contextWindow = getContextWindowForModel(model, getSdkBetas())
  modelUsage.maxOutputTokens = getModelMaxOutputTokens(model).default
  return modelUsage
}
```

Note the separate tracking of `cacheReadInputTokens` and `cacheCreationInputTokens`. Cache reads cost roughly 10% of normal input token price; precisely tracking these two numbers is necessary for accurately calculating the cost savings from prompt caching.

Cost is simultaneously reported to OTel counters, differentiated by fast mode:

```typescript
// cost-tracker.ts:287-300
const attrs =
  isFastModeEnabled() && usage.speed === 'fast'
    ? { model, speed: 'fast' }
    : { model }

getCostCounter()?.add(cost, attrs)
getTokenCounter()?.add(usage.input_tokens, { ...attrs, type: 'input' })
getTokenCounter()?.add(usage.output_tokens, { ...attrs, type: 'output' })
getTokenCounter()?.add(usage.cache_read_input_tokens ?? 0, {
  ...attrs,
  type: 'cacheRead',
})
getTokenCounter()?.add(usage.cache_creation_input_tokens ?? 0, {
  ...attrs,
  type: 'cacheCreation',
})
```

Advisor (nested model calls) costs accumulate recursively -- a single main call can trigger multiple advisor sub-calls:

```typescript
// cost-tracker.ts:302-321
for (const advisorUsage of getAdvisorUsage(usage)) {
  const advisorCost = calculateUSDCost(advisorUsage.model, advisorUsage)
  logEvent('tengu_advisor_tool_token_usage', {
    advisor_model: advisorUsage.model as ...,
    // ...
    cost_usd_micros: Math.round(advisorCost * 1_000_000),
  })
  totalCost += addToTotalSessionCost(
    advisorCost,
    advisorUsage,
    advisorUsage.model,
  )
}
```

Session costs are persisted to project config, supporting `--resume` to restore accumulated costs:

```typescript
// cost-tracker.ts:130-137
export function restoreCostStateForSession(sessionId: string): boolean {
  const data = getStoredSessionCosts(sessionId)
  if (!data) {
    return false
  }
  setCostStateForRestore(data)
  return true
}
```

### Why Not Per-Request Logs Instead of In-Memory Aggregation?

Writing a log entry per API call then aggregating produces heavy disk I/O in long sessions with 100+ calls. In-memory aggregation writes only once at session end or on `saveCurrentSessionCosts`. The real-time cost display in the terminal's upper right reads a memory variable -- zero I/O.

## 3. withRetry: The AsyncGenerator Retry Pipeline

`withRetry` is not an ordinary async function -- it is an `AsyncGenerator` that yields `SystemAPIErrorMessage` during each retry wait, letting the UI display a real-time countdown:

```typescript
// services/api/withRetry.ts:170-178
export async function* withRetry<T>(
  getClient: () => Promise<Anthropic>,
  operation: (
    client: Anthropic,
    attempt: number,
    context: RetryContext,
  ) => Promise<T>,
  options: RetryOptions,
): AsyncGenerator<SystemAPIErrorMessage, T> {
```

Core parameters: maximum 10 retries (`DEFAULT_MAX_RETRIES = 10`), base delay of 500ms with exponential growth:

```typescript
// services/api/withRetry.ts:54-55
const DEFAULT_MAX_RETRIES = 10
export const BASE_DELAY_MS = 500
```

Exponential backoff with random jitter (25% range) to prevent thundering herd:

```typescript
// services/api/withRetry.ts:530-548
export function getRetryDelay(
  attempt: number,
  retryAfterHeader?: string | null,
  maxDelayMs = 32000,
): number {
  if (retryAfterHeader) {
    const seconds = parseInt(retryAfterHeader, 10)
    if (!isNaN(seconds)) {
      return seconds * 1000
    }
  }

  const baseDelay = Math.min(
    BASE_DELAY_MS * Math.pow(2, attempt - 1),
    maxDelayMs,
  )
  const jitter = Math.random() * 0.25 * baseDelay
  return baseDelay + jitter
}
```

## 4. 529 Error Handling: Three Consecutive Failures Trigger Fallback

529 (Overloaded) is not an ordinary retry scenario. After 3 consecutive 529s, model fallback triggers:

```typescript
// services/api/withRetry.ts:53
const MAX_529_RETRIES = 3
```

```typescript
// services/api/withRetry.ts:327-351
if (is529Error(error) && ...) {
  consecutive529Errors++
  if (consecutive529Errors >= MAX_529_RETRIES) {
    if (options.fallbackModel) {
      logEvent('tengu_api_opus_fallback_triggered', {
        original_model: options.model as ...,
        fallback_model: options.fallbackModel as ...,
        provider: getAPIProviderForStatsig(),
      })
      throw new FallbackTriggeredError(
        options.model,
        options.fallbackModel,
      )
    }
    // ...
  }
}
```

529 errors from non-foreground sources are dropped immediately, without retry -- preventing background tasks (summarization, title generation, classifiers) from amplifying traffic during capacity cascades:

```typescript
// services/api/withRetry.ts:62-82
const FOREGROUND_529_RETRY_SOURCES = new Set<QuerySource>([
  'repl_main_thread',
  // ...
  'auto_mode',
  ...(feature('BASH_CLASSIFIER') ? (['bash_classifier'] as const) : []),
])
```


## 5. Fast Mode Degradation

Fast mode's degradation strategy when encountering 429/529 is layered:

```typescript
// services/api/withRetry.ts:284-305
const retryAfterMs = getRetryAfterMs(error)
if (retryAfterMs !== null && retryAfterMs < SHORT_RETRY_THRESHOLD_MS) {
  // Short retry-after (<20s): wait and retry with fast mode still active
  await sleep(retryAfterMs, options.signal, { abortError })
  continue
}
// Long or unknown retry-after: enter cooldown (switches to standard speed)
const cooldownMs = Math.max(
  retryAfterMs ?? DEFAULT_FAST_MODE_FALLBACK_HOLD_MS,
  MIN_COOLDOWN_MS,
)
triggerFastModeCooldown(Date.now() + cooldownMs, cooldownReason)
retryContext.fastMode = false
continue
```

- **Short delay (<20s)**: Keep fast mode, wait and retry -- preserves prompt cache (model name unchanged).
- **Long delay (>=20s)**: Enter cooldown, degrade to standard model, do not attempt fast mode for at least 10 minutes.

```typescript
// services/api/withRetry.ts:799-801
const DEFAULT_FAST_MODE_FALLBACK_HOLD_MS = 30 * 60 * 1000 // 30 minutes
const SHORT_RETRY_THRESHOLD_MS = 20 * 1000 // 20 seconds
const MIN_COOLDOWN_MS = 10 * 60 * 1000 // 10 minutes
```

### Why Not a Classic Circuit Breaker Pattern?

A classic circuit breaker is binary (OPEN/CLOSED). Claude Code's retry pipeline is multi-layered and progressive: retry on same model -> degrade model -> fallback model -> give up. A 529 does not mean "service is down" -- it may just mean the current model is overloaded, and switching models resolves it. The circuit breaker's "fully disconnect" approach is too coarse for the agent experience.

## 6. Unattended Infinite Retry

The ant-exclusive `CLAUDE_CODE_UNATTENDED_RETRY` mode retries infinitely on 429/529, supporting wait windows up to 6 hours:

```typescript
// services/api/withRetry.ts:96-98
const PERSISTENT_MAX_BACKOFF_MS = 5 * 60 * 1000
const PERSISTENT_RESET_CAP_MS = 6 * 60 * 1000 * 1000
const HEARTBEAT_INTERVAL_MS = 30_000
```

Long waits are chunked into 30-second heartbeats, each yielding a `SystemAPIErrorMessage`. This prevents the host environment (Kubernetes, CI runner) from marking the process as idle due to prolonged stdout silence:

```typescript
// services/api/withRetry.ts:489-501
let remaining = delayMs
while (remaining > 0) {
  if (options.signal?.aborted) throw new APIUserAbortError()
  if (error instanceof APIError) {
    yield createSystemAPIErrorMessage(
      error,
      remaining,
      reportedAttempt,
      maxRetries,
    )
  }
  const chunk = Math.min(remaining, HEARTBEAT_INTERVAL_MS)
  await sleep(chunk, options.signal, { abortError })
  remaining -= chunk
}
```

`attempt` is clamped at `maxRetries` to prevent the for loop from terminating; the real attempt count is tracked by `persistentAttempt`:

```typescript
// services/api/withRetry.ts:506
if (attempt >= maxRetries) attempt = maxRetries
```

## Key Takeaways

- `withRetry` is an AsyncGenerator, not an ordinary async function: each retry wait yields a `SystemAPIErrorMessage`, letting the UI display a real-time countdown without blocking control flow.
- Per-model cost tracking precisely distinguishes five token categories -- `input`/`output`/`cacheRead`/`cacheCreation`/`webSearch` -- and advisor sub-call costs recursively accumulate to the total.
- 529 (overloaded) errors have a dedicated `consecutive529Errors` counter; 3 consecutive failures trigger `FallbackTriggeredError` to switch models; 529s from non-foreground sources are dropped immediately to prevent capacity cascade amplification.
- Fast mode degradation is layered: short delays (<20s) retain fast mode to preserve prompt cache; long delays trigger at least a 10-minute cooldown switching to the standard model.
- Unattended mode chunks long waits into 30-second heartbeat yields to prevent the host environment from marking the process as idle; `attempt` is clamped to prevent for-loop termination while `persistentAttempt` tracks the real attempt count.
