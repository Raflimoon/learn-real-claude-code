# Four-Level Compression Pipeline: From cache_edits to Full Conversation Summary

> The 200K context window is not an infinite resource. Four independent mechanisms trigger in increasing order of invasiveness: API-level context edits -> microcompact -> autocompact -> full summary. Each level strikes a different balance between preserving information and freeing space.

---

## The Problem

In a coding session, every `Read`, `Bash`, and `Grep` call consumes context window space. A single "read -> edit -> verify" cycle costs about 8,000 tokens. After 50 tool call rounds, the context exceeds 150K tokens.

If left unmanaged, the model hits the context limit, the API returns `prompt_too_long`, and the entire session crashes. The core tension: **old tool results can be discarded (files can be re-read), but the reasoning chain and decisions within the conversation cannot.** The system must know what can be compressed and what must be preserved -- and it must choose different compression granularity for different situations.

---

## Architecture Overview

```
Trigger Condition       Mechanism                Invasiveness  Cache Impact
----------------        -----------------        ----------    ------------------
API-level threshold     context edits (API)      Lowest        No prompt cache impact
Before each request     microcompact             Low           cache_edits preserves cache
~93% context usage      autocompact              High          Rebuilds cache prefix
User manual /compact    full/partial summary     Highest       Complete rebuild
```

---

## Level 1: API-Level Context Management

The lightest strategy does not execute on the client -- it tells the API server to clean up:

```typescript
// services/compact/apiMicrocompact.ts:16-17
const DEFAULT_MAX_INPUT_TOKENS = 180_000      // trigger threshold
const DEFAULT_TARGET_INPUT_TOKENS = 40_000    // post-cleanup target
```

Two server-side strategies: **`clear_thinking_20251015`** clears extended thinking blocks (Opus/Sonnet reasoning traces, potentially thousands of tokens each); **`clear_tool_uses_20250919`** clears tool call blocks. But not all tools are clearable:

```typescript
// services/compact/apiMicrocompact.ts:28-32
const TOOLS_CLEARABLE_USES = [
  FILE_EDIT_TOOL_NAME,    // diff content already written to disk
  FILE_WRITE_TOOL_NAME,   // written content already on disk
  NOTEBOOK_EDIT_TOOL_NAME,
]
```

These tools' inputs can be safely cleared -- their effects have been persisted to the filesystem. The model does not need the complete diff text in context to remember "what I changed."

By contrast, `FILE_READ` and `GREP` results are not in `TOOLS_CLEARABLE_USES` -- their `tool_use` blocks (call parameters) must be preserved because the model needs to know "which file I read" to maintain contextual coherence. The result content (`tool_result`) is handled by client-side microcompact.

## Level 2: Microcompact

Microcompact runs before every API request. The entry function routes to two paths:

```typescript
// services/compact/microCompact.ts:253-293 (simplified)
export async function microcompactMessages(messages, toolUseContext?, querySource?) {
  // Path 1: cache has gone cold (>60 minutes), modify messages directly
  const timeBasedResult = maybeTimeBasedMicrocompact(messages, querySource)
  if (timeBasedResult) return timeBasedResult

  // Path 2: cache still warm, use cache_edits API without breaking cache prefix
  if (feature('CACHED_MICROCOMPACT')) {
    // ...model compatibility checks...
    return await cachedMicrocompactPath(messages, querySource)
  }

  return { messages }  // no viable path, hand off to autocompact
}
```

### Path 1: Time-Triggered

Anthropic's prompt cache TTL is 1 hour. After 60+ minutes of inactivity, the cache has already gone cold -- might as well clean up:

```typescript
// services/compact/timeBasedMCConfig.ts:30-34
const TIME_BASED_MC_CONFIG_DEFAULTS: TimeBasedMCConfig = {
  enabled: false,
  gapThresholdMinutes: 60,    // 60 minutes of inactivity
  keepRecent: 5,              // keep the 5 most recent tool results
}
```

The logic calculates the time since the last assistant message (`microCompact.ts:422-444`); when it exceeds the threshold, old tool results are replaced with `'[Old tool result content cleared]'`. Only results from specific tools are eligible for cleanup:

```typescript
// services/compact/microCompact.ts:41-50
const COMPACTABLE_TOOLS = new Set<string>([
  FILE_READ_TOOL_NAME, ...SHELL_TOOL_NAMES, GREP_TOOL_NAME, GLOB_TOOL_NAME,
  WEB_SEARCH_TOOL_NAME, WEB_FETCH_TOOL_NAME, FILE_EDIT_TOOL_NAME, FILE_WRITE_TOOL_NAME,
])
```

### Path 2: Cached Microcompact

When the cache is still warm, directly modifying messages would break the prompt cache. Cached MC uses the `cache_edits` API to tell the server "delete these tool_result content blocks" while leaving local messages unmodified (`microCompact.ts:305-399`). The key difference -- return original messages, compression instructions go via `compactionInfo`:

```typescript
// services/compact/microCompact.ts:385-394 (key return)
    return {
      messages,  // messages unmodified
      compactionInfo: {
        pendingCacheEdits: {
          trigger: 'auto',
          deletedToolIds: toolsToDelete,
          baselineCacheDeletedTokens: baseline,
        },
      },
    }
```

## Level 3: Autocompact

When microcompact is insufficient to control context bloat, autocompact activates -- forking an agent to perform a **full LLM summarization call**.

### Threshold Calculation

```typescript
// services/compact/autoCompact.ts:62-65
export const AUTOCOMPACT_BUFFER_TOKENS = 13_000
export const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000
export const ERROR_THRESHOLD_BUFFER_TOKENS = 20_000
export const MANUAL_COMPACT_BUFFER_TOKENS = 3_000
```

effectiveWindow = contextWindow - maxOutputTokens (max 20K, based on p99.99 compaction summary output of 17,387 tokens, per `autoCompact.ts:29-30`). Autocompact threshold = effectiveWindow - 13,000. For a 200K model: 180,000 - 13,000 = 167,000, approximately **~93%** usage triggers it.

The environment variable `CLAUDE_CODE_AUTO_COMPACT_WINDOW` can override the context window size (for testing); `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` can set the trigger threshold by percentage.

### Trigger Logic and Recursion Guards

`shouldAutoCompact()` (`autoCompact.ts:160-239`) includes multiple guards: forked agents for compact and session_memory do not trigger (prevents deadlock); Context Collapse mode disables it (avoids races). Source comments confirm the design intent:

```
// autoCompact.ts:201-205
// Context-collapse mode: the 90% commit / 95% blocking-spawn
// flow owns the headroom problem. Autocompact firing at ~93%
// sits right between collapse's commit-start (90%) and blocking (95%),
// so it would race collapse and usually win.
```

### Circuit Breaker

```typescript
// services/compact/autoCompact.ts:67-70
// BQ 2026-03-10: 1,279 sessions had 50+ consecutive failures (up to 3,272)
// in a single session, wasting ~250K API calls/day globally.
const MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3
```

After 3 consecutive failures, retries stop. Without this protection, sessions with irrecoverably oversized contexts would loop endlessly attempting compaction. The failure count resets to 0 on a successful compaction (`autoCompact.ts:332`), so transient failures do not permanently disable compaction.

## Level 4: Full Conversation Summary

`compactConversation()` (`compact.ts:387-749`) initiates a full LLM summary. The post-summary message structure:

```typescript
// services/compact/compact.ts:330-338
export function buildPostCompactMessages(result: CompactionResult): Message[] {
  return [
    result.boundaryMarker,           // compaction boundary marker
    ...result.summaryMessages,       // AI-generated summary
    ...(result.messagesToKeep ?? []),  // preserved messages (partial compaction)
    ...result.attachments,           // file attachments re-injected
    ...result.hookResults,           // hook results
  ]
}
```

### Partial Compaction: Two Directions

`partialCompactConversation()` (`compact.ts:772-998`) supports selective compaction:

- **`'up_to'`**: Summarize messages before the pivot, preserve those after (old becomes summary, new stays verbatim).
- **`'from'`**: Summarize messages after the pivot, preserve those before (preserves the early cache prefix).

The `'up_to'` direction filters out old compaction boundary markers and summary messages (`compact.ts:793-799`); otherwise nested summaries would cause `findLastCompactBoundaryIndex`'s reverse scan to locate the wrong boundary.

Additionally, when the compaction request itself triggers `prompt_too_long` due to oversized context, `truncateHeadForPTLRetry()` (`compact.ts:243-291`) discards the earliest messages grouped by API-round, retrying up to 3 times. This is the last escape hatch -- ensuring users never get stuck in a state where compaction itself cannot run.

### Post-Compaction Cleanup

`runPostCompactCleanup()` (`postCompactCleanup.ts:31-77`) resets a series of caches. A key design distinction -- differentiating main thread from child agents:

```typescript
// services/compact/postCompactCleanup.ts:36-39
  const isMainThreadCompact =
    querySource === undefined ||
    querySource.startsWith('repl_main_thread') ||
    querySource === 'sdk'
```

Child agents (session_memory, prompt_suggestion) share module-level state within the same process. If a child agent's compaction reset the main thread's caches, the next main thread request would lose state.

## Why Not a Single Compression Strategy

**Rejected alternative: use only full summarization.** A single summary call consumes ~20K output tokens, takes about 3 seconds, and destroys the entire prompt cache prefix. Microcompact completes in milliseconds and preserves the cache via `cache_edits`. Design principle: **use the cheap option when possible; preserve the cache when possible.**

## Why Not Compress All Old Content at Once

**Rejected alternative: summarize everything once the threshold is reached.** `cache_edits` lets the server delete specific `tool_result` blocks while the message array's token prefix remains unchanged and the cache keeps hitting. Replacing everything with a summary forces 180K tokens to be billed at cache_creation price (Opus: $3.75/M) instead of cache_read price ($0.30/M).

Time-triggered microcompact first checks whether the cache is warm or cold (`microCompact.ts:261-269` comments): it modifies message content only when the cache is cold; when warm, it uses `cache_edits` for precision deletion.

## Token Budget and Diminishing Returns

The token budget mechanism works alongside the compression system to control when the model should stop:

```typescript
// query/tokenBudget.ts:3-4
const COMPLETION_THRESHOLD = 0.9   // stop when 90% of budget is used
const DIMINISHING_THRESHOLD = 500  // increments < 500 tokens considered diminishing returns
```

Diminishing returns detection -- when 3 consecutive iterations produce fewer than 500 tokens each, early termination is triggered (`tokenBudget.ts:59-62`), preventing the model from spinning idly. This complements the compression system: compression controls input context bloat, while the budget controls output efficiency. Together they ensure long tasks neither blow up the context nor waste tokens meaninglessly.

## Verification

1. **Compare token usage.** In a long conversation, perform 20+ tool calls then run `/cost` to record token consumption. Run `/compact`, then run `/cost` again. A 150K-token conversation typically compresses to 30-50K tokens.
2. **Check summary quality.** After `/compact`, the model's first reply works from the summary. Verify it remembers key decisions and file modifications -- if important context is lost, the summary prompt may need custom instructions (`/compact` supports passing `customInstructions`).
3. **Check compaction logs.** Search for `compactBoundary`-type messages in JSONL files under `~/.claude/sessions/` to locate where compaction occurred.
4. **Observe auto-triggering.** Set `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=50` to lower the threshold, perform a few file reads, and watch for the `Compacting conversation...` prompt.
5. **Compare cache hit rates.** In `/cost`, look at the cache_read vs cache_creation ratio. After `/compact`, the first request shows a spike in cache_creation; subsequent requests recover.

## Key Takeaways

- The four-level pipeline escalates in invasiveness: API-level context edits (zero client cost) -> microcompact (millisecond-level, cache-aware) -> autocompact (~93% trigger, forked agent summary) -> manual `/compact` (maximum compression ratio, breaks cache).
- Microcompact's two paths embody "cache awareness": when the cache is warm, `cache_edits` API keeps local messages unmodified; when the cache is cold (60+ minutes of inactivity), content is replaced directly.
- A circuit breaker limits consecutive autocompact failures to 3 -- production data showed 1,279 sessions wasting 250K API calls per day due to unlimited retries.
- Partial compaction supports two directions: `'up_to'` (summarize old, preserve new) and `'from'` (preserve old, summarize new). The choice of direction determines whether the prompt cache prefix survives.
- Post-compaction cleanup distinguishes main thread from child agents, preventing child agents sharing module-level state from corrupting the main thread's caches.
- Token budget diminishing returns detection (3 consecutive deltas < 500 tokens) prevents the model from spinning idly, complementing the compression system.
