# MCP: The Infinite Extension Protocol

> MCP tools are bridged to native Tool interfaces via `fetchToolsForClient`; ToolSearchTool uses weighted keyword scoring for deferred loading, with automatic threshold triggering when MCP tools exceed 10% of context.

## Overview

MCP (Model Context Protocol) is Claude Code's standard protocol for connecting external tools. But those four words -- "standard protocol" -- mask the engineering complexity in the source: 5 transport types, a complete tool bridging layer, a search algorithm with weighted scoring, and a three-mode deferred loading decision system.

The real MCP implementation is not as simple as "connect server, call tool."

## Tool Bridging: fetchToolsForClient

MCP protocol tool definitions must be converted to Claude Code's internal `Tool` interface. This bridging happens in `fetchToolsForClient`:

```typescript
// services/mcp/client.ts:1743
export const fetchToolsForClient = memoizeWithLRU(
  async (client: MCPServerConnection): Promise<Tool[]> => {
    if (client.type !== 'connected') return []

    const result = (await client.client.request(
      { method: 'tools/list' },
      ListToolsResultSchema,
    )) as ListToolsResult

    const toolsToProcess = recursivelySanitizeUnicode(result.tools)

    return toolsToProcess.map((tool): Tool => {
      const fullyQualifiedName = buildMcpToolName(client.name, tool.name)
      return {
        ...MCPTool,    // Inherit MCPTool's base implementation
        name: skipPrefix ? tool.name : fullyQualifiedName,
        mcpInfo: { serverName: client.name, toolName: tool.name },
        isMcp: true,
        searchHint:
          typeof tool._meta?.['anthropic/searchHint'] === 'string'
            ? tool._meta['anthropic/searchHint'].replace(/\s+/g, ' ').trim()
            : undefined,
        alwaysLoad: tool._meta?.['anthropic/alwaysLoad'] === true,
        async prompt() {
          const desc = tool.description ?? ''
          return desc.length > MAX_MCP_DESCRIPTION_LENGTH
            ? desc.slice(0, MAX_MCP_DESCRIPTION_LENGTH) + '... [truncated]'
            : desc
        },
        isConcurrencySafe() {
          return tool.annotations?.readOnlyHint ?? false
        },
        isDestructive() {
          return tool.annotations?.destructiveHint ?? false
        },
        inputJSONSchema: tool.inputSchema as Tool['inputJSONSchema'],
        // ...
      }
    })
  },
)
```

Key details:

1. **Name normalization**: `buildMcpToolName` generates fully qualified names in `mcp__{server}__{tool}` format.
2. **Description truncation**: `MAX_MCP_DESCRIPTION_LENGTH = 2048` -- OpenAPI-generated MCP servers can produce 15-60KB descriptions that must be truncated.
3. **MCP annotations mapping**: `readOnlyHint` maps to `isConcurrencySafe()`, `destructiveHint` maps to `isDestructive()` -- MCP protocol annotations directly affect permission and concurrency decisions.
4. **searchHint**: MCP servers can provide search keywords via `_meta['anthropic/searchHint']` to improve ToolSearchTool matching accuracy.
5. **alwaysLoad**: `_meta['anthropic/alwaysLoad']` lets tools bypass deferred loading and always appear in the initial prompt.

## MCPTool: The Overridable Skeleton

All MCP tools share a base template `MCPTool`, with key methods overridden during bridging:

```typescript
// tools/MCPTool/MCPTool.ts:27

    return inputSchema()     // passthrough: z.object({}).passthrough()
  },
  async call() { return { data: '' } },  // Overridden
  async checkPermissions(): Promise<PermissionResult> {
    return { behavior: 'passthrough', message: 'MCPTool requires permission.' }
  },
} satisfies ToolDef<InputSchema, Output>)
```

`inputSchema` is `z.object({}).passthrough()` -- it accepts any object. The real schema for MCP tools lives in the `inputJSONSchema` field, passed directly to the model at the API layer.

## ToolSearchTool: Weighted Keyword Search

ToolSearchTool is not simple string matching. It has two search modes and a weighted scoring system.

### Mode 1: select -- Direct Selection

```typescript
// tools/ToolSearchTool/ToolSearchTool.ts:363
const selectMatch = query.match(/^select:(.+)$/i)
if (selectMatch) {
  const requested = selectMatch[1]!
    .split(',').map(s => s.trim()).filter(Boolean)

  const found: string[] = []
  for (const toolName of requested) {
    const tool =
      findToolByName(deferredTools, toolName) ??
      findToolByName(tools, toolName)  // Also searches already-loaded tools
    if (tool) {
      if (!found.includes(tool.name)) found.push(tool.name)
    }
  }
  return buildSearchResult(found, query, deferredTools.length)
}
```

`select:Read,Edit,Grep` -- fetch tools by name directly. Note that it also searches the already-loaded tool list -- selecting an already-loaded tool is a harmless no-op, avoiding model retry loops.

### Mode 2: Keyword Search

```typescript
// tools/ToolSearchTool/ToolSearchTool.ts:186
async function searchToolsWithKeywords(
  query: string, deferredTools: Tools, tools: Tools, maxResults: number,
): Promise<string[]> {
  // Fast path: exact match
  const exactMatch =
    deferredTools.find(t => t.name.toLowerCase() === queryLower) ??
    tools.find(t => t.name.toLowerCase() === queryLower)
  if (exactMatch) return [exactMatch.name]

  // MCP prefix match
  if (queryLower.startsWith('mcp__') && queryLower.length > 5) {
    const prefixMatches = deferredTools
      .filter(t => t.name.toLowerCase().startsWith(queryLower))
    if (prefixMatches.length > 0) return prefixMatches.map(t => t.name)
  }

  // Weighted scoring
  const scored = await Promise.all(
    candidateTools.map(async tool => {
      const parsed = parseToolName(tool.name)
      const description = await getToolDescriptionMemoized(tool.name, tools)
      let score = 0
      for (const term of allScoringTerms) {
        const pattern = termPatterns.get(term)!
        if (parsed.parts.includes(term))
          score += parsed.isMcp ? 12 : 10       // Exact part match
        else if (parsed.parts.some(part => part.includes(term)))
          score += parsed.isMcp ? 6 : 5         // Partial substring match
        if (hintNormalized && pattern.test(hintNormalized))
          score += 4                             // searchHint match
        if (pattern.test(descNormalized))
          score += 2                             // Description match
      }
      return { name: tool.name, score }
    }),
  )

  return scored.filter(i => i.score > 0)
    .sort((a, b) => b.score - a.score)
    .slice(0, maxResults).map(i => i.name)
}
```

Scoring weights:

| Match Type | MCP Tools | Native Tools |
|---------|---------|---------|
| Name part exact match | 12 | 10 |
| Name substring match | 6 | 5 |
| searchHint word boundary match | 4 | 4 |
| Description word boundary match | 2 | 2 |

MCP tools get higher name match weights because the server name portion of the `mcp__server__action` format is the strongest intent signal.

## Return Format: tool_reference

ToolSearchTool's return value is not JSON text but a `tool_reference` content block -- a first-class citizen at the API level:

```typescript
// tools/ToolSearchTool/ToolSearchTool.ts:444
mapToolResultToToolResultBlockParam(content: Output, toolUseID: string) {
  if (content.matches.length === 0) {
    return { type: 'tool_result', tool_use_id: toolUseID,
      content: 'No matching deferred tools found' }
  }
  return {
    type: 'tool_result', tool_use_id: toolUseID,
    content: content.matches.map(name => ({
      type: 'tool_reference' as const,
      tool_name: name,
    })),
  } as unknown as ToolResultBlockParam
}
```

`tool_reference` tells the API: "inject this tool's full schema into the model's context." This is not a JSON string for the model to parse -- it is a server-side reference expansion executed by the API.

## Deferred Loading Decision: The Three-Mode System

Deferred loading is not a simple on/off switch but three modes:

```typescript
// utils/toolSearch.ts:161
export type ToolSearchMode = 'tst' | 'tst-auto' | 'standard'

// utils/toolSearch.ts:172
export function getToolSearchMode(): ToolSearchMode {
  if (isEnvTruthy(process.env.CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS))
    return 'standard'

  const value = process.env.ENABLE_TOOL_SEARCH
  if (isAutoToolSearchMode(value)) return 'tst-auto'
  if (isEnvTruthy(value)) return 'tst'
  if (isEnvDefinedFalsy(process.env.ENABLE_TOOL_SEARCH)) return 'standard'
  return 'tst'  // default: always defer
}
```

- **tst**: Always defer (default behavior).
- **tst-auto**: Automatically enable when MCP tool descriptions exceed N% of the context window.
- **standard**: No deferral, all tools loaded directly.

The `tst-auto` threshold calculation:

```typescript
// utils/toolSearch.ts:49
const DEFAULT_AUTO_TOOL_SEARCH_PERCENTAGE = 10 // 10%

// utils/toolSearch.ts:104
function getAutoToolSearchTokenThreshold(model: string): number {
  const betas = getMergedBetas(model)
  const contextWindow = getContextWindowForModel(model, betas)
  const percentage = getAutoToolSearchPercentage() / 100
  return Math.floor(contextWindow * percentage)
}
```

Default 10% -- for a model with a 200K token context window, the threshold is 20K tokens. Once exceeded, deferred loading is automatically enabled.

## Which Tools Are Never Deferred

The `isDeferredTool` function contains a series of never-defer exceptions:

```typescript
// tools/ToolSearchTool/prompt.ts:62
export function isDeferredTool(tool: Tool): boolean {
  if (tool.alwaysLoad === true) return false         // MCP _meta opt-out
  if (tool.isMcp === true) return true               // MCP tools always deferred
  if (tool.name === TOOL_SEARCH_TOOL_NAME) return false  // Cannot defer itself
  if (feature('FORK_SUBAGENT') && tool.name === AGENT_TOOL_NAME) {
    if (m.isForkSubagentEnabled()) return false      // Agent not deferred in fork mode
  }
  return tool.shouldDefer === true
}
```

ToolSearchTool is never deferred (otherwise who discovers it?), AgentTool is not deferred in fork mode (needs to be available on the first turn), and MCP tools can opt out via `_meta['anthropic/alwaysLoad']`.

## Compaction: Protecting Tool Discovery

Deferred loading introduces a challenge: when compaction deletes old messages, `tool_reference` blocks are also deleted. The source uses two strategies for protection:

```typescript
// utils/toolSearch.ts:545
export function extractDiscoveredToolNames(messages: Message[]): Set<string> {
  const discoveredTools = new Set<string>()

  for (const msg of messages) {
    // Compact boundary carries the pre-compact discovered set
    if (msg.type === 'system' && msg.subtype === 'compact_boundary') {
      const carried = msg.compactMetadata?.preCompactDiscoveredTools
      if (carried) {
        for (const name of carried) discoveredTools.add(name)
      }
      continue
    }
    // Scan user messages for tool_reference blocks
    if (msg.type !== 'user') continue
    for (const block of content) {
      if (isToolResultBlockWithContent(block)) {
        for (const item of block.content) {
          if (isToolReferenceWithName(item))
            discoveredTools.add(item.tool_name)
        }
      }
    }
  }
  return discoveredTools
}
```

Before compaction, discovered tool names are snapshotted into `compactMetadata.preCompactDiscoveredTools`; after restoration, they are read back from the compact boundary message.

## Connection Management: Batched Concurrency + Exponential Backoff

MCP server connections are not established one at a time but in batches:

```typescript
// services/mcp/client.ts:552
export function getMcpServerConnectionBatchSize(): number {
  return parseInt(process.env.MCP_SERVER_CONNECTION_BATCH_SIZE || '', 10) || 3
}

function getRemoteMcpServerConnectionBatchSize(): number {
  return parseInt(process.env.MCP_REMOTE_SERVER_CONNECTION_BATCH_SIZE || '', 10) || 20
}
```

Local stdio servers default to 3 concurrent connections, remote HTTP/SSE servers default to 20. Failed connections use exponential backoff for reconnection:

```typescript
// services/mcp/useManageMCPConnections.ts:88
const MAX_RECONNECT_ATTEMPTS = 5
const INITIAL_BACKOFF_MS = 1000
const MAX_BACKOFF_MS = 30000
```

5 retries, 1 second to 30 second backoff.

## Why Not a Unified GraphQL/REST API?

A unified API gateway might seem cleaner, but MCP chose per-server independent connections:

1. **Transport heterogeneity**: stdio child processes, SSE streams, WebSocket bidirectional channels, in-process SDK -- these transports have fundamentally different semantics; a unified API would erase each transport's advantages.
2. **Independent lifecycles**: One MCP server crashing should not affect others. Independent connections mean independent reconnection and error handling.
3. **Authentication isolation**: Different servers may need different OAuth flows, API keys, or no authentication at all (like IDE servers); a unified gateway's auth model becomes the lowest common denominator.

## Why Not Preload All Tools?

In theory, a 200K context window is large enough -- why not load everything?

1. **Token economics**: The 10% threshold (default 20K tokens) is not trivial -- these tokens are repeated with every API call, and even with cache hits, cost and latency still grow linearly.
2. **Inter-tool interference**: Large numbers of tool descriptions in the prompt compete for model attention. Experiments show that when tools exceed 40, the model's accuracy in selecting the correct tool decreases.
3. **Haiku incompatibility**: `tool_reference` is a beta API feature that Haiku models do not support; deferred loading automatically falls back to standard mode on Haiku.

## Key Takeaways

- MCP tools are bridged to native `Tool` interfaces via `fetchToolsForClient`, including name normalization (`mcp__{server}__{tool}`), description truncation (2048 character limit), and MCP annotations-to-permission/concurrency property mapping.
- ToolSearchTool has two modes (select direct selection / keyword search) and a weighted scoring system; MCP tool name exact match scores 12 points, description word boundary match scores 2 points, and `+` prefix marks keywords as required.
- Deferred loading has three modes (tst/tst-auto/standard); `tst-auto` triggers automatically when MCP tool descriptions exceed 10% of the context window; the default behavior is always defer (tst).
- `tool_reference` is a first-class citizen at the API level, not a JSON string for the model to parse -- the API server expands references into full tool schemas injected into the model context.
- Compaction protects discovered tool names via `preCompactDiscoveredTools` snapshots, preventing `tool_reference` loss during message compression that would cause tools to become unreachable.
- MCP server connections are established in batches (local 3 concurrent / remote 20 concurrent), with exponential backoff reconnection on failure (1s-30s, max 5 attempts).
