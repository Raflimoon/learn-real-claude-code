# Context Engineering: The 2026 Paradigm Shift

> Every token threshold constant in the source encodes an engineering decision about attention decay and cache cost.

## Overview

The AI application development paradigm of 2024 was "prompt engineering": write clearer instructions, better role definitions, more precise constraints. By 2026, the key factor determining agent output quality has shifted: from "what static prompt you wrote" to "what is in the model's context window on each call." This paradigm shift is called **Context Engineering**.

Claude Code's source is the most complete industrial implementation of this paradigm. But community discussion usually stops at "it has a compression feature." This chapter starts from three numerical constants in the source to reveal the real engineering tradeoffs of Context Engineering.

## Source Insight 1: The 13,000-Token Buffer -- Why Not 5,000 or 30,000?

`autoCompact.ts:62-65` defines four key constants:

```typescript
// autoCompact.ts:62-65
export const AUTOCOMPACT_BUFFER_TOKENS = 13_000
export const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000
export const ERROR_THRESHOLD_BUFFER_TOKENS = 20_000
export const MANUAL_COMPACT_BUFFER_TOKENS = 3_000
```

`AUTOCOMPACT_BUFFER_TOKENS = 13_000` means: in a 200K context window, automatic compaction triggers when token usage reaches `effectiveWindow - 13,000` (`autoCompact.ts:72-91`). This number encodes a precise tradeoff:

**Why not 5,000?** A 5K buffer means only ~2.5% headroom. A single tool call result (e.g., `Read` loading a large file) can consume 3,000-8,000 tokens. If there is one round of tool calls between compaction trigger and actual execution, a 5K buffer is easily breached, causing the API to return a `prompt_too_long` error.

**Why not 30,000?** A 30K buffer (15% headroom) means compaction fires too early every time, discarding conversation history that may still be useful. Also note the comment at `autoCompact.ts:29-30`:

```typescript
// autoCompact.ts:29-30
// Reserve this many tokens for output during compaction
// Based on p99.99 of compact summary output being 17,387 tokens.
const MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20_000
```

The compaction operation itself reserves 20K output tokens (based on p99.99 statistics). `effectiveWindow = contextWindow - 20,000`. So the actual auto-compaction trigger point is `200,000 - 20,000 - 13,000 = 167,000` tokens -- about 83.5% of the context window.

13K is a product of statistics and engineering compromise: enough to cover a single-turn tool call spike, yet not so much that it wastes usable context.

### Why Not a Percentage Threshold?

Community suggestions often propose "compact at 80%." But percentages ignore a critical fact: the same percentage corresponds to completely different absolute space under different context window sizes (200K vs 1M). 13K is an absolute value -- regardless of context window size, the tokens consumed by one large file read are fixed. The source uses `getEffectiveContextWindowSize()` for dynamic calculation (`autoCompact.ts:33-49`), but the buffer itself is a fixed absolute value.

### Why Not Check Before Each API Call Instead of Using a Fixed Threshold?

Because compaction is not free. `compactConversation()` is itself an API call, consuming time and money. Checking before every API call adds latency. A fixed threshold combined with a circuit breaker (`autoCompact.ts:67-70`, stops retrying after 3 consecutive failures) is a more efficient strategy.

## Source Insight 2: Where CLAUDE.md Ends -- The Cost Implications of Cache Boundaries

`prompts.ts:105-115` defines a critical boundary marker:

```typescript
// prompts.ts:113-115
export const SYSTEM_PROMPT_DYNAMIC_BOUNDARY =
  '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'
```

The comment (`prompts.ts:105-112`) explicitly states: this marker splits the system prompt into two parts -- a **static part** (cacheable across organizations, scope: 'global') and a **dynamic part** (unique per session, not cacheable).

`prompts.ts:560-576` shows the full assembly logic:

```typescript
// prompts.ts:560-576
return [
  // --- Static content (cacheable) ---
  getSimpleIntroSection(outputStyleConfig),
  getSimpleSystemSection(),
  getSimpleDoingTasksSection(),
  getActionsSection(),
  getUsingYourToolsSection(enabledTools),
  getSimpleToneAndStyleSection(),
  getOutputEfficiencySection(),
  // === BOUNDARY MARKER - DO NOT MOVE OR REMOVE ===
  ...(shouldUseGlobalCacheScope() ? [SYSTEM_PROMPT_DYNAMIC_BOUNDARY] : []),
  // --- Dynamic content (registry-managed) ---
  ...resolvedDynamicSections,
].filter(s => s !== null)
```

**Where is the CLAUDE.md content?** It is loaded via `loadMemoryPrompt()` (`prompts.ts:495`) and belongs to the `dynamicSections` array -- meaning CLAUDE.md is after the boundary marker, in the **dynamic/non-cached region**.

This means: each user's CLAUDE.md content differs, so it cannot be globally cached. Modifying CLAUDE.md does not break the static part's prompt cache. But at the same time, every token of CLAUDE.md is a "cold" token on every API call -- charged at full price.

### systemPromptSection vs DANGEROUS_uncachedSystemPromptSection

`systemPromptSections.ts:20-38` defines two section types:

```typescript
// systemPromptSections.ts:20-25
export function systemPromptSection(
  name: string, compute: ComputeFn,
): SystemPromptSection {
  return { name, compute, cacheBreak: false }
}

// systemPromptSections.ts:32-38
export function DANGEROUS_uncachedSystemPromptSection(
  name: string, compute: ComputeFn, _reason: string,
): SystemPromptSection {
  return { name, compute, cacheBreak: true }
}
```

`systemPromptSection` computes once and caches until `/clear` or `/compact`. `DANGEROUS_uncachedSystemPromptSection` recomputes every turn -- breaking the prompt cache. Only one place in the source uses `DANGEROUS_`: MCP server instructions (`prompts.ts:513-519`), because MCP servers can connect/disconnect between turns.

### Why Not Put Everything in the Static Region?

Because prompt cache matches by prefix. If any dynamic byte changes in the prefix (username, environment path, date), the entire cache misses. Concentrating stable content in the prefix and pushing dynamic content to the end maximizes cache hit rate.

## Source Insight 3: 180K / 40K -- Microcompact's Safety Margin Design

`apiMicrocompact.ts:14-17` defines the API-based microcompact thresholds:

```typescript
// apiMicrocompact.ts:15-16
const DEFAULT_MAX_INPUT_TOKENS = 180_000  // Typical warning threshold
const DEFAULT_TARGET_INPUT_TOKENS = 40_000 // Keep last 40k tokens like client-side
```

180K trigger / 40K retain in a 200K context window means:

- **Trigger timing**: When input tokens reach 180K (90% of context). This is later than autocompact's ~167K -- microcompact supplements autocompact, not replaces it.
- **Retain target**: Keep only the most recent 40K tokens (20%), clearing `180K - 40K = 140K` tokens of tool results.
- **Safety margin**: 200K - 180K = 20K tokens -- exactly equal to `MAX_OUTPUT_TOKENS_FOR_SUMMARY` (`autoCompact.ts:30`). This is not coincidence: the system needs to reserve space for model output.

`apiMicrocompact.ts:19-33` defines two categories of clearable tools:

```typescript
// apiMicrocompact.ts:19-26
const TOOLS_CLEARABLE_RESULTS = [
  ...SHELL_TOOL_NAMES, GLOB_TOOL_NAME, GREP_TOOL_NAME,
  FILE_READ_TOOL_NAME, WEB_FETCH_TOOL_NAME, WEB_SEARCH_TOOL_NAME,
]

const TOOLS_CLEARABLE_USES = [
  FILE_EDIT_TOOL_NAME, FILE_WRITE_TOOL_NAME, NOTEBOOK_EDIT_TOOL_NAME,
]
```

Read operation results (Bash/Read/Grep/Glob) can be safely cleared -- the model has already extracted information from them. Write operation results (Edit/Write) can also be cleared -- write return values are typically just confirmation messages.

### Why Not a 50K Retain Target?

40K is approximately "the last 2-3 complete interaction turns." More retention means less history cleared, weaker per-microcompact effect, and potentially more frequent triggers. 40K balances "retaining enough context for the model to continue working" against "clearing enough space for the conversation to continue."

## Cross-Chapter Connections: How Three Systems Cooperate for Context Engineering

Context Engineering in Claude Code is not a single module but the cooperation of at least three subsystems:

**1. compact.ts (chapter 06) -- Conversation-level compression**
When token usage exceeds `effectiveWindow - 13,000`, `autoCompactIfNeeded()` triggers. It first attempts session memory compaction (lightweight, reusing existing summaries), falling back to full `compactConversation()` (forking a sub-agent to generate summaries). A circuit breaker ensures no more than 3 consecutive failures (`autoCompact.ts:70`, with comments citing BQ data: 1,279 sessions had 50+ consecutive failures, wasting ~250K API calls per day).

**2. systemPromptSections.ts (chapter 05) -- Prompt-level cache management**
The cache boundary splits the system prompt into a static prefix and dynamic suffix. The static prefix (7 sections, ~5,000-8,000 tokens) is cached at global scope. The dynamic suffix (CLAUDE.md, MCP instructions, environment info, etc.) recomputes each time but mostly avoids repeated computation through `systemPromptSection`'s internal caching. Only MCP instructions use `DANGEROUS_uncachedSystemPromptSection` -- and even then, the source is migrating to incremental attachments via `isMcpInstructionsDeltaEnabled()` (`prompts.ts:510-511`), further reducing prompt cache breakage.

**3. ToolSearchTool (chapter 11) -- Tool definition-level deferred loading**
`ToolSearchTool/prompt.ts:62-108`'s `isDeferredTool()` function determines which tools are deferred. MCP tools default to fully deferred (`tool.isMcp === true`). This directly reduces the tool definition tokens per API call -- in scenarios with 40+ MCP tools, this can save thousands of tokens.

The cooperative relationship: ToolSearchTool reduces tool definition tokens per call, systemPromptSections reduces prompt cache invalidation frequency, and compact manages conversation history volume. They each act on different components of the context window.

Note that `isDeferredTool()` has explicit exemption lists (`ToolSearchTool/prompt.ts:70-107`): ToolSearchTool itself is never deferred (otherwise no tool can be discovered), Agent tool is not deferred in fork-subagent mode (needs to be available on the first turn), and Brief tool is not deferred in KAIROS mode (it is the primary communication channel). These exemption rules embody a core principle: **deferred loading boundaries are determined by the communication critical path, not simply by usage frequency.**

Equally important, the execution timing of autocompact in query.ts (detailed in chapter p02) also considers the interaction of these three systems. `query.ts:440-447` shows that context collapse runs **before** autocompact -- if collapse already reduces token usage below the threshold, autocompact does not need to trigger, preserving more fine-grained context instead of generating a coarse-grained summary. This ordering dependency is a product of hand-written orchestration; framework abstractions cannot discover it automatically.

```
Context Window 200K tokens
+-- System Prompt (~8K)     <-- systemPromptSections manages caching
+-- Tool Definitions (~2-5K) <-- ToolSearchTool deferred loading reduces footprint
+-- Conversation History     <-- compact + microcompact manage volume
+-- Safety Buffer (13K)      <-- autoCompact's fixed buffer
```

## Key Takeaways

- The 13,000-token buffer is not an arbitrary number: it is the statistical upper bound of "maximum token consumption in a single tool call turn," combined with a 20K output reserve and a 3-consecutive-failure circuit breaker, forming a precise engineering design based on production data (p99.99 = 17,387 tokens).
- CLAUDE.md content sits in the dynamic region after the prompt cache boundary, meaning every one of its tokens is charged at full price -- keeping CLAUDE.md concise is not just best practice but direct cost optimization.
- The 180K/40K microcompact thresholds form a two-tier defense with autocompact's 167K threshold: autocompact fires at 83.5% (proactive compression), microcompact fires at 90% (clearing tool results) -- they complement rather than replace each other.
- Context Engineering in Claude Code is not a single module but the cooperation of three subsystems: compact (conversation history management), systemPromptSections (prompt cache optimization), and ToolSearchTool (tool definition deferred loading), each acting on different components of the context window.
- The naming of `DANGEROUS_uncachedSystemPromptSection` is itself a cultural artifact of Context Engineering: the word "DANGEROUS" in the function name is an explicit warning about the cost consequence of "per-turn recomputation breaks the prompt cache."
