# Chapter 2 -- Agent Loop: The Streaming Tool Execution Engine in 1,729-Line query.ts

> StreamingToolExecutor starts tool execution mid-stream while the model is still generating its response. A 530-line custom scheduler uses concurrency-safety checks, error cascading, and Promise.race to achieve zero-dependency pipelined parallelism.

---

## Overview

Most agent tutorials show a loop like: send message -> wait for complete response -> parse tool calls -> execute sequentially -> append results -> repeat. Each stage waits for the previous one to finish, and latency compounds at every step.

Claude Code's `query.ts` (1,729 lines) and `StreamingToolExecutor.ts` (530 lines) form a fundamentally different engine. As streaming response bytes arrive, the parser is already looking for `tool_use` blocks; the moment a tool's parameters finish streaming, concurrency safety is assessed and execution starts -- while the model may still be generating the next tool call.

All code blocks in this chapter come from real source code, annotated with filenames and line numbers.

---

## 1. while(true): The Loop That Never Exits on Its Own

The core of `queryLoop` is a state-machine-style `while (true)` loop that only exits when the model stops requesting tools:

```typescript
// query.ts:306-311
while (true) {
  let { toolUseContext } = state
  const {
    messages, autoCompactTracking, maxOutputTokensRecoveryCount,
    hasAttemptedReactiveCompact, maxOutputTokensOverride,
    pendingToolUseSummary, stopHookActive, turnCount,
  } = state
```

`toolUseContext` uses `let` (can be reassigned within the iteration); all other fields use `const` (read-only within the iteration). The loop contains 7 `continue` sites, each writing a complete state snapshot via `state = { ... }`, eliminating the risk of missing one of 9 independent variable assignments.

## 2. StreamingToolExecutor Initialization: Feature-Gate Controlled

At the start of each iteration, a feature gate determines whether streaming tool execution is enabled:

```typescript

const useStreamingToolExecution = config.gates.streamingToolExecution
```

This is not a decorative flag. When disabled, the entire streaming execution path is skipped and the system falls back to sequential execution. Anthropic can roll back with a single toggle -- no new release required.

## 3. API Streaming Call and Mid-Stream Injection

The loop's core is a `for await` iterating over the model's streaming response:

```typescript
// query.ts:659-664
for await (const message of deps.callModel({
  messages: prependUserContext(messagesForQuery, userContext),
  systemPrompt: fullSystemPrompt,
  tools: toolUseContext.options.tools,
  signal: toolUseContext.abortController.signal,
  // ...
```

Whenever an assistant message arrives containing `tool_use` blocks, it is immediately injected into the executor:

```typescript
// query.ts:841-843
for (const toolBlock of msgToolUseBlocks) {
  streamingToolExecutor.addTool(toolBlock, message)
}
```

Immediately after, within the same streaming loop, completed results are harvested non-blockingly:

```typescript
// query.ts:851-856
for (const result of streamingToolExecutor.getCompletedResults()) {
  if (result.message) { yield result.message }
}
```

The timing implication: **while the model is still generating the third tool call's parameters, the first tool may have already finished executing and its results may have already been yielded to the UI.**

```
Model stream: --[text]--[tool_A done]--[tool_B done]--[tool_C done]--[end]-->
                   |              |              |
              addTool(A)    addTool(B)     addTool(C)
              execute now   check safety   queue for later
                   |        parallel ok         |
              getCompleted() getCompleted() getRemainingResults()
              yield A result yield B result yield C result
```

### Why Not Wait for the Full Response Before Executing Tools?

Waiting for `message_stop` means all tool latency stacks up after the stream ends. Suppose the model takes 2s generating three tool calls, and each tool takes 500ms to execute:

- **Wait for full response**: 2s (stream) + 1.5s (sequential execution) = 3.5s
- **Mid-stream execution**: 2s (stream) + ~0s (execution overlaps with stream) = ~2s

Tool execution overlaps with stream generation, cutting user-perceived latency by 40-60%.

## 4. StreamingToolExecutor: 530-Line Concurrency Scheduler

### 4.1 TrackedTool State Machine

Each tool passes through four states: `queued` -> `executing` -> `completed` -> `yielded`.

```typescript
// StreamingToolExecutor.ts:20-31
type TrackedTool = {
  id: string
  block: ToolUseBlock
  assistantMessage: AssistantMessage
  status: ToolStatus
  isConcurrencySafe: boolean
  promise?: Promise<void>
  results?: Message[]
  pendingProgress: Message[]        // progress messages stored separately, yielded immediately
  contextModifiers?: Array<(context: ToolUseContext) => ToolUseContext>
}
```

### 4.2 Concurrency Safety Assessment

`addTool` parses the input parameters and then calls the tool's own `isConcurrencySafe` method:

```typescript
// StreamingToolExecutor.ts:104-113
const parsedInput = toolDefinition.inputSchema.safeParse(block.input)
const isConcurrencySafe = parsedInput?.success
  ? (() => {
      try { return Boolean(toolDefinition.isConcurrencySafe(parsedInput.data)) }
      catch { return false }
    })()
  : false
```

Double defense: if `safeParse` fails, the tool is immediately marked unsafe; if `isConcurrencySafe` throws, it also falls back to `false`. Safety is declared by the tool itself rather than guessed by the scheduler -- Read is safe for files, Bash is unsafe for write commands. Only the tool knows its own semantics.

### 4.3 Scheduling Logic: Three Rules

```typescript
// StreamingToolExecutor.ts:129-135
private canExecuteTool(isConcurrencySafe: boolean): boolean {
  const executingTools = this.tools.filter(t => t.status === 'executing')
  return (
    executingTools.length === 0 ||
    (isConcurrencySafe && executingTools.every(t => t.isConcurrencySafe))
  )
}
```

1. **No tools currently executing** -> start immediately.
2. **All executing tools are concurrency-safe** -> run in parallel.
3. **Otherwise** -> block and wait.

Queue processing further guarantees ordering for non-concurrent tools:

```typescript
// StreamingToolExecutor.ts:140-151
private async processQueue(): Promise<void> {
  for (const tool of this.tools) {
    if (tool.status !== 'queued') continue
    if (this.canExecuteTool(tool.isConcurrencySafe)) {
      await this.executeTool(tool)
    } else {
      if (!tool.isConcurrencySafe) break  // non-concurrent tool: block all subsequent tools
    }
  }
}
```

The key is `break`: when a non-executable, non-concurrent tool is encountered, the loop exits immediately -- even if subsequent concurrency-safe tools could run. This guarantees strict ordering.

### Why Not Run All Tools in Parallel?

Reading five files can be parallelized, but `mkdir -p /tmp/foo` followed by `echo hello > /tmp/foo/bar.txt` has an implicit dependency. Bash tools executing write operations may have dependency chains; blind parallelism leads to non-deterministic results. `isConcurrencySafe` lets tools declare their concurrency semantics while the scheduler only makes execution decisions -- never semantic guesses.

## 5. Error Cascading: Bash-Only

Not all tool errors trigger cascading cancellation:

```typescript
// StreamingToolExecutor.ts:356-363
// Only Bash errors cancel siblings. Bash commands often have implicit
// dependency chains (e.g. mkdir fails -> subsequent commands pointless).
// Read/WebFetch/etc are independent -- one failure shouldn't nuke the rest.
if (tool.block.name === BASH_TOOL_NAME) {
  this.hasErrored = true
  this.erroredToolDescription = this.getToolDescription(tool)
  this.siblingAbortController.abort('sibling_error')  // cascade cancel siblings
}
```

`siblingAbortController` is a child controller of `toolUseContext.abortController` (line 58-61). Canceling siblings does not cancel the parent controller -- `query.ts`'s main loop does not end the current turn because of this; it simply receives the error results and continues.

## 6. Two Result Harvesters: Synchronous and Asynchronous

### 6.1 getCompletedResults(): Synchronous Mid-Stream Harvest

```typescript
// StreamingToolExecutor.ts:412-440
*getCompletedResults(): Generator<MessageUpdate, void> {
  for (const tool of this.tools) {
    while (tool.pendingProgress.length > 0) {           // yield progress messages immediately
      yield { message: tool.pendingProgress.shift()! }
    }
    if (tool.status === 'yielded') continue
    if (tool.status === 'completed' && tool.results) {
      tool.status = 'yielded'
      for (const message of tool.results) { yield { message } }
    } else if (tool.status === 'executing' && !tool.isConcurrencySafe) {
      break  // non-concurrent tool not done: block subsequent results to preserve causal order
    }
  }
}
```

A synchronous Generator (`*` not `async *`), completely non-blocking. The key is the final `break`: if the model called `bash mkdir` then `read file.txt`, even if read finishes first, mkdir's results must be yielded first.

### 6.2 getRemainingResults(): Async Wait After Stream Ends

```typescript
// StreamingToolExecutor.ts:453-490
async *getRemainingResults(): AsyncGenerator<MessageUpdate, void> {
  while (this.hasUnfinishedTools()) {
    await this.processQueue()
    for (const result of this.getCompletedResults()) { yield result }

    if (this.hasExecutingTools() && !this.hasCompletedResults()) {
      const executingPromises = this.tools
        .filter(t => t.status === 'executing' && t.promise).map(t => t.promise!)
      const progressPromise = new Promise<void>(resolve => {
        this.progressAvailableResolve = resolve
      })
      if (executingPromises.length > 0) {
        await Promise.race([...executingPromises, progressPromise])
      }
    }
  }
}
```

`Promise.race` simultaneously awaits tool completion and progress updates. Whichever event fires first triggers another harvest pass, allowing the UI to display progress messages in real time.

### Why Not Use an Off-the-Shelf Task Queue Library?

p-queue, bull, workerpool are all mature libraries, but StreamingToolExecutor's semantics are too specialized: concurrency safety is declared by the tools themselves, Bash errors cascade while Read errors do not, results must be yielded in causal order, and progress messages must bypass the queue immediately. These semantics are deeply interleaved. Composing them from a generic library's hook system would be more complex than the custom implementation. 530 lines, zero external dependencies, fully under control.

## 7. Dual-Path Dispatch and State Continuation

After the streaming response ends, a ternary expression selects the execution path:

```typescript
// query.ts:1380-1382
const toolUpdates = streamingToolExecutor
  ? streamingToolExecutor.getRemainingResults()   // async generator
  : runTools(toolUseBlocks, assistantMessages, canUseTool, toolUseContext)  // sequential fallback
```

Both paths produce the same `AsyncGenerator<MessageUpdate>`, so downstream code does not need to know the execution mode.

After tool execution completes, results are appended to the message history and the next iteration's state is constructed:

```typescript
// query.ts:1715-1728
state = {
  messages: [...messagesForQuery, ...assistantMessages, ...toolResults],
  toolUseContext: toolUseContextWithQueryTracking,
  autoCompactTracking: tracking,
  turnCount: nextTurnCount,
  // ...
  transition: { reason: 'next_turn' },
}
} // while (true)
```

Control flow returns to the top of the loop, destructures the new state, and fires a new API request. The message history grows continuously until the model stops making tool calls or the token budget is exhausted.

## 8. Token Budget: The In-Loop Life Extension Mechanism

The `TOKEN_BUDGET` feature checks token usage before the end of each loop iteration:

```typescript
// query.ts:1308-1340
if (feature('TOKEN_BUDGET')) {
  const decision = checkTokenBudget(budgetTracker!, ...)
  if (decision.action === 'continue') {
    state = {
      messages: [...messagesForQuery, ...assistantMessages,
        createUserMessage({ content: decision.nudgeMessage, isMeta: true }),
      ],
      // ...
      transition: { reason: 'token_budget_continuation' },
    }
    continue  // <-- jumps back to while(true) at line 307
  }
}
```

A `isMeta: true` user message is injected, then `continue` jumps back to the top of the loop. The model sees this message and continues working, unaware that it was system-injected -- a transparent "life extension" mechanism.

## Verification

**Mid-stream execution**: Open Claude Code and request "read these five files and tell me what they have in common." Watch the terminal -- file read results appear while the response is still streaming. This is `getCompletedResults()` being called within the streaming loop.

**Parallel execution**: Ask Claude Code to read 5 different files simultaneously. All files begin reading and return nearly simultaneously -- Read tools declare `isConcurrencySafe = true`, and the scheduler permits parallelism.

**Sequential scheduling**: Ask Claude Code to execute a bash command and a file read. Bash is not concurrency-safe, so the file read only starts after bash completes -- the scheduler `break`s the queue upon encountering a non-concurrent tool.

**Error cascading**: Ask Claude Code to run two bash commands in parallel, with one deliberately failing. The failing command triggers `siblingAbortController.abort('sibling_error')`, and the other receives a synthetic error rather than continuing execution.

## Key Takeaways

- The `while (true)` loop in `query.ts` (line 307) manages cross-iteration state with a single `State` object. Its 7 `continue` sites each write a complete state snapshot, eliminating the risk of forgetting one of many individual variable assignments.
- `StreamingToolExecutor` harvests tool results in two phases: mid-stream via the synchronous non-blocking `getCompletedResults()`, and post-stream via the async `Promise.race`-based `getRemainingResults()`, maximizing parallelism between tool execution and model generation.
- Concurrency safety is declared by each tool via `isConcurrencySafe()`. The scheduler applies three rules: start immediately if nothing is executing; run in parallel if all are concurrency-safe; block otherwise. Upon encountering a non-executable, non-concurrent tool, it `break`s the queue to preserve causal ordering.
- Error cascading is limited to Bash tools: Bash commands have implicit dependency chains, so failures cascade via a child `AbortController` to cancel siblings. Read/WebFetch and other independent tools' failures do not affect other tools.
- Token budget checks are embedded at the end of the loop, injecting an `isMeta: true` nudge message that lets the model transparently continue work. `continue` jumps back to the top of the loop for "life extension."
- Dual-path dispatch (line 1380-1382) lets streaming execution and sequential fallback share the same `AsyncGenerator<MessageUpdate>` interface, so downstream code is agnostic to execution mode.
- The 530-line custom scheduler has zero external dependencies because the scheduling semantics (tool-declared concurrency safety, Bash-only error cascading, causal-order yielding, progress message bypass) are too specialized for generic task queue libraries to accommodate directly.
