# 真正的 Agent Loop

> StreamingToolExecutor 在流式响应中途就触发工具，无需等待完整响应。

## 概述

大多数 agent 教程展示的是这样的循环：发送消息 → 等待完整响应 → 解析工具调用 → 执行工具 → 追加结果 → 重复。这个模式简单易懂，但它在等待上浪费了大量时间。

Claude Code 的 `query.ts` 实现了一个截然不同的循环。它不等待完整响应——当流式响应的字节到达时，解析器就在寻找工具调用的开始标记，一旦找到，工具就立即开始执行，而模型仍在生成剩余的响应文本。

理解这个机制，能帮你构建出延迟更低、用户体验更流畅的 agent 系统。

## query.ts：核心的 while(true) 循环

agent 循环的骨架位于 `query.ts`：

```typescript
// query.ts（概念性重现）
async function* query(
  messages: Message[],
  tools: Tool[],
  options: QueryOptions
): AsyncGenerator<AssistantEvent> {
  while (true) {
    // 发起流式 API 请求
    const stream = await anthropic.messages.stream({
      messages,
      tools: tools.map(t => t.schema),
      system: buildSystemPrompt(),
    });

    // StreamingToolExecutor 处理流
    const executor = new StreamingToolExecutor(tools);

    for await (const event of stream) {
      yield* executor.process(event);
    }

    const result = executor.getResult();

    // 如果模型没有调用工具，循环结束
    if (!result.hasToolCalls) break;

    // 将工具结果追加到消息历史，继续循环
    messages = [...messages, result.assistantMessage, result.toolResults];

    // 检查 context 使用率，必要时触发压缩
    if (await shouldAutoCompact(messages)) {
      messages = await autoCompact(messages);
    }
  }
}
```

这是一个真正的 `while(true)` 循环，不是简单的请求/响应。循环只有在模型不再调用工具时才退出。每一次迭代都可能触发多个工具调用，每个工具调用的结果又成为下一次 API 请求的输入。

## StreamingToolExecutor：流中的并行执行

这是 Claude Code 最精妙的工程决策之一。教科书版本的工具执行等待完整响应：

```
教科书版本：
Model Response ──────────────────────────────► [完整响应]
                                                    │
                                              解析工具调用
                                                    │
                                              执行工具 A
                                                    │
                                              执行工具 B
                                                    │
                                              [返回结果]
```

`StreamingToolExecutor` 的版本：

```
生产版本：
Model Stream: ──[text]──[tool_A_start]──[tool_A_args]──[tool_B_start]──[tool_B_args]──►
                              │                               │
                        立即开始执行                     立即开始执行
                        Tool A (并行)                   Tool B (并行)
                              │                               │
                              └──────────────┬────────────────┘
                                         结果就绪
```

```typescript
// StreamingToolExecutor 核心逻辑（概念性重现）
class StreamingToolExecutor {
  private pendingToolCalls: Map<string, Promise<ToolResult>> = new Map();

  async *process(event: StreamEvent): AsyncGenerator<AssistantEvent> {
    if (event.type === 'content_block_start' &&
        event.content_block.type === 'tool_use') {
      // 流开始包含工具调用，立即准备执行环境
      this.currentToolCall = {
        id: event.content_block.id,
        name: event.content_block.name,
        argsBuffer: '',
      };
    }

    if (event.type === 'content_block_delta' &&
        event.delta.type === 'input_json_delta') {
      // 逐步接收工具参数
      this.currentToolCall.argsBuffer += event.delta.partial_json;
    }

    if (event.type === 'content_block_stop' && this.currentToolCall) {
      // 参数接收完毕，立即触发工具执行（不等待其他内容）
      const tool = this.findTool(this.currentToolCall.name);
      const args = JSON.parse(this.currentToolCall.argsBuffer);

      // 并发安全的工具立即开始执行
      if (tool.isConcurrencySafe()) {
        this.pendingToolCalls.set(
          this.currentToolCall.id,
          tool.invoke(args)  // 非阻塞，立即返回 Promise
        );
      }
    }
  }
}
```

分隔符是 `content_block_stop` 事件——这是 Anthropic streaming API 中标识一个内容块结束的信号。`StreamingToolExecutor` 不需要整个响应的 `message_stop` 事件，在每个工具调用块结束时就立即触发执行。

## Auto-compact：透明的 Context 续命

当对话很长，context 使用率接近限制时，继续对话会失败。Claude Code 在 `query.ts` 的循环中内置了自动压缩：

```typescript
// tokenBudget.ts 中的预算检查
function shouldAutoCompact(messages: Message[]): boolean {
  const usage = estimateTokenUsage(messages);
  const limit = getCurrentContextWindow();
  return usage / limit > 0.95; // 超过 95% 时触发
}
```

Auto-compact 对用户完全透明——对话在不中断的情况下继续。第六章会详细讲压缩策略；这里的关键点是：**压缩逻辑嵌入在 agent 循环内部**，而不是作为调用方的责任。

这是一个重要的架构原则：context 管理是基础设施问题，不应该渗漏到业务逻辑层。

## Token 预算管理

`tokenBudget.ts` 追踪每类内容消耗的 token：

```typescript
interface TokenBudget {
  systemPrompt: number;    // 通常被缓存，不计入实时消耗
  conversationHistory: number;
  toolResults: number;     // 往往是最大的消耗来源
  currentResponse: number;
  total: number;
  limit: number;
}
```

工具结果通常是 context 爆炸的主要原因——读取一个大文件的结果可能有几千个 token。这就是为什么工具实现中有结果截断逻辑（第三章会讲到），以及为什么压缩策略优先处理旧的工具结果。

## 关键收获

- `query.ts` 实现真正的 `while(true)` 循环，只在模型不再发起工具调用时才退出，每次迭代可触发多个工具并将结果反馈为下轮输入。
- `StreamingToolExecutor` 在收到 `content_block_stop` 信号时就立即触发工具执行，无需等待整个响应的 `message_stop`，从而将工具延迟与模型生成并行化。
- 并发安全的工具（通过 `isConcurrencySafe()` 声明）可以在同一个流中并行执行，互不阻塞，显著缩短多工具调用的总耗时。
- Auto-compact 在 context 使用率超过 95% 时自动触发压缩，嵌入在 agent 循环内部，对上层调用方完全透明。
- `tokenBudget.ts` 按消息类型（系统提示、对话历史、工具结果、当前响应）分类追踪 token 消耗，其中工具结果是最主要的 context 膨胀来源。
