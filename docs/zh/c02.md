# 第二章 -- Agent Loop：1,729 行 query.ts 的流式工具执行引擎

> StreamingToolExecutor 在模型流式响应的中途就启动工具执行，530 行自研调度器用并发安全判定、错误级联和 Promise.race 实现了零依赖的流水线并行。

---

## 概述

大多数 agent 教程的循环是：发送消息 -> 等完整响应 -> 解析工具调用 -> 逐个执行 -> 追加结果 -> 重复。每个阶段都在等上一个阶段结束，延迟逐步累加。

Claude Code 的 `query.ts`（1,729 行）和 `StreamingToolExecutor.ts`（530 行）构成了截然不同的引擎。流式响应字节到达时，解析器就在寻找 `tool_use` 块；一旦参数流完毕，立即判定并发安全性并启动执行——此时模型可能还在生成下一个工具调用。

本章所有代码块均来自真实源码，标注文件名和行号。

---

## 1. while(true)：永不退出的主循环

`queryLoop` 的核心是一个状态机式的 `while (true)` 循环，只有在模型不再请求工具时才退出：

```typescript
// query.ts:306-311
while (true) {
  let { toolUseContext } = state
  const {
    messages, autoCompactTracking, maxOutputTokensRecoveryCount,
    hasAttemptedReactiveCompact, maxOutputTokensOverride,
    pendingToolUseSummary, stopHookActive, turnCount,
  } = state
```

`toolUseContext` 用 `let`（迭代内可重新赋值），其余字段用 `const`（迭代内只读）。循环内有 7 个 `continue` 站点，每个都用 `state = { ... }` 一次性写入完整状态，避免了 9 个独立变量赋值的遗漏风险。

## 2. StreamingToolExecutor 初始化：Feature Gate 控制

每轮迭代开始时根据 feature gate 决定是否启用流式工具执行：

```typescript
// query.ts:561-568
const useStreamingToolExecution = config.gates.streamingToolExecution
let streamingToolExecutor = useStreamingToolExecution
  ? new StreamingToolExecutor(
      toolUseContext.options.tools, canUseTool, toolUseContext,
    )
  : null
```

这不是装饰性的 flag。关闭时整条流式执行路径被跳过，退回顺序执行。Anthropic 可以在出现问题时一键回退，不需要发新版本。

## 3. API 流式调用与中途注入

循环核心是一个 `for await` 遍历模型的流式响应：

```typescript
// query.ts:659-664
for await (const message of deps.callModel({
  messages: prependUserContext(messagesForQuery, userContext),
  systemPrompt: fullSystemPrompt,
  tools: toolUseContext.options.tools,
  signal: toolUseContext.abortController.signal,
  // ...
```

每当 assistant message 到达且包含 `tool_use` 块时，立即注入到执行器：

```typescript
// query.ts:841-843
for (const toolBlock of msgToolUseBlocks) {
  streamingToolExecutor.addTool(toolBlock, message)
}
```

紧接着，在同一个流式循环内，非阻塞地收割已完成的结果：

```typescript
// query.ts:851-856
for (const result of streamingToolExecutor.getCompletedResults()) {
  if (result.message) { yield result.message }
}
```

时序含义：**模型还在生成第三个工具调用的参数时，第一个工具可能已经执行完毕、结果已经 yield 给 UI 了。**

```
模型流:  ──[text]──[tool_A 完毕]──[tool_B 完毕]──[tool_C 完毕]──[结束]──►
                        │                │               │
                   addTool(A)       addTool(B)      addTool(C)
                   立即执行           判定并发安全      排队等待
                        │            立即并行            │
                   getCompleted()  getCompleted()  getRemainingResults()
                   yield A 结果     yield B 结果     yield C 结果
```

### 为什么不等完整响应再执行工具？

等待 `message_stop` 意味着所有工具延迟堆叠在流结束之后。假设模型生成三个工具调用耗时 2s，每个工具执行 500ms：

- **等完整响应**：2s (流) + 1.5s (顺序执行) = 3.5s
- **流式中途执行**：2s (流) + ~0s (执行与流并行) = ~2s

工具执行与流生成重叠，用户感知延迟砍掉 40-60%。

## 4. StreamingToolExecutor：530 行并发调度器

### 4.1 TrackedTool 状态机

每个工具经历四个状态：`queued` -> `executing` -> `completed` -> `yielded`。

```typescript
// StreamingToolExecutor.ts:20-31
type TrackedTool = {
  id: string
  block: ToolUseBlock
  assistantMessage: AssistantMessage
  status: ToolStatus
  isConcurrencySafe: boolean
  promise?: Promise<void>
  results?: Message[]
  pendingProgress: Message[]        // 进度消息单独存储，立即 yield
  contextModifiers?: Array<(context: ToolUseContext) => ToolUseContext>
}
```

### 4.2 并发安全判定

`addTool` 解析输入参数后调用工具自身的 `isConcurrencySafe` 方法：

```typescript
// StreamingToolExecutor.ts:104-113
const parsedInput = toolDefinition.inputSchema.safeParse(block.input)
const isConcurrencySafe = parsedInput?.success
  ? (() => {
      try { return Boolean(toolDefinition.isConcurrencySafe(parsedInput.data)) }
      catch { return false }
    })()
  : false
```

双重防御：`safeParse` 失败直接标记不安全，`isConcurrencySafe` 抛异常也回退到 `false`。安全性由工具自己声明而非调度器猜测——Read 读文件安全，Bash 写命令不安全，这些语义只有工具自身知道。

### 4.3 调度逻辑：三条规则

```typescript
// StreamingToolExecutor.ts:129-135
private canExecuteTool(isConcurrencySafe: boolean): boolean {
  const executingTools = this.tools.filter(t => t.status === 'executing')
  return (
    executingTools.length === 0 ||
    (isConcurrencySafe && executingTools.every(t => t.isConcurrencySafe))
  )
}
```

1. **无工具执行中** -> 直接启动。
2. **全是并发安全的** -> 并行执行。
3. **否则** -> 阻塞等待。

队列处理进一步保证非并发工具的顺序性：

```typescript
// StreamingToolExecutor.ts:140-151
private async processQueue(): Promise<void> {
  for (const tool of this.tools) {
    if (tool.status !== 'queued') continue
    if (this.canExecuteTool(tool.isConcurrencySafe)) {
      await this.executeTool(tool)
    } else {
      if (!tool.isConcurrencySafe) break  // 非并发工具：阻塞后续所有工具
    }
  }
}
```

关键是 `break`：遇到不能执行的非并发工具时直接退出，即使后面有可执行的并发安全工具也不执行，保证严格有序性。

### 为什么不把所有工具都并行执行？

Read 五个文件可以并行，但 `mkdir -p /tmp/foo` 然后 `echo hello > /tmp/foo/bar.txt` 有隐式依赖。Bash 工具执行写操作时可能存在依赖链，盲目并行导致不确定结果。`isConcurrencySafe` 让工具声明并发语义，调度器只做执行决策不做语义猜测。

## 5. 错误级联：仅 Bash 触发

不是所有工具错误都触发级联取消：

```typescript
// StreamingToolExecutor.ts:356-363
// Only Bash errors cancel siblings. Bash commands often have implicit
// dependency chains (e.g. mkdir fails -> subsequent commands pointless).
// Read/WebFetch/etc are independent -- one failure shouldn't nuke the rest.
if (tool.block.name === BASH_TOOL_NAME) {
  this.hasErrored = true
  this.erroredToolDescription = this.getToolDescription(tool)
  this.siblingAbortController.abort('sibling_error')  // 级联取消兄弟工具
}
```

`siblingAbortController` 是 `toolUseContext.abortController` 的子控制器（line 58-61）。取消兄弟工具不会取消父控制器——`query.ts` 的主循环不会因此结束当前 turn，只是收到错误结果并继续。

## 6. 两个结果收割器：同步与异步

### 6.1 getCompletedResults()：流式中途同步收割

```typescript
// StreamingToolExecutor.ts:412-440
*getCompletedResults(): Generator<MessageUpdate, void> {
  for (const tool of this.tools) {
    while (tool.pendingProgress.length > 0) {           // 进度消息立即 yield
      yield { message: tool.pendingProgress.shift()! }
    }
    if (tool.status === 'yielded') continue
    if (tool.status === 'completed' && tool.results) {
      tool.status = 'yielded'
      for (const message of tool.results) { yield { message } }
    } else if (tool.status === 'executing' && !tool.isConcurrencySafe) {
      break  // 非并发工具未完成：阻塞后续结果，保证因果顺序
    }
  }
}
```

同步 Generator（`*` 非 `async *`），完全非阻塞。关键是末尾的 `break`：如果模型先调用 `bash mkdir` 再调用 `read file.txt`，即使 read 先完成，也要等 mkdir 的结果先 yield。

### 6.2 getRemainingResults()：流结束后异步等待

```typescript
// StreamingToolExecutor.ts:453-490
async *getRemainingResults(): AsyncGenerator<MessageUpdate, void> {
  while (this.hasUnfinishedTools()) {
    await this.processQueue()
    for (const result of this.getCompletedResults()) { yield result }

    if (this.hasExecutingTools() && !this.hasCompletedResults()) {
      const executingPromises = this.tools
        .filter(t => t.status === 'executing' && t.promise).map(t => t.promise!)
      const progressPromise = new Promise<void>(resolve => {
        this.progressAvailableResolve = resolve
      })
      if (executingPromises.length > 0) {
        await Promise.race([...executingPromises, progressPromise])
      }
    }
  }
}
```

`Promise.race` 同时等待工具完成和进度更新。任一事件发生就继续收割，让 UI 能实时显示进度消息。

### 为什么不用现成的任务队列库？

p-queue、bull、workerpool 都是成熟库，但 StreamingToolExecutor 的语义太特殊：并发安全性由工具自身声明、Bash 错误级联而 Read 不级联、结果必须按因果顺序 yield、进度消息必须立即旁路。这些语义交织在一起，用通用库的 hook 机制拼凑反而更复杂。530 行自研代码，零外部依赖，完全可控。

## 7. 双路径分发与状态延续

流式响应结束后，一个三元表达式选择执行路径：

```typescript
// query.ts:1380-1382
const toolUpdates = streamingToolExecutor
  ? streamingToolExecutor.getRemainingResults()   // 异步生成器
  : runTools(toolUseBlocks, assistantMessages, canUseTool, toolUseContext)  // 顺序回退
```

两条路径产出相同的 `AsyncGenerator<MessageUpdate>`，下游代码无需感知执行模式。

工具执行完毕后，将结果追加到消息历史，构建下一轮状态：

```typescript
// query.ts:1715-1728
state = {
  messages: [...messagesForQuery, ...assistantMessages, ...toolResults],
  toolUseContext: toolUseContextWithQueryTracking,
  autoCompactTracking: tracking,
  turnCount: nextTurnCount,
  // ...
  transition: { reason: 'next_turn' },
}
} // while (true)
```

控制流回到循环顶部，解构新状态，发起新的 API 请求。消息历史不断增长，直到模型不再发起工具调用或 token 预算耗尽。

## 8. Token 预算：循环内的续命机制

`TOKEN_BUDGET` feature 在每轮循环结束前检查 token 使用量：

```typescript
// query.ts:1308-1340
if (feature('TOKEN_BUDGET')) {
  const decision = checkTokenBudget(budgetTracker!, ...)
  if (decision.action === 'continue') {
    state = {
      messages: [...messagesForQuery, ...assistantMessages,
        createUserMessage({ content: decision.nudgeMessage, isMeta: true }),
      ],
      // ...
      transition: { reason: 'token_budget_continuation' },
    }
    continue  // <-- 跳回 line 307 的 while(true)
  }
}
```

注入一条 `isMeta: true` 的用户消息，然后 `continue` 跳回循环顶部。模型看到这条消息后继续工作，但不知道它是系统自动注入的——对模型透明的"续命"机制。

## 验证

**流式中途执行**：打开 Claude Code，请求"读取这五个文件然后告诉我共同点"。观察终端——文件读取结果在响应还在流式输出时就已经出现了，这就是 `getCompletedResults()` 在流式循环中被调用的效果。

**并行执行**：让 Claude Code 同时读取 5 个不同文件。所有文件几乎同时开始读取、同时返回——Read 工具声明了 `isConcurrencySafe = true`，调度器允许并行。

**顺序调度**：让 Claude Code 执行一个 bash 命令和一个文件读取。Bash 不是并发安全的，bash 先执行完毕后文件读取才开始——调度器遇到非并发工具时 `break` 了队列。

**错误级联**：让 Claude Code 并行执行两个 bash 命令，其中一个故意失败。失败命令触发 `siblingAbortController.abort('sibling_error')`，另一个收到 synthetic error 而不是继续执行。

## 关键收获

- `query.ts` 的 `while (true)` 循环（line 307）用单个 `State` 对象管理跨迭代状态，7 个 `continue` 站点各自写入完整状态快照，消除多变量赋值的遗漏风险。
- `StreamingToolExecutor` 分两阶段收割工具结果：流式中途用同步非阻塞的 `getCompletedResults()`，流结束后用异步 `Promise.race` 的 `getRemainingResults()`，让工具执行与模型生成最大程度并行。
- 并发安全性由工具自身通过 `isConcurrencySafe()` 声明，调度器执行三条规则：无执行中直接启动、全并发安全则并行、否则阻塞。遇到不可执行的非并发工具时 `break` 队列，保证因果顺序。
- 错误级联仅限 Bash 工具：Bash 命令有隐式依赖链，通过子 `AbortController` 级联取消兄弟工具；Read/WebFetch 等独立工具的失败不影响其他工具。
- Token 预算检查嵌入循环末尾，注入 `isMeta: true` 的 nudge message 让模型透明地继续工作，`continue` 跳回循环顶部实现"续命"。
- 双路径分发（line 1380-1382）让流式执行和顺序回退共用同一个 `AsyncGenerator<MessageUpdate>` 接口，下游代码无需感知执行模式差异。
- 530 行自研调度器零外部依赖，因为调度语义（并发安全声明、Bash 专属级联、因果顺序 yield、进度消息旁路）过于特殊，通用任务队列库无法直接适配。
