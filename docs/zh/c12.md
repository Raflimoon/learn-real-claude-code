# 在终端里跑 React

> 三个对象池（字符/样式/超链接）将 O(n) 字符串比较转为 O(1) 整数检查。

## 概述

Claude Code 的终端界面不是用 ncurses 或手写的 ANSI 转义序列构建的——它是一个完整的 React 应用。Ink 库让这成为可能：同样的 JSX 语法，同样的组件模型，同样的 hooks，只不过渲染目标不是浏览器 DOM，而是终端里的字符格子。

这不只是一个有趣的技术选择。React 的声明式模型和虚拟 DOM diff 机制，在终端这个高频刷新的环境里提供了一套天然的性能优化框架——只要你解决了终端渲染的特殊挑战。Claude Code 用三个对象池解决了这个问题。

理解这套系统，能让你从全新角度理解"渲染器"这个概念的本质。

## Ink：React 的自定义 Reconciler

React 的设计从一开始就允许多个渲染目标。React DOM 负责浏览器，React Native 负责移动端，而 Ink 实现了一个自定义 Reconciler，将 React 的虚拟 DOM 映射到终端字符网格。

```tsx
// 这是真实的 Ink 代码，在终端里运行
import { Box, Text } from 'ink';

const App = () => (
  <Box flexDirection="column">
    <Text color="green">✓ 任务完成</Text>
    <Box borderStyle="round" padding={1}>
      <Text>Claude 正在思考...</Text>
    </Box>
  </Box>
);
```

Reconciler 的核心职责：将 React 的 `createInstance`、`appendChild`、`commitUpdate` 等操作翻译为对字符缓冲区的写入。每个"DOM 节点"在这里对应终端网格中的一个矩形区域。

## Yoga 布局引擎：终端的 Flexbox

布局计算交给了 Yoga——Facebook 为 React Native 开发的跨平台 CSS Flexbox 引擎（现在也用于 React Native 和 Litho）。

```
┌─────────────────────────────────────┐
│  flexDirection: "row"               │
│  ┌──────────┐  ┌───────────────┐    │
│  │  左侧面板 │  │    主内容区    │    │
│  │  width:20│  │  flex: 1      │    │
│  └──────────┘  └───────────────┘    │
└─────────────────────────────────────┘
```

Yoga 以像素为单位计算布局，Ink 将像素 1:1 映射到终端字符位置（每个字符 = 一个"像素"）。这意味着你可以用标准的 CSS 思维来布局终端界面：`flexDirection`、`padding`、`margin`、`alignItems`……

## 双缓冲屏幕：只发送差异

终端不像浏览器——你不能精确地"更新某个元素"。终端的原始操作是：移动光标、写字符、设置颜色。全量重绘每次刷新会导致闪烁和带宽浪费。

Ink 的解决方案是**双缓冲**：

```
前缓冲区（当前显示）:    后缓冲区（新渲染）:
  [A][B][C][D]             [A][X][C][D]
  [E][F][G][H]             [E][F][Y][H]

diff → 只发送变化的格子：
  位置(0,1): B → X
  位置(1,2): G → Y
```

只有真正发生变化的字符位置才会触发终端写入操作。在典型的 Claude Code 会话中，每帧可能只有 5-10% 的字符格子发生变化，这意味着 90% 的终端 I/O 被省略了。

## 三个对象池：O(n) → O(1)

这是性能优化的核心，也是最精妙的部分。

双缓冲的 diff 算法需要逐格比较"当前字符是否发生了变化"。如果直接比较字符串和样式对象：

```typescript
// 朴素比较：O(n) 字符串操作
if (prevCell.char !== nextCell.char ||
    prevCell.style.color !== nextCell.style.color ||
    prevCell.style.bold !== nextCell.style.bold) {
  // 需要更新...
}
```

对于一个 200×50 的终端，每帧有 10,000 个格子需要比较。在 60fps 的刷新率下，这是每秒 600,000 次字符串比较。

对象池的解决方案：

```typescript
// 字符池、样式池、超链接池
const charPool   = new Map<string, number>();  // "A" → 42
const stylePool  = new Map<string, number>();  // "color:green;bold" → 17
const linkPool   = new Map<string, number>();  // "https://..." → 3

// 比较变为整数相等性检查
if (prevCell.charId !== nextCell.charId ||
    prevCell.styleId !== nextCell.styleId) {
  // 需要更新...O(1) 整数比较！
}
```

每个唯一字符、唯一样式组合、唯一超链接都被分配一个整数 ID。比较从字符串比较（O(n) where n = string length）变为整数比较（O(1)）。对于 60fps 的终端刷新，这个差异是性能与卡顿之间的分界线。

## 渲染管道全貌

```
JSX 组件树
    ↓ React reconciler
虚拟 DOM diff
    ↓ Yoga 布局计算
字符网格（带样式/位置）
    ↓ 对象池 ID 化
后缓冲区
    ↓ 与前缓冲区 diff
变化的格子列表
    ↓ ANSI 转义序列生成
终端输出
```

## 关键收获

- Ink 使用 React 自定义 Reconciler 将 JSX 渲染到终端字符网格，而非浏览器 DOM，让 Claude Code 用标准 React 组件模型构建终端 UI。
- Yoga 布局引擎提供完整的 Flexbox 支持，将每个终端字符位置视为一个"像素"，与 React Native 共享同一套布局计算引擎。
- 双缓冲机制通过前后缓冲区 diff 确保每帧只写入真正变化的字符格子，消除全量重绘导致的闪烁和冗余终端 I/O。
- 三个对象池（字符池、样式池、超链接池）将逐格比较的复杂度从 O(n) 字符串比较降至 O(1) 整数比较，是支撑 60fps 终端渲染的性能关键。
- 这套架构的深层启示：**渲染器是可替换的**——React 的声明式模型和虚拟 DOM diff 机制可以驱动任何基于字符网格的输出系统。
