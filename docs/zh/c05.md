# 110+ 字符串、两个缓存区、一条系统提示词

> `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 是一个字面量字符串 `'__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'`，将系统提示词切成全局缓存的前缀和每会话变动的后缀。用 `DANGEROUS_uncachedSystemPromptSection` 注册的 section 会在每次值变动时打断缓存 -- `DANGEROUS_` 前缀的存在，就是为了让开发者在付出代价之前先感受到代价。

## 概述

Claude Code 发送给模型的系统提示词不是一份静态文档。它是在运行时从超过 110 条独立字符串中组装出来的 -- 由大约 20 个 section 函数产出，按特定顺序合并，被一个缓存边界标记一分为二，再过滤掉所有 null。最终结果是一个字符串数组 -- 不是一整段拼接好的文本 -- API 以独立的 system message block 形式接收它们。

精妙之处在两个大多数 agent 开发者从不实现的机制上：

1. **双区缓存架构**：让每次请求的前 ~40,000 个 token 几乎免费（命中 Anthropic 的 prompt cache，享受 90% 折扣），同时允许剩余的 ~5,000-15,000 个 token 在每个会话中自由变动。

2. **一个 section 注册系统**：提供两个函数 -- 一个安全，一个故意起得吓人 -- 强制开发者声明自己的 section 能否被缓存，还是必须在每次变动时打断缓存。

本章解剖 `systemPromptSections.ts`（注册与缓存层）和 `prompts.ts`（组装管线），展示 110+ 条字符串是如何变成模型遵循的指令文档的。

## 两个注册函数

系统提示词的每一个 section，都必须通过两个函数中的一个来注册。这是整个系统最核心的设计决策：

```typescript
// systemPromptSections.ts

// 安全：在 /clear 或 /compact 之前跨 turn 缓存
function systemPromptSection(
  name: string,
  compute: () => Promise<string | null> | string | null,
): SystemPromptSection {
  return { name, compute, cacheBreak: false }
}

// 易变：每次值变动时打断 prompt cache
function DANGEROUS_uncachedSystemPromptSection(
  name: string,
  compute: () => Promise<string | null> | string | null,
  reason: string,
): SystemPromptSection {
  return { name, compute, cacheBreak: true }
}
```

`DANGEROUS_` 前缀不是玩笑。它是一个 forcing function -- 一个刻意的 API 设计选择，逼迫开发者停下来，为自己的 section 无法被缓存给出理由。`reason` 参数是强制的：你必须用一个字符串字面量解释为什么这个 section 需要打断缓存。代码库中的真实例子：

```typescript
DANGEROUS_uncachedSystemPromptSection(
  'mcp_instructions',
  () => isMcpInstructionsDeltaEnabled() ? null : getMcpInstructionsSection(),
  'MCP servers connect/disconnect between turns',
)
```

`reason` 字符串不会被日志记录或显示 -- 它纯粹作为调用点的文档存在。但要求它的存在改变了开发者的行为。没有它，每一个新 section 都会默认选择 uncached，因为"它可能会变"。有了它，开发者必须说清楚*为什么*它会变 -- 而这个过程往往揭示出，它其实不会变。

## Section 解析与缓存

组装提示词时，每个 section 经过一个解析步骤，检查一个按名称索引的缓存：

```typescript
// systemPromptSections.ts (lines 42-58)
export async function resolveSystemPromptSections(
  sections: SystemPromptSection[],
): Promise<(string | null)[]> {
  const cache = getSystemPromptSectionCache()
  return Promise.all(
    sections.map(async (s) => {
      if (!s.cacheBreak && cache.has(s.name)) {
        return cache.get(s.name) ?? null
      }
      const value = await s.compute()
      setSystemPromptSectionCacheEntry(s.name, value)
      return value
    }),
  )
}
```

三个要点：

1. **Section 并行解析**，通过 `Promise.all`。各 compute 函数相互独立 -- 加载 memory 不会阻塞环境信息，环境信息不会阻塞 MCP 指令。

2. **缓存按名称查找，不按内容哈希**。一个名为 `'memory'` 的 section 会一直返回缓存值，直到缓存被显式清除。这意味着缓存是会话级的 memoization，不是内容寻址存储。

3. **`cacheBreak: true` 的 section 完全跳过缓存**。它们总是调用 `compute()`，总是存储结果（所以下一个碰巧同名的非 cacheBreak section 会拿到最新值 -- 不过实践中名称都是唯一的）。

## 缓存失效

缓存只在两个地方被清除：

```typescript
// 仅被 /clear 和 /compact 命令调用
export function clearSystemPromptSections(): void {
  systemPromptSectionCache.clear()
  // 同时重置 beta header latches
}
```

这意味着 section 缓存在整个对话期间都存活，除非用户显式运行 `/clear` 或 `/compact`。在对话过程中编辑了 CLAUDE.md 文件？不会被重新读取 -- 直到上述命令被触发。这是有意为之的：在每个 turn 都重新读取所有 CLAUDE.md 文件会给每次请求增加磁盘 I/O。

## 静态 Section（可缓存、稳定）

提示词的前半部分 -- 边界之前的一切 -- 由在会话内几乎不变的 section 组成。它们通过 `systemPromptSection`（没有 `DANGEROUS_` 前缀）注册：

```
Section 名称              来源函数                       内容
─────────────────────────────────────────────────────────────────────────
intro                     getSimpleIntroSection()        "You are Claude Code..."
                                                         核心身份段落

system                    getSimpleSystemSection()       工具执行规则、
                                                         输出格式、hooks 行为

doing_tasks               getSimpleDoingTasksSection()   代码风格指导、
                                                         任务完成模式

actions                   getActionsSection()            可逆性评估、
                                                         风险分级规则

using_tools               getUsingYourToolsSection()     何时使用哪个工具、
                                                         工具选择启发式规则

tone_and_style            getSimpleToneAndStyleSection()  不用 emoji、不说废话、
                                                         参考风格规范
```

这六个 section 构成了身份核心。它们定义了 *Claude Code 是什么*，以及*它如何行动*。因为它们编译进二进制文件且不依赖任何运行时状态，所以在每个会话、每个用户、每个项目中都完全相同。这正是它们完美适合 prompt caching 的原因 -- Anthropic 的缓存基于前缀匹配，而这些 section 构成一个稳定的前缀。

边界之上所有内容的缓存 scope 是 `'global'` -- 意味着它可以跨组织共享，不仅仅是同一用户的不同请求之间。一个静态内容的哈希值决定缓存命中。如果哪怕一个字符变了（比如 intro section 里修了个 typo），所有用户的缓存都会 miss，直到新版本预热完成。

## 动态边界

边界本身是一个常量字符串：

```typescript
const SYSTEM_PROMPT_DYNAMIC_BOUNDARY = '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'
```

它作为一个字面量元素出现在组装后的数组中。API 层（或 prompt cache 实现）用这个标记来知道可缓存前缀在哪里结束。它之前的一切可以匹配全局缓存的前缀。它之后的一切是每会话独有的，需要重新处理。

```
┌─────────────────────────────────────────────────────────┐
│  intro                    "You are Claude Code..."      │
│  system                   工具规则、输出格式              │
│  doing_tasks              代码风格、任务指导              │  scope: 'global'
│  actions                  风险评估规则                    │  (跨组织缓存)
│  using_tools              工具选择启发式规则              │
│  tone_and_style           风格规范                       │
│                                                         │
├── '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__' ─────────────────┤
│                                                         │
│  session_guidance         当前可用的工具 + 技能           │
│  memory                   ~/.claude/CLAUDE.md 内容       │
│  mcp_instructions  [!]    MCP 服务器指令                 │  scope: 每会话
│  claude_md                项目 CLAUDE.md 文件             │  (对 [!] section
│  environment              OS、shell、cwd、git 状态       │   每 turn 重算，
│  active_context           当前打开的文件                  │   其余缓存)
│  deferred_tools           仅有名称的工具列表              │
│  ... (12-15 个 section)                                  │
└─────────────────────────────────────────────────────────┘
```

## 动态 Section（每会话变动）

边界之下，section 因项目、因会话、有时因 turn 而异。在这里 `DANGEROUS_` 函数才真正名副其实：

```typescript
// 每会话缓存（安全 -- 在一次对话中很少变化）
systemPromptSection('session_guidance', () =>
  getSessionSpecificGuidanceSection(enabledTools, skillToolCommands),
)

systemPromptSection('memory', () => loadMemoryPrompt())

// DANGEROUS -- 每 turn 重新计算，因为 MCP 服务器
// 可以在任意两条消息之间连接或断开
DANGEROUS_uncachedSystemPromptSection(
  'mcp_instructions',
  () => isMcpInstructionsDeltaEnabled() ? null : getMcpInstructionsSection(),
  'MCP servers connect/disconnect between turns',
)
```

`session_guidance` 是缓存权衡的好案例。它取决于哪些工具被启用、哪些技能可用 -- 两者都在会话启动时确定并在此后保持稳定。所以它使用安全的注册函数：计算一次，缓存整个会话。

`mcp_instructions` 则不同。MCP 服务器随时可能连接或断开（用户可能在另一个终端启动了新服务器）。如果这个 section 被缓存，模型就不会知道新可用的服务器，直到运行 `/clear`。所以它在每个 turn 都付出打断缓存的代价。

## 延迟工具 -- 控制在 200K token 以内

系统提示词有硬性的 token 预算。40+ 个工具，每个都携带完整的 JSON schema，光工具描述就能消耗 50,000+ token。Claude Code 用工具延迟加载解决这个问题：

```
完整注册（在系统提示词中）：
  Read, Write, Edit, Bash, Glob, Grep, Skill  -- 7 个核心工具
  每个包含：名称、描述、完整参数 schema

延迟注册（在系统提示词中）：
  NotebookEdit, WebFetch, WebSearch, CronCreate, ...  -- 18+ 个工具
  每个仅包含：名称（没有 schema，没有描述）
```

`ToolSearchTool`（本身是一个完整注册的工具）充当 schema 加载器。当模型需要一个延迟工具时，它先调用 ToolSearch：

```
模型想使用 WebSearch：
  1. 在延迟工具列表中看到 "WebSearch"（仅名称）
  2. 调用 ToolSearchTool，query "select:WebSearch"
  3. 收到完整的 schema 定义
  4. 现在可以用正确的参数调用 WebSearch
```

这是一个两步调用模式，用一次额外的 API 往返换取每条系统提示词少 ~30,000 个 token。在规模化运行中，这比它的开销省得多得多。

## 组装管线

`prompts.ts` 中的最终组装遵循这个结构：

```typescript
// prompts.ts (lines 444-577, 简化版)
async function assembleSystemPrompt(session: Session): Promise<(string | null)[]> {
  const staticSections = [
    systemPromptSection('intro', getSimpleIntroSection),
    systemPromptSection('system', getSimpleSystemSection),
    systemPromptSection('doing_tasks', getSimpleDoingTasksSection),
    systemPromptSection('actions', getActionsSection),
    systemPromptSection('using_tools', getUsingYourToolsSection),
    systemPromptSection('tone_and_style', getSimpleToneAndStyleSection),
    // ... 可能还有更多
  ]

  const dynamicSections = [
    systemPromptSection('session_guidance', ...),
    systemPromptSection('memory', ...),
    DANGEROUS_uncachedSystemPromptSection('mcp_instructions', ..., '...'),
    // ... 12-15 个更多 section
  ]

  const resolvedStatic = await resolveSystemPromptSections(staticSections)
  const resolvedDynamic = await resolveSystemPromptSections(dynamicSections)

  return [
    ...resolvedStatic,
    SYSTEM_PROMPT_DYNAMIC_BOUNDARY,
    ...resolvedDynamic,
  ].filter(s => s !== null)
}
```

返回类型是字符串数组，不是拼接好的单一字符串。API 层负责 join。这很重要，因为 Anthropic 的 API 接受 system prompt 作为 content block 的数组，保持它们分离才能保留缓存边界的语义。

末尾的 `filter(s => s !== null)` 很关键：任何 `compute()` 返回 `null` 的 section 会被静默省略。条件性 section 就是这样工作的 -- 如果没有 MCP 服务器连接，`getMcpInstructionsSection()` 返回 `null`，MCP 区块就不会出现在提示词中。

## Proactive/KAIROS 模式 -- 一套完全不同的提示词

当 Claude Code 在 proactive 模式（也称 KAIROS）下运行时，整个组装管线被替换：

```
普通模式：
  20 个 section 函数 -> 110+ 条字符串 -> 完整的助手身份

Proactive/KAIROS 模式：
  精简的自主 agent 提示词 -> 最小化工具集 -> 目标导向的身份
```

这不是一个微调几个 section 的标志位。它是一条完全独立的提示词组装路径，产出的是根本不同的 agent 身份。普通提示词说"你是一个帮助开发者的助手"。KAIROS 提示词说的更接近"你是一个追求特定目标的自主 agent"。不同的工具被启用，不同的行为规则生效，缓存边界的位置也可能不同。

## 成本计算

要理解这个架构为什么重要，看看 token 经济学：

```
Prompt cache 定价（Anthropic，近似值）：
  未缓存输入：    $15.00 / 1M tokens  (Opus)
  缓存输入：       $1.50 / 1M tokens  (90% 折扣)

典型系统提示词：
  静态 section：  ~40,000 tokens
  动态 section：  ~10,000 tokens
  总计：           ~50,000 tokens

每请求成本（冷启动，无缓存）：
  50,000 tokens x $15.00/1M = $0.75

每请求成本（热缓存，静态部分命中）：
  40,000 tokens x $1.50/1M  = $0.06  (缓存命中)
  10,000 tokens x $15.00/1M = $0.15  (重新处理)
  总计：                       $0.21

每请求节省：$0.54
每天 100 次请求：节省 $54/天
```

那个边界标记不是一个装饰性分隔符。它是一个每天价值 $54 的架构决策。

## Section 生命周期

```
会话启动
    |
    +-- 静态 section 计算完成，全局缓存
    +-- 动态 section 计算完成，每会话缓存
    |
    v
Turn 1: 所有 section 解析（动态 section 缓存冷启动）
    |
    v
Turn 2: 缓存的 section 立即返回
         DANGEROUS_ section 重新计算
    |
    v
Turn N: 同样的模式继续
    |
    v
用户运行 /clear 或 /compact
    |
    +-- clearSystemPromptSections() 被调用
    +-- 所有缓存清空
    +-- Beta header latches 重置
    |
    v
下一个 turn: 一切重新计算
```

## 核心要点

- 系统提示词是一个由 ~20 个 section 函数组装出的 110+ 条字符串的数组，被 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 切成全局缓存的前缀和每会话变动的后缀。边界是一个字面量字符串：`'__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'`。

- 两个注册函数执行缓存纪律：`systemPromptSection()`（安全、缓存）和 `DANGEROUS_uncachedSystemPromptSection()`（易变、每 turn 重算）。`DANGEROUS_` 前缀和强制的 `reason` 参数是故意的 API 摩擦，防止开发者随意打断缓存。

- Section 解析通过 `Promise.all` 并行执行，配合一个按名称索引的缓存，该缓存持续到 `/clear` 或 `/compact`。非 cacheBreak 的 section 只计算一次，在对话剩余时间内返回缓存值。

- 六个静态 section（intro、system、doing_tasks、actions、using_tools、tone_and_style）构成稳定的身份核心。它们使用 `scope: 'global'` 缓存 -- 跨组织共享，仅在源代码变动时失效。

- 动态 section（session_guidance、memory、mcp_instructions、claude_md、environment、deferred_tools 等）位于边界之下。大多数按会话缓存；只有 `DANGEROUS_` section 每 turn 重算。

- 工具延迟加载控制提示词在预算之内：7 个核心工具包含完整 schema，18+ 个工具仅包含名称。`ToolSearchTool` 按需加载完整 schema，用一次额外的往返换取每次请求少 ~30,000 个 token。

- Proactive/KAIROS 模式完全绕过正常组装，使用精简的自主 agent 提示词，配合不同的行为规则和不同的工具集。

- 缓存边界不是一个抽象概念 -- 它是一个直接的成本优化。40,000 个 token 以 90% 折扣缓存，每次请求节省约 $0.54。
