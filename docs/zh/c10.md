# 一个 Claude 指挥一群 Claude

> Fork 的子 agent 共享父节点的 prompt cache，实现近乎零成本的上下文继承。

## 概述

当一个 Claude 实例不够用时，怎么办？Claude Code 的答案是：再生一个。Coordinator Mode 是 Claude Code 源码中最令人着迷的架构之一——一个父级 Claude 进程扮演"指挥官"角色，负责分解任务、分发工作、汇总结果，而多个 Worker Claude 进程则并行执行具体操作。

这套系统没有依赖任何 LangGraph、CrewAI 或其他编排框架。它的底层机制朴素得令人惊讶：进程生成（process spawning）加上文件 I/O。这不是偷懒，而是一种深思熟虑的工程选择。

理解 Coordinator Mode 的关键在于 prompt cache。当子进程从父进程 fork 出来时，它天然继承了父节点已经缓存的上下文——这意味着数万 token 的"理解成本"几乎为零。这是整个多 agent 架构中最优雅的一笔。

## coordinatorMode.ts：编排的核心

Coordinator 的逻辑集中在 `coordinatorMode.ts` 中。它的工作流程大致如下：

```typescript
// 伪代码：Coordinator 的任务分发逻辑
async function runCoordinator(task: Task, workerCount: number) {
  const subtasks = await decompose(task);        // 分解任务
  const workerProcesses = subtasks.map(subtask =>
    spawnWorker(subtask, getTempFilePath())       // 生成子进程，指定结果文件
  );

  await Promise.all(workerProcesses);            // 并行等待所有 Worker

  const results = subtasks.map(s =>
    fs.readFileSync(s.resultPath, 'utf-8')       // 读取各 Worker 的输出
  );

  return synthesize(results);                    // 合并结果
}
```

关键细节：每个 Worker 进程有一个专属的临时文件路径。Worker 完成工作后，将结果序列化写入该文件；Coordinator 轮询或等待后读取。这就是所谓的**基于文件的 IPC**（Inter-Process Communication）。

## 基于文件的 IPC：为什么不用管道或消息队列？

你可能会问：为什么用文件，而不是 Unix 管道、WebSocket、或者 Redis 消息队列？

几个原因：

1. **持久性**：文件可以在进程崩溃后保留，便于调试和恢复
2. **调试友好**：你可以直接打开 `/tmp/` 目录查看每个 Worker 写了什么
3. **进程无关**：Worker 完成后就退出，不需要保持连接
4. **天然并发**：每个 Worker 写自己的文件，没有竞争条件

```
Coordinator Process
    │
    ├── spawn Worker-1 → writes /tmp/worker-1-result.json
    ├── spawn Worker-2 → writes /tmp/worker-2-result.json
    └── spawn Worker-3 → writes /tmp/worker-3-result.json
         │
         └── await all → read files → synthesize
```

这个模式在大规模分布式系统里会显得"土"，但在单机多进程场景下，它的简洁性是一种优势。

## Prompt Cache 共享：零成本的上下文继承

这是整个 Coordinator Mode 中最值得深入理解的机制。

当 Coordinator 生成一个 Worker 子进程时，子进程携带的初始 context 已经被缓存。Anthropic 的 prompt caching 机制会对相同的前缀内容计算缓存命中——如果 Worker 的系统提示、当前对话历史与 Coordinator 高度重叠（事实上它们几乎完全相同），那么这部分 token 的处理成本接近零。

```
父进程 context（已缓存）：
  [系统提示 8K] [对话历史 20K] [当前代码库快照 30K]
         ↓ fork
子进程：
  ✓ 系统提示    → cache HIT（$0.00）
  ✓ 对话历史    → cache HIT（$0.00）
  ✓ 代码库快照  → cache HIT（$0.00）
  + 子任务指令  → 新增 token（计费）
```

这意味着你可以同时运行 10 个 Worker，而上下文传递的边际成本几乎可以忽略不计。

## 与 LangGraph/CrewAI 的对比

| 维度 | LangGraph/CrewAI | Claude Code Coordinator |
|------|-----------------|------------------------|
| 编排方式 | 框架 DAG / 角色定义 | 原生进程生成 |
| 通信机制 | 框架内部状态 | 文件 I/O |
| 上下文传递 | 显式序列化 | Prompt cache 继承 |
| 调试难度 | 框架抽象层 | 直接查看临时文件 |
| 并行上限 | 框架限制 | 操作系统进程限制 |

框架的优势是结构化和可视化；Claude Code 的优势是透明、低依赖、以及 prompt cache 带来的经济性。

## 关键收获

- Coordinator Mode 的本质是进程生成加文件 I/O，没有依赖 LangGraph、CrewAI 或任何编排框架。
- Fork 的子 Worker 继承父节点的 prompt cache context，数万 token 的上下文传递成本接近零，使并行化在经济上可行。
- 基于文件的 IPC 虽然看似"朴素"，但提供了崩溃后持久性、可直接查看 `/tmp/` 目录的调试友好性，以及每个 Worker 写独立文件的天然并发安全。
- Coordinator 可以同时管理 10 个以上的并行 Worker，瓶颈在于操作系统的进程限制而非框架约束。
- 这套架构的核心洞察：**廉价的上下文等于廉价的并行化**——当 prompt cache 解决了上下文共享的成本问题，多 agent 协作的边际成本几乎可以忽略。
