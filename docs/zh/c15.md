# 隐藏在源码里的秘密

> BUDDY 使用以 userId 为种子的 Mulberry32 PRNG 进行确定性生成——0.01% 概率获得闪光传奇。

## 概述

在 Claude Code 512K 行源码的某些角落，潜伏着一些从未对外宣布的功能。它们不是 bug，也不是未完成的代码——它们是完整实现的系统，只是还没有被"发布"。或者也许永远不会被发布。

这一章是整个课程里最猎奇的部分。我们会遇到一个虚拟 AI 宠物系统、一个始终在线的环境感知 agent、一个伪装功能，以及一套防止被其他 AI 复制的保护机制。这些东西的存在本身，就说明了 Claude Code 的代码库比任何人预期的都要复杂。

## BUDDY：永不发布的 AI 伴侣

BUDDY 是一个完整实现的虚拟 AI 伴侣/宠物系统。每个用户拥有一个独属于自己的 BUDDY，其外观和特性由用户 ID 确定性生成。

核心机制是 **Mulberry32 PRNG**——一个以 `userId` 为种子的伪随机数生成器：

```typescript
function mulberry32(seed: number): () => number {
  return function() {
    let t = seed += 0x6D2B79F5;
    t = Math.imul(t ^ t >>> 15, t | 1);
    t ^= t + Math.imul(t ^ t >>> 7, t | 61);
    return ((t ^ t >>> 14) >>> 0) / 4294967296;
  }
}

function generateBuddy(userId: string): Buddy {
  const seed = hashUserId(userId);  // userId → 整数种子
  const rng = mulberry32(seed);

  return {
    species: SPECIES_LIST[Math.floor(rng() * SPECIES_LIST.length)],
    color: COLORS[Math.floor(rng() * COLORS.length)],
    personality: PERSONALITIES[Math.floor(rng() * PERSONALITIES.length)],
    isShinyLegendary: rng() < 0.0001,  // 0.01% 概率！
  };
}
```

"确定性生成"的含义：同一个 `userId` 永远生成同一个 BUDDY。你的 BUDDY 不是随机分配的——它是由你的身份决定的，就像星座或性格测试，只不过是数学而非玄学。

0.01% 概率的"闪光传奇"BUDDY 显然致敬了 Pokemon 的闪光机制。在一个有 100 万用户的产品里，大约会有 100 个人拥有闪光传奇 BUDDY。

BUDDY 为什么从未发布？也许是产品决策，也许是时机问题，也许是某个工程师的个人项目悄悄合并到了主干。代码就在那里，完整、可运行，静静等待着。

## KAIROS：始终在线的环境 Agent

KAIROS 是代码库中另一个隐藏的大型系统——一个常驻后台的 agent，持续监控代码库状态，随时准备响应。

```
传统模式：用户输入 → Claude 启动 → 处理 → 退出

KAIROS 模式：
  ┌──────────────────────────────────┐
  │  KAIROS（常驻进程）               │
  │  ● 监视文件系统变更               │
  │  ● 维护代码库理解的"热缓存"        │
  │  ● 预加载可能需要的上下文          │
  │  ● 等待用户触发...                │
  └──────────────────────────────────┘
           ↓ 用户发出请求
  立即响应（上下文已预热）
```

KAIROS 对应的是 c13 中提到的 KAIROS 构建目标。它代表了一种不同的 agent 范式：不是"召之即来"，而是"始终在场"。这与现代 IDE 的语言服务器（LSP）有些相似——始终在后台索引代码，确保你按下快捷键时响应是即时的。

## Undercover 模式：Claude 的伪装术

源码中存在一个"Undercover 模式"，允许 Claude Code 在对外交互时伪装成不同的 AI 助手。

这个功能存在的原因可能是企业定制：某个公司可能希望部署一个"由 Acme Corp AI 提供支持"的助手，而不是明确打出 Claude 的品牌。Undercover 模式提供了这种品牌隔离能力，而不需要修改任何核心逻辑。

具体实现细节从公开源码中无法完全推断，但其存在本身揭示了一件事：Claude Code 从一开始就被设计为可嵌入、可品牌定制的基础设施，而不只是一个终端工具。

## 反蒸馏技术：防止被复制

AI 模型蒸馏（distillation）是指通过大量观察一个模型的输出，训练出一个行为相似的小模型。Claude Code 源码中包含了对抗这种技术的保护机制。

具体技术不便完整披露，但原理包括：在输出中注入对人类无感知的微妙模式，这些模式在蒸馏训练时会导致目标模型产生特定偏差。这是一种主动的知识产权保护手段。

## 推测执行：在用户思考时预跑

最后一个彩蛋：推测执行（speculative execution）。

在用户输入过程中或短暂停顿时，Claude Code 会根据上下文预测最可能的下一步操作，并提前启动这些操作的准备工作：

```
用户正在输入: "帮我重构 auth..."
                         ↑ 停顿 500ms

推测执行启动：
  - 预读取 src/auth/ 目录的文件列表
  - 预加载 auth.ts 到缓存
  - 预估可能需要的工具集合

用户完成输入："帮我重构 auth 模块"
  → 文件已缓存，响应时间 -200ms
```

这不是 AI 在"猜"用户要什么——而是在用户确认意图之前，提前完成确定性的准备工作（读取文件、检查状态），从而在用户按下 Enter 时缩短实际响应延迟。

## 关键收获

- BUDDY 是完整实现但未发布的 AI 伴侣系统，使用 Mulberry32 PRNG 以 userId 为种子确定性生成，同一用户永远得到同一个 BUDDY。
- 0.01% 的闪光传奇概率是写在真实代码中的逻辑而非注释——在 100 万用户规模下，约有 100 人会拥有闪光传奇 BUDDY。
- KAIROS 代表"始终在线"的 agent 范式：常驻后台持续监控文件系统变更、预热上下文缓存，等待用户触发时即时响应，而非每次从零启动。
- Undercover 模式揭示了 Claude Code 从设计之初就是可嵌入、可品牌定制的基础设施，而不仅仅是一个终端工具。
- 推测执行通过在用户输入停顿时预运行确定性操作（如预读取文件、预加载缓存），将用户感知的响应延迟缩短约 200ms。
