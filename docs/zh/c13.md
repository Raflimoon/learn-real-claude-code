# 双重特性门控与构建管道

> Bun feature() 在编译时消除代码；GrowthBook/Statsig 在运行时控制特性——两个系统，互补角色。

## 概述

Claude Code 不是一个单一的二进制文件。它的源码通过一套精密的构建管道，生成多个针对不同目标的构建版本：面向终端用户的 CLI 构建、面向 IDE 集成的 Bridge 构建、面向多 agent 场景的 Coordinator 构建，以及代号 KAIROS 的常驻监控构建。

每个构建版本包含不同的特性子集。这是通过两个互补的机制实现的：**编译时的死代码消除**（通过 Bun 的 `feature()` 函数）和**运行时的特性门控**（通过 GrowthBook 和 Statsig）。

理解这套双重门控系统，能帮助你思考如何为多目标、多阶段发布的复杂产品构建可维护的代码库。

## 编译时门控：Bun feature() 函数

Bun 的构建系统支持在打包阶段进行死代码消除（Dead Code Elimination，DCE）。`feature()` 函数在打包时被替换为布尔字面量，未使用的分支被彻底移除：

```typescript
// 源码
if (feature('coordinator-mode')) {
  await runCoordinatorLoop(config);
} else {
  await runStandardLoop(config);
}

// CLI 构建结果（feature = false）
await runStandardLoop(config);  // coordinator 代码完全消失

// Coordinator 构建结果（feature = true）
await runCoordinatorLoop(config);  // standard 代码完全消失
```

这不是运行时的 if/else——在打包后的产物中，未激活的分支根本不存在。这带来两个好处：更小的包体积，以及企业版/内部功能的代码不会泄露到面向用户的构建中。

## 四种构建目标

```
┌─────────────────────────────────────────────────┐
│                  源码（单一代码库）                 │
└────────────┬──────────┬──────────┬──────────────┘
             ↓          ↓          ↓          ↓
          CLI 构建   Bridge 构建  Coordinator  KAIROS
          (用户终端)  (IDE 集成)   (多 agent)  (常驻后台)
             │          │          │          │
         标准工具    精简工具集   编排工具    监控工具集
```

**Bridge 构建**：用于 VS Code、JetBrains 等 IDE 集成。移除了终端 UI（Ink）相关代码，改为通过协议与 IDE 通信。

**Coordinator 构建**：启用多 agent 编排能力，包含 `coordinatorMode.ts` 和 Worker 进程管理逻辑。

**KAIROS 构建**：常驻后台 agent，启用持续监控和环境感知特性（详见 c15）。

## 108 个缺失模块

源码分析揭示了一个有趣的现象：代码库中存在 **108 个被 import 但实际不存在的模块**。

```typescript
// 在源码中出现的 import，但对应文件不存在：
import { EnterpriseSSO } from './enterprise/sso';
import { AuditLogger } from './enterprise/audit-log';
import { TeamPermissions } from './enterprise/permissions';
// ... 共 108 个
```

这些是**存根模块**（stub modules）——企业版功能或尚未发布特性的占位符。它们存在于类型系统中，定义了接口契约，但实现代码存储在私有仓库或尚未开发。编译时门控确保这些 import 在公开构建中被消除，不会导致运行时错误。

## 运行时门控：GrowthBook + Statsig

编译时门控解决的是"哪个构建包含哪些代码"的问题。运行时门控解决的是"同一个构建中，哪些用户看到哪些特性"的问题。

**GrowthBook**：开源特性标志平台，支持基于用户属性的规则引擎：

```typescript
const gb = new GrowthBook({ attributes: { userId, plan, region } });

if (gb.isOn('new-streaming-ui')) {
  // 只有 20% 的 Pro 用户看到新 UI
  renderStreamingUI();
} else {
  renderLegacyUI();
}
```

**Statsig**：更完整的实验平台，支持 A/B 测试和统计显著性分析。Claude Code 用它来测量特性变更对用户行为指标的影响。

## 两个系统的分工

| 维度 | 编译时（Bun feature()） | 运行时（GrowthBook/Statsig） |
|------|----------------------|---------------------------|
| 目的 | 架构分割，多构建目标 | A/B 测试，灰度发布 |
| 粒度 | 整个构建版本 | 单个用户/百分比 |
| 变更成本 | 需要重新打包发布 | 实时调整，无需发布 |
| 代码消除 | 完全移除（DCE） | 代码存在但不执行 |
| 适用场景 | 企业版 vs 社区版 | 新特性 A/B 测试 |

这两个系统不是竞争关系，而是覆盖不同维度的补充。一个特性可能先用编译时门控隔离在特定构建中，再用运行时门控在该构建内做灰度测试。

## 关键收获

- Bun `feature()` 在打包时将特性开关替换为布尔字面量并执行死代码消除，未激活的代码分支在构建产物中完全不存在。
- Claude Code 从单一代码库维护四种构建目标：CLI（终端用户）、Bridge（IDE 集成）、Coordinator（多 agent 编排）、KAIROS（常驻后台监控），各自包含不同的特性子集。
- 108 个缺失模块是企业版或未发布功能的存根占位符，它们定义了接口契约但不包含实现代码，由编译时 DCE 保护不会导致运行时错误。
- GrowthBook 处理基于用户属性的特性标志和规则引擎，Statsig 处理 A/B 测试和统计显著性分析，两者分工明确。
- 双重门控的核心洞察：**编译时分割架构边界，运行时控制用户体验**——一个特性可以先用编译时门控隔离在特定构建中，再用运行时门控在该构建内做灰度发布。
