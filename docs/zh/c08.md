# 五层安全栈与权限竞赛

> Hooks/分类器与用户确认并发竞速——最先返回的获胜。

## 概述

给 AI agent 执行任意代码的能力，同时保持用户安全，是一个极其困难的工程问题。Claude Code 构建了五层安全栈，从硬编码规则到 AI 分类器，层层递进。

但更有意思的是层与层之间的交互方式。并不是所有五层都串行执行——某些层是并发竞速的。理解这个"权限竞赛"模式，对构建既安全又响应迅速的 agent 系统有直接的借鉴价值。

## 五层安全栈

```
┌─────────────────────────────────────────────────────┐
│  层 5：YOLO 分类器                                    │
│  AI 评估操作安全性，动态决策                           │
├─────────────────────────────────────────────────────┤
│  层 4：用户确认对话框                                  │
│  高风险操作请求显式用户批准                            │
├─────────────────────────────────────────────────────┤
│  层 3：工具级权限检查（needsPermission）               │
│  每个工具声明自己的权限需求                            │
├─────────────────────────────────────────────────────┤
│  层 2：权限配置                                       │
│  allowedCommands、allowedPaths、dangerouslyAllowAll  │
├─────────────────────────────────────────────────────┤
│  层 1：硬编码安全规则                                  │
│  永远不执行 rm -rf /，永远不做这些事                   │
└─────────────────────────────────────────────────────┘
```

层级越低，越难绕过。层级越高，越灵活但也越可能被欺骗。

### 层 1：硬编码安全规则

```typescript
// 这些规则用代码写死，无法通过任何配置覆盖
const HARDCODED_DENIES: RegExp[] = [
  /rm\s+-rf\s+\//,          // rm -rf /
  /rm\s+-rf\s+~\b/,         // rm -rf ~
  /:\s*\(\)\s*\{.*\}.*:/,   // Fork bomb
  /dd\s+if=.*of=\/dev\//,   // 覆盖磁盘
  /mkfs\./,                  // 格式化文件系统
];

function isHardcodedDenied(command: string): boolean {
  return HARDCODED_DENIES.some(pattern => pattern.test(command));
}
```

这是第一道防线，在任何其他检查之前执行。无论权限配置如何宽松，这些操作永远被拒绝。

### 层 2：权限配置

用户可以在 `settings.json` 中配置允许或拒绝的操作：

```json
{
  "permissions": {
    "allowedCommands": ["git", "npm", "tsc"],
    "allowedPaths": ["/Users/susu/Projects"],
    "denyCommands": ["curl", "wget"],
    "dangerouslyAllowAll": false
  }
}
```

`dangerouslyAllowAll: true` 是 YOLO 模式的开关——它绕过层 3、4、5，但**不绕过层 1**。硬编码规则始终生效。

### 层 3：工具级权限检查

每个工具通过 `needsPermission(input)` 声明权限需求（第三章已详述）。这一层让权限决策更细粒度——`Bash` 工具对 `ls` 命令不需要权限，但对 `rm` 命令需要。

### 层 4：用户确认对话框

当工具判定需要权限时，触发用户确认：

```typescript
async function requestUserConfirmation(
  tool: Tool,
  input: unknown
): Promise<boolean> {
  // 渲染确认 UI
  const response = await showConfirmationDialog({
    title: `允许 ${tool.name}？`,
    detail: formatInputForDisplay(input),
    options: ['允许', '拒绝', '本次会话始终允许'],
  });

  return response !== '拒绝';
}
```

"本次会话始终允许"选项很重要——它允许用户一次性批准某类操作，避免在密集工作时反复被打断。

## 层 5 与层 4：并发竞速

这是最有意思的设计。在需要权限检查的场景下，层 4（用户确认）和层 5（YOLO 分类器）并发启动：

```typescript
// 权限竞赛：并发执行，第一个安全响应获胜
async function checkPermissionRace(
  tool: Tool,
  input: unknown,
  context: PermissionContext
): Promise<PermissionResult> {
  const results = await Promise.race([
    // 选手 1：AI 分类器快速评估
    yoloClassifier.evaluate(tool, input, context),

    // 选手 2：用户确认（需要人工响应）
    requestUserConfirmation(tool, input),

    // 选手 3：超时保护（不能无限等待）
    timeout(5000).then(() => ({ decision: 'timeout' })),
  ]);

  return interpretResult(results);
}
```

`Promise.race` 的语义是：第一个 resolve 的 Promise 胜出，其余的被取消。

```
时间轴示例：
t=0ms    : 用户确认对话框显示
t=0ms    : YOLO 分类器开始评估
t=150ms  : 分类器返回 "safe" （用户还没反应）
           → 直接放行，用户看到对话框消失

另一个场景：
t=0ms    : 分类器开始评估
t=0ms    : 用户确认对话框显示
t=80ms   : 用户点击"拒绝"
           → 用户确认先返回，操作被拒绝
           → 分类器结果即使后续返回"safe"也被忽略
```

这个设计兼顾了速度（分类器通常很快）和用户控制（用户可以随时覆盖）。

## YOLO 分类器：AI 评估安全性

YOLO 分类器是一个专门训练的 AI 模型，评估工具调用的安全性：

```typescript
async function yoloClassify(
  tool: Tool,
  input: unknown,
  context: PermissionContext
): Promise<ClassifierResult> {
  const verdict = await classifierModel.predict({
    tool: tool.name,
    input: JSON.stringify(input),
    conversationContext: context.recentMessages,
    workingDirectory: context.cwd,
  });

  return {
    safe: verdict.safetyScore > 0.9,
    confidence: verdict.safetyScore,
    reasoning: verdict.explanation,
  };
}
```

分类器考虑的因素：
- 操作是否可逆（删除文件 vs 读取文件）
- 操作的范围（修改单个文件 vs 修改整个目录）
- 当前对话上下文（这个操作在当前任务中是否合理）

## 不可逆操作的额外审查

即使分类器标记为"safe"，某些不可逆操作也受到额外审查：

```typescript
function requiresExtraScrutiny(tool: Tool, input: unknown): boolean {
  const IRREVERSIBLE_OPERATIONS = [
    { tool: 'Bash', pattern: /\brm\b/ },
    { tool: 'Bash', pattern: /\bmv\b.*\/dev\/null/ },
    { tool: 'WriteFile', check: (i: any) => !existsSync(i.path) === false },
  ];

  return IRREVERSIBLE_OPERATIONS.some(op =>
    op.tool === tool.name &&
    (op.pattern?.test(input as string) || op.check?.(input))
  );
}
```

不可逆操作即使分类器认为安全，也仍然展示用户确认对话框。

## 关键收获

- 五层安全栈从硬编码规则到 AI 分类器层层递进，低层规则无法被高层绕过，确保即使高层被欺骗也有底线保护。
- `dangerouslyAllowAll` 绕过层 3-5 的权限检查，但永远不绕过层 1 的硬编码安全规则（如 `rm -rf /` 和 fork bomb）。
- 层 4（用户确认）和层 5（YOLO 分类器）通过 `Promise.race` 并发竞速，第一个返回的响应获胜，兼顾速度与用户控制权。
- 分类器通常在 150ms 内返回结果，但用户随时可以通过点击确认/拒绝来覆盖分类器的决定。
- 不可逆操作（如删除文件、覆盖磁盘）受到额外审查——即使分类器判定为安全，也仍然保留用户确认步骤。
