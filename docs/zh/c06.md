# 四级压缩管线：从 cache_edits 到完整对话摘要

> 200K context window 不是无限资源。四种独立机制按侵入性递增触发：API 层 context edits → 微压缩 → 自动压缩 → 完整摘要。每一级都在「保留信息」和「释放空间」之间选择了不同的平衡点。

---

## 问题

一个 coding session 中，每次 `Read`、`Bash`、`Grep` 调用都在消耗 context window。一轮「读取 → 编辑 → 验证」循环消耗约 8,000 tokens。50 轮工具调用后，context 已超过 150K tokens。

如果不管理，模型撞上 context 上限，API 返回 `prompt_too_long`，整个 session 崩溃。核心矛盾：**旧的工具结果可以丢弃（文件可以重新读取），但对话中的推理链和决策不能丢弃。** 系统必须知道什么可以压缩、什么必须保留——而且必须在不同场景下选择不同的压缩粒度。

---

## 架构总览

```
触发条件            机制                  侵入性    缓存影响
────────────────    ─────────────────     ────────  ──────────────────
API 层阈值          context edits (API)   最低      不影响 prompt cache
每次请求前          微压缩 (microcompact)  低       cache_edits 保留缓存
~93% context 使用   自动压缩 (autocompact) 高       重建缓存前缀
用户手动 /compact   完整/部分摘要          最高      完全重建
```

---

## 第一级：API 层 context management

最轻量的策略不在客户端执行，而是告诉 API 服务器端清理：

```typescript
// services/compact/apiMicrocompact.ts:16-17
const DEFAULT_MAX_INPUT_TOKENS = 180_000      // 触发阈值
const DEFAULT_TARGET_INPUT_TOKENS = 40_000    // 清理后目标
```

两种服务端策略：**`clear_thinking_20251015`** 清除扩展思考块（Opus/Sonnet 的推理过程，单次可达数千 tokens）；**`clear_tool_uses_20250919`** 清除工具调用块。但不是所有工具都能清除：

```typescript
// services/compact/apiMicrocompact.ts:28-32
const TOOLS_CLEARABLE_USES = [
  FILE_EDIT_TOOL_NAME,    // diff 内容已写入磁盘
  FILE_WRITE_TOOL_NAME,   // 写入内容已写入磁盘
  NOTEBOOK_EDIT_TOOL_NAME,
]
```

这些工具的 input 可以安全清除——它们的效果已经持久化到文件系统。模型不需要在 context 中保留完整的 diff 文本来记住"我修改了什么"。

对比之下，`FILE_READ` 和 `GREP` 的结果不在 `TOOLS_CLEARABLE_USES` 中——它们的 tool_use 块（调用参数）必须保留，因为模型需要知道"我读了哪个文件"来维持上下文连贯性。结果内容（tool_result）则由客户端微压缩处理。

## 第二级：微压缩 (Microcompact)

微压缩在每次 API 请求前运行。入口函数路由到两条路径：

```typescript
// services/compact/microCompact.ts:253-293 (简化)
export async function microcompactMessages(messages, toolUseContext?, querySource?) {
  // 路径 1: 缓存已冷（>60 分钟），直接修改消息内容
  const timeBasedResult = maybeTimeBasedMicrocompact(messages, querySource)
  if (timeBasedResult) return timeBasedResult

  // 路径 2: 缓存仍热，使用 cache_edits API 不打断缓存前缀
  if (feature('CACHED_MICROCOMPACT')) {
    // ...模型兼容性检查...
    return await cachedMicrocompactPath(messages, querySource)
  }

  return { messages }  // 无可用路径，交给 autocompact
}
```

### 路径 1: 时间触发

Anthropic 的 prompt cache TTL 是 1 小时。超过 60 分钟无活动后，缓存已经冷了，不如趁机清理：

```typescript
// services/compact/timeBasedMCConfig.ts:30-34
const TIME_BASED_MC_CONFIG_DEFAULTS: TimeBasedMCConfig = {
  enabled: false,
  gapThresholdMinutes: 60,    // 60 分钟无活动
  keepRecent: 5,              // 保留最近 5 个工具结果
}
```

判断逻辑计算距上次 assistant 消息的时间差（`microCompact.ts:422-444`），超过阈值则将旧工具结果替换为 `'[Old tool result content cleared]'`。只有特定工具的结果才会被清理：

```typescript
// services/compact/microCompact.ts:41-50
const COMPACTABLE_TOOLS = new Set<string>([
  FILE_READ_TOOL_NAME, ...SHELL_TOOL_NAMES, GREP_TOOL_NAME, GLOB_TOOL_NAME,
  WEB_SEARCH_TOOL_NAME, WEB_FETCH_TOOL_NAME, FILE_EDIT_TOOL_NAME, FILE_WRITE_TOOL_NAME,
])
```

### 路径 2: Cached 微压缩

缓存仍热时，直接修改消息会打断 prompt cache。Cached MC 通过 `cache_edits` API 告诉服务器"删除这些 tool_result 内容块"，本地消息不修改（`microCompact.ts:305-399`）。核心区别——返回原消息，压缩指令走 `compactionInfo`：

```typescript
// services/compact/microCompact.ts:385-394 (关键返回)
    return {
      messages,  // 消息不修改
      compactionInfo: {
        pendingCacheEdits: {
          trigger: 'auto',
          deletedToolIds: toolsToDelete,
          baselineCacheDeletedTokens: baseline,
        },
      },
    }
```

## 第三级：自动压缩 (Autocompact)

微压缩不足以控制膨胀时，自动压缩启动——fork 一个 agent 做**完整的 LLM 摘要调用**。

### 阈值计算

```typescript
// services/compact/autoCompact.ts:62-65
export const AUTOCOMPACT_BUFFER_TOKENS = 13_000
export const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000
export const ERROR_THRESHOLD_BUFFER_TOKENS = 20_000
export const MANUAL_COMPACT_BUFFER_TOKENS = 3_000
```

effectiveWindow = contextWindow - maxOutputTokens（最大 20K，基于 p99.99 的压缩摘要输出为 17,387 tokens，见 `autoCompact.ts:29-30`）。autocompact 阈值 = effectiveWindow - 13,000。以 200K 模型为例：180,000 - 13,000 = 167,000，约 **~93%** 使用率触发。

环境变量 `CLAUDE_CODE_AUTO_COMPACT_WINDOW` 可以覆盖 context window 大小（用于测试）；`CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` 可以按百分比设置触发阈值。

### 触发判断与递归保护

`shouldAutoCompact()`（`autoCompact.ts:160-239`）包含多重守卫：compact 和 session_memory 的 forked agent 不触发（防死锁）；Context Collapse 模式下禁用（避免竞争）。源码注释确认了设计意图：

```
// autoCompact.ts:201-205
// Context-collapse mode: the 90% commit / 95% blocking-spawn
// flow owns the headroom problem. Autocompact firing at ~93%
// sits right between collapse's commit-start (90%) and blocking (95%),
// so it would race collapse and usually win.
```

### 熔断器 (Circuit Breaker)

```typescript
// services/compact/autoCompact.ts:67-70
// BQ 2026-03-10: 1,279 sessions had 50+ consecutive failures (up to 3,272)
// in a single session, wasting ~250K API calls/day globally.
const MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3
```

连续 3 次失败后停止重试。没有这个保护，context 不可恢复地超限的 session 会无限循环尝试压缩。失败计数在成功压缩时重置为 0（`autoCompact.ts:332`），所以短暂的失败不会永久禁用压缩。

## 第四级：完整对话摘要

`compactConversation()`（`compact.ts:387-749`）发起完整 LLM 摘要。摘要后的消息结构：

```typescript
// services/compact/compact.ts:330-338
export function buildPostCompactMessages(result: CompactionResult): Message[] {
  return [
    result.boundaryMarker,           // 压缩边界标记
    ...result.summaryMessages,       // AI 生成的摘要
    ...(result.messagesToKeep ?? []),  // 保留的消息（部分压缩时）
    ...result.attachments,           // 文件附件重新注入
    ...result.hookResults,           // hook 结果
  ]
}
```

### 部分压缩：两个方向

`partialCompactConversation()`（`compact.ts:772-998`）支持选择性压缩：

- **`'up_to'`**：摘要 pivot 之前的消息，保留之后的（旧变摘要，新保留原样）。
- **`'from'`**：摘要 pivot 之后的消息，保留之前的（保留早期缓存前缀）。

`'up_to'` 方向过滤掉旧的压缩边界标记和摘要消息（`compact.ts:793-799`），否则嵌套摘要会导致 `findLastCompactBoundaryIndex` 的反向扫描定位到错误的边界。

另外，当压缩请求本身因 context 过大触发 `prompt_too_long` 时，`truncateHeadForPTLRetry()`（`compact.ts:243-291`）会按 API-round 分组丢弃最早的消息，最多重试 3 次。这是最后的逃生舱口——确保即使在极端情况下，用户也不会卡死在无法压缩的状态。

### 压缩后清理

`runPostCompactCleanup()`（`postCompactCleanup.ts:31-77`）重置一系列缓存。关键设计——区分主线程和子 agent：

```typescript
// services/compact/postCompactCleanup.ts:36-39
  const isMainThreadCompact =
    querySource === undefined ||
    querySource.startsWith('repl_main_thread') ||
    querySource === 'sdk'
```

子 agent（session_memory、prompt_suggestion）在同一进程中共享模块级状态。如果子 agent 压缩时重置主线程缓存，主线程下次请求会丢失状态。

---

## 为什么不用单一压缩策略

**被拒绝的方案：只用完整摘要。** 一次摘要调用消耗 ~20K output tokens，耗时约 3 秒，还会摧毁整个 prompt cache 前缀。微压缩在毫秒级完成且通过 `cache_edits` 保留缓存。设计原则：**能用便宜的就不用贵的，能保留缓存的就不打断缓存。**

## 为什么不一次压缩所有旧内容

**被拒绝的方案：达到阈值时一次性摘要全部。** `cache_edits` 让服务器端删除特定 tool_result，消息数组的 token 前缀不变，缓存继续命中。一次性替换为摘要会让 180K tokens 全部按 cache_creation 价格计费（Opus: $3.75/M），而非 cache_read 价格（$0.30/M）。

时间触发微压缩先检查缓存冷热（`microCompact.ts:261-269` 注释）：缓存冷了才修改消息内容；缓存热的时候用 `cache_edits` 精准删除。

## Token 预算与边际递减

与压缩系统配合的 token budget 机制控制模型何时停止：

```typescript
// query/tokenBudget.ts:3-4
const COMPLETION_THRESHOLD = 0.9   // 90% 预算用完时停止
const DIMINISHING_THRESHOLD = 500  // 连续增量 < 500 tokens 视为边际递减
```

边际递减检测——连续 3 次迭代产出不到 500 tokens 时提前终止（`tokenBudget.ts:59-62`），防止模型空转。这与压缩系统互补：压缩控制 context 输入的膨胀，budget 控制输出的效率。两者共同保证长任务既不会撑爆 context，也不会无意义地消耗 tokens。

## 验证

1. **对比 token 用量。** 打开一个长对话，执行 20+ 次工具调用后运行 `/cost` 记录 token 消耗。执行 `/compact`，再运行 `/cost`。150K tokens 的对话典型压缩到 30-50K tokens。
2. **检查摘要质量。** `/compact` 后模型的第一条回复会基于摘要继续工作。验证它是否记住了关键决策和文件修改——如果丢失了重要上下文，说明摘要 prompt 可能需要自定义指令（`/compact` 支持传入 `customInstructions`）。
3. **检查压缩日志。** `~/.claude/sessions/` 的 JSONL 文件中搜索 `compactBoundary` 类型消息，定位压缩发生的位置。
4. **观察自动触发。** 设置 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=50` 降低阈值，执行几次文件读取后观察 `Compacting conversation...` 提示。
5. **对比缓存命中率。** `/cost` 中关注 cache_read vs cache_creation 比例。`/compact` 后首次请求 cache_creation 飙升，后续恢复。

## 关键收获

- 四级管线按侵入性递增：API 层 context edits（零客户端成本）→ 微压缩（毫秒级，缓存感知）→ 自动压缩（~93% 触发，fork agent 摘要）→ 手动 `/compact`（最大压缩比，打断缓存）。
- 微压缩两条路径体现「缓存感知」：缓存热时用 `cache_edits` API 不修改本地消息；缓存冷时（60 分钟无活动）直接替换内容。
- 熔断器限制连续自动压缩失败最多 3 次——线上数据：1,279 个 session 因无限重试每天浪费 250K 次 API 调用。
- 部分压缩支持 `'up_to'`（摘要旧保留新）和 `'from'`（保留旧摘要新）两个方向，方向选择决定 prompt cache 前缀是否存活。
- 压缩后清理区分主线程和子 agent，防止共享模块级状态的子 agent 损坏主线程缓存。
- Token budget 的边际递减检测（连续 3 次 delta < 500 tokens）防止模型空转，与压缩系统互补。
