# MCP：无限扩展协议

> 18 个延迟工具通过 ToolSearchTool 按需加载，将基础提示词保持在 200K token 以下。

## 概述

Claude Code 拥有超过 18 个"延迟工具"（deferred tools）——它们存在于代码库中，但在会话开始时并不加载到模型的上下文里。只有当模型主动调用 `ToolSearchTool` 搜索相关工具时，这些工具才会被按需实例化并注入到上下文中。

这个设计解决了一个真实的工程问题：如果把所有 MCP 工具的 schema 都塞进基础系统提示词，仅工具描述就会超过 200K token。对于大多数任务来说，这是巨大的浪费——你不需要 Slack 工具来修改一个 Python 文件。

Model Context Protocol（MCP）是这一切的标准化基础。它定义了外部工具如何与 Claude 集成，而延迟加载则是 Claude Code 在 MCP 之上构建的优化层。

## MCP 的核心概念

MCP 是 Anthropic 提出的开放协议，定义了 AI 模型与外部工具之间的通信标准。从模型的角度看，MCP 工具与原生工具（如 `Bash`、`Read`、`Edit`）**完全相同**——同样的 JSON Schema 描述，同样的调用方式，同样的结果格式。

```
模型视角：
  原生工具  → Bash({ command: "ls" })
  MCP 工具  → mcp__github__create_pr({ title: "fix", branch: "main" })
              ↑ 完全相同的调用接口
```

这种透明性是 MCP 的核心价值：工具的实现可以是任何东西（本地进程、远程服务、数据库），但对模型来说，调用体验是统一的。

## 三种传输类型

MCP 支持三种传输机制，适用于不同的部署场景：

**stdio（子进程）**

最常见的本地工具形式。Claude Code 生成一个子进程，通过标准输入/输出与 MCP 服务器通信：

```
Claude Code Process
    │
    └── spawn: node mcp-server.js
              stdin  ← JSON-RPC 请求
              stdout → JSON-RPC 响应
```

**SSE（Server-Sent Events）**

适用于远程 HTTP 服务。服务器通过 SSE 流式推送事件，Claude Code 通过 HTTP POST 发送请求：

```
Claude Code → POST /mcp/call → Remote MCP Server
                              ← SSE stream (result)
```

**WebSocket**

双向实时通信，适合需要主动推送通知的场景（如文件监听、数据库变更）。

## ToolSearchTool：元工具

`ToolSearchTool` 是整个延迟加载机制的关键——它是一个**关于工具的工具**。

```typescript
// ToolSearchTool 的简化接口
interface ToolSearchParams {
  query: string;       // 自然语言描述，如 "github pull request"
  max_results?: number; // 最多返回几个工具定义
}
```

当模型需要一个它不确定是否存在的能力时，它会先搜索：

```
模型思考：我需要创建一个 GitHub PR...
  → 调用 ToolSearch({ query: "github pull request create" })
  → 返回：mcp__github__create_pr 的完整 schema
  → 现在可以调用这个工具了
```

这个模式有一个优雅的递归性质：工具系统可以无限扩展，而模型永远只需要知道"如何搜索工具"这一件事。

## 延迟加载的经济学

为什么 200K token 是关键数字？

```
基础系统提示：       ~15K tokens
工具 schema（全加载）: ~85K tokens（如果预加载所有 MCP 工具）
对话历史 + 代码：     ~100K tokens
---------------------------------
总计：               ~200K tokens  ← 已接近多数模型上下文窗口上限
```

通过延迟加载，工具 schema 的基础成本降低到只需要 `ToolSearchTool` 本身的描述（约 500 tokens）。其余工具只在被调用时才占用上下文空间，用完即可丢弃（或保留，取决于任务需求）。

## 工具桥接：对模型透明

一个重要的实现细节：MCP 工具在内部会被"桥接"为与原生工具相同的格式。这意味着：

- 模型不需要知道某个工具是"本地原生"还是"通过 MCP 接入"
- 工具描述、参数验证、错误格式都经过统一化处理
- 开发者添加新 MCP 服务器时，不需要修改模型的系统提示

这种透明桥接是 MCP 协议最重要的设计决策之一。

## 关键收获

- MCP 是外部工具集成的标准协议，MCP 工具对模型完全透明——使用与原生工具（Bash、Read、Edit）相同的 JSON Schema 和调用接口。
- 三种传输类型覆盖不同部署场景：stdio 用于本地子进程通信，SSE 用于远程 HTTP 服务，WebSocket 用于需要双向实时推送的场景。
- 延迟工具加载是 Claude Code 在 MCP 之上构建的优化层：18 个工具按需注入上下文，基础提示词开销仅为 `ToolSearchTool` 自身描述的约 500 token。
- `ToolSearchTool` 是一个关于工具的元工具，赋予模型自主发现和加载能力的能力，使工具系统可以无限扩展而不增加基础 context 成本。
- 关键约束：预加载所有 MCP 工具的 schema 会使基础提示词超过 200K token，延迟加载不是优化偏好而是工程上的必要选择。
