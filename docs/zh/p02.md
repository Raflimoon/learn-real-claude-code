# Agent 脚手架：框架比提示词更重要

> query.ts 的 1,729 行中，LLM 调用本身只占 1 行。其余 1,728 行决定了这 1 行能否可靠执行。

## 概述

"5 分钟构建 AI agent"的教程和 Claude Code 之间的差距，不在于提示词质量，而在于 **scaffolding** -- 围绕 LLM 调用的框架代码。这个判断听起来抽象，但 Claude Code 的源码给出了精确的量化证据。

本章不再用伪代码类比。我们直接拆解 `query.ts`（1,729 行）和 `cli.tsx`（302 行），用行号和行数证明脚手架的真实规模。

## 源码洞察 1：query.ts 的 1,729 行在做什么

`query.ts` 是 Claude Code 的核心 query loop -- 每次用户发送消息后，模型生成响应、调用工具、再次生成响应的完整循环。它只有 3 个顶层函数（`yieldMissingToolResultBlocks`、`query`、`queryLoop`），但覆盖了 agent 运行时的全部复杂性。

**实际的 LLM 调用发生在哪里？**

`query.ts:659` -- `deps.callModel()` 调用。这是一个 `for await` 循环的数据源，从 streaming API 接收消息。这就是"LLM 调用本身"。

**那其余 1,728 行在做什么？** 以下是按行号范围的功能拆解：

### 状态管理（~80 行）

```
query.ts:181-199   QueryParams 类型定义 -- 18 个参数字段
query.ts:202-218   State 类型定义 -- 12 个可变状态字段
query.ts:252-280   初始状态构造 + budgetTracker
```

State 类型（`query.ts:202-218`）包含：messages、toolUseContext、autoCompactTracking、maxOutputTokensRecoveryCount、hasAttemptedReactiveCompact、maxOutputTokensOverride、pendingToolUseSummary、stopHookActive、turnCount、transition。每一个字段都对应一个你在 50 行 demo 中不会处理的 edge case。

### 上下文预处理管道（~150 行）

```
query.ts:369-394   applyToolResultBudget -- per-message 工具结果大小限制
query.ts:396-410   snipCompact -- history snip 预处理
query.ts:412-427   microcompact -- API-based 工具结果清除
query.ts:440-447   context collapse -- 折叠投影
query.ts:449-468   autocompact -- 对话级压缩
query.ts:470-543   compaction 结果处理和 task budget 追踪
```

在模型被调用之前，消息数组经过至少 5 道预处理。注意顺序：snip -> microcompact -> context collapse -> autocompact。这不是随意排列 -- `query.ts:396` 的注释说明 snip 先于 microcompact，因为"both may run -- they are not mutually exclusive"。

### Streaming 消息处理（~200 行）

```
query.ts:654-863   for await 流式消息循环
query.ts:748-787   backfillObservableInput -- 工具输入字段回填
query.ts:799-825   withheld 错误判断（prompt_too_long, max_output_tokens, media_size）
query.ts:826-862   streaming tool executor 集成
```

流式消息不是简单的 `stream.on('text', console.log)`。每条消息都需要判断是否应该被"扣留"（withheld）-- 即暂不发送给用户，等待恢复机制尝试处理。`query.ts:799-822` 展示了四种独立的扣留判断，它们可以同时生效。

### 错误恢复机制（~300 行）

```
query.ts:893-953    FallbackTriggeredError -- 模型降级重试
query.ts:955-997    通用错误处理（image errors, API errors）
query.ts:1062-1183  prompt_too_long 恢复链：
                    1. context collapse drain（1090-1117）
                    2. reactive compact（1119-1183）
query.ts:1188-1256  max_output_tokens 恢复链：
                    1. 8K -> 64K 升级（1199-1221）
                    2. recovery message 注入（1223-1252）
                    3. 最多重试 3 次（MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3）
```

这是 query.ts 中最精妙的部分。`prompt_too_long` 错误的恢复是一个有序的 fallback 链：先尝试 context collapse drain（轻量级），失败后尝试 reactive compact（完整压缩），再失败才报错。`max_output_tokens` 恢复有三层：先尝试从 8K 升级到 64K（单次，`query.ts:1199`），然后注入 "Resume directly -- no apology, no recap" 消息让模型继续，最多 3 次。

### 工具执行编排（~350 行）

```
query.ts:560-568   StreamingToolExecutor 初始化
query.ts:838-862   streaming 中收集已完成的工具结果
query.ts:1012-1052 abort 处理（streaming abort 和 tool abort 两种路径）
query.ts:1360-1408 非 streaming 工具执行 fallback
query.ts:1411-1482 tool use summary 异步生成（Haiku 模型）
```

工具执行有两种模式：StreamingToolExecutor（模型还在生成时就开始执行工具）和传统模式（等模型完成后批量执行）。streaming 模式下，当用户中断时需要为所有进行中的工具生成 synthetic tool_result blocks（`query.ts:1016-1029`），否则 API 会因为 tool_use 缺少对应 tool_result 而报错。

### Telemetry 和 analytics（~100 行）

散布在整个文件中，`logEvent('tengu_*', {...})` 调用记录了每个关键决策点。这些事件名以 `tengu_` 为前缀 -- 这是 Claude Code 的内部代号。

### Turn 管理和循环控制（~200 行）

```
query.ts:1267-1306  stop hooks -- 模型完成后检查是否需要继续
query.ts:1308-1355  token budget -- +500K 自动继续功能
query.ts:1523-1527  turnCounter + maxTurns 限制
query.ts:1580-1628  attachment messages 注入（memory prefetch、skill discovery）
query.ts:1659-1671  MCP 工具刷新
query.ts:1704-1712  maxTurns 检查
query.ts:1715-1727  下一轮 state 构造
```

## 源码洞察 3：50 行 demo agent vs 1,729 行 query.ts

一个典型的 demo agent 是这样的：

```typescript
// 50 行 demo agent
const response = await anthropic.messages.create({
  model: 'claude-sonnet-4-6', messages, tools: TOOLS,
});
for (const block of response.content) {
  if (block.type === 'tool_use') {
    const result = await executeTool(block.name, block.input);
    messages.push(/* tool_result */);
  }
}
```

**query.ts 的 1,679 行"额外"代码在做什么？** 精确分解：

| 功能类别 | 行数（估） | demo 中是否存在 |
|---------|-----------|---------------|
| 状态管理（12 个可变字段） | ~80 | 无 |
| 上下文预处理管道（5 层） | ~150 | 无 |
| Streaming 消息处理 | ~200 | 无（demo 用 await） |
| 错误恢复（3 条 fallback 链） | ~300 | 无（demo throw） |
| 工具执行编排（streaming + abort） | ~350 | 无（demo 同步执行） |
| Telemetry | ~100 | 无 |
| Turn 管理和循环控制 | ~200 | 无（demo 无限循环） |
| 类型定义和导入 | ~250 | ~5 行 |
| **总计** | **~1,630** | **~50** |

demo agent 是一个 `while(true)` 循环，没有退出条件、没有错误分类、没有 context 管理。query.ts 的 `while(true)`（`query.ts:307`）虽然形式相同，但它的每次迭代都经过了 5 层预处理、4 类错误恢复、3 种终止条件检查。

### 为什么不用 LangChain 的现成 agent loop？

`query.ts` 是自建的，不依赖任何框架。原因在源码中可以读出来：

1. **State 的粒度**。`query.ts:202-218` 的 State 类型有 12 个字段，其中 `transition`（`query.ts:216`）记录了上一次迭代为什么 continue -- 这让后续迭代能根据前一次的恢复路径决定下一步策略。框架的抽象层不提供这种粒度。

2. **Continue sites 的多样性**。query.ts 有 7 处 `state = next; continue` -- 每一处都是不同的恢复场景（collapse drain retry、reactive compact retry、max output tokens escalate、max output tokens recovery、stop hook blocking、token budget continuation、next turn）。框架的 `onError` hook 无法区分这些场景。

3. **Generator 架构**。`query()` 是一个 `AsyncGenerator`（`query.ts:219-238`），它 `yield` 消息给调用方，让 UI 层能实时渲染。这种反压控制在框架的 callback-based 架构中很难实现。

### 为什么不把错误恢复拆分成独立模块？

恢复逻辑确实引用了外部模块（`reactiveCompact.ts`、`contextCollapse/index.ts`），但编排逻辑（"先试 A，失败后试 B"）必须在 query loop 内部 -- 因为恢复涉及修改 `state` 并 `continue` 到下一次迭代。如果拆到外部，就需要在恢复模块和 query loop 之间传递 State，增加了不必要的耦合。

## 源码洞察 4：transition 字段 -- 脚手架的"记忆"

query.ts 的 State 类型中有一个字段值得特别关注（`query.ts:216`）：

```typescript
// query.ts:216
transition: Continue | undefined
```

`transition` 记录了"上一次 loop 迭代为什么 continue"。这个字段不影响 LLM 调用 -- 模型永远看不到它。它的唯一消费者是 query loop 自身的恢复逻辑。

具体例子：`query.ts:1092-1093` 检查上一次 transition 是否是 `collapse_drain_retry`：

```typescript
// query.ts:1092-1093
if (state.transition?.reason !== 'collapse_drain_retry') {
  // ... 尝试 collapse drain
}
```

如果上一次已经尝试过 collapse drain 但 API 仍然返回 413，就不再重试 collapse drain，而是直接跳到 reactive compact。这种"恢复路径感知"是 demo agent 中完全不存在的概念 -- demo agent 要么无限重试，要么直接崩溃。

**7 个 transition reason 的完整列表：**

1. `collapse_drain_retry` -- context collapse 清除后重试
2. `reactive_compact_retry` -- 完整压缩后重试
3. `max_output_tokens_escalate` -- 从 8K 升级到 64K
4. `max_output_tokens_recovery` -- 注入 recovery message 继续
5. `stop_hook_blocking` -- stop hook 阻止后重试
6. `token_budget_continuation` -- token 预算自动继续
7. `next_turn` -- 正常的下一轮工具执行

每一个 reason 都编码了一个特定的恢复语义。这是脚手架的"记忆" -- 它让 query loop 在 continue 之后仍然知道自己从哪里来、为什么来。

### 为什么不用 retry count 替代 transition reason？

因为 retry count 只知道"重试了几次"，不知道"为什么重试"。collapse drain 失败后跳到 reactive compact，和 reactive compact 失败后报错，是两种完全不同的语义。transition reason 编码了恢复链中的位置，而不仅仅是次数。

## 关键收获

- query.ts 的 1,729 行中，LLM 调用本身只有 1 行（`deps.callModel()`，第 659 行），其余 1,728 行是状态管理、上下文预处理、streaming 处理、错误恢复、工具编排、telemetry 和 turn 管理 -- 这是"脚手架比提示词更重要"的最精确量化证据。
- 每次 LLM 调用前，消息数组经过 5 层预处理管道（tool result budget -> snip -> microcompact -> context collapse -> autocompact），顺序不可互换，每层解决不同粒度的 context 管理问题。
- 错误恢复不是简单的 try/catch：prompt_too_long 有 3 级 fallback 链（collapse drain -> reactive compact -> 报错），max_output_tokens 有 3 级恢复机制（8K->64K 升级 -> recovery message 注入 -> 最多 3 次重试），模型不可用有模型降级（FallbackTriggeredError）。
- cli.tsx 的 302 行中有 15 种 fast-path 分流，展示了生产 CLI 的真实复杂度：一个二进制文件服务于 daemon worker、bridge、background session、template job、environment runner、self-hosted runner 等多种运行模式。
- query.ts 不使用框架的核心原因是 7 处不同的 continue 站点，每一处对应不同的恢复场景 -- 这种"恢复上下文感知"的循环控制是框架的抽象层无法提供的。
