# Agent 脚手架：框架比提示词更重要

> 演示 agent 与生产 agent 的区别几乎完全在于脚手架。

## 概述

如果你参加过任何 AI 开发的 workshop 或看过"5 分钟构建 AI agent"的教程，你见过的那种 agent 和 Claude Code 之间的差距，大部分不在于提示词的质量，而在于**脚手架**（scaffolding）——围绕 LLM 调用的框架代码。

这个观察反直觉，但重要。当你能用 10 行代码构建一个"可以工作"的 agent 时，为什么 Claude Code 有 512K 行代码？因为那 512K 行几乎都是脚手架：流式处理、并发工具执行、错误恢复、上下文管理、权限控制、成本追踪……

提示词决定模型"想"什么，脚手架决定这些"想法"能否可靠地执行。

## 朴素 Agent vs 生产级 Agent

来看一个具体对比：

**朴素 agent（~20 行）**：

```typescript
async function naiveAgent(userMessage: string) {
  const messages = [{ role: 'user', content: userMessage }];

  while (true) {
    const response = await anthropic.messages.create({
      model: 'claude-3-5-sonnet',
      messages,
      tools: TOOLS,
    });

    if (response.stop_reason === 'end_turn') break;

    // 执行工具调用
    for (const block of response.content) {
      if (block.type === 'tool_use') {
        const result = await executeTool(block.name, block.input);
        messages.push({ role: 'tool', content: result });
      }
    }
    messages.push({ role: 'assistant', content: response.content });
  }

  return response;
}
```

这能工作。对于演示、原型验证、简单任务，它完全够用。

**生产级 agent（~500 行核心逻辑）**：

```typescript
async function productionAgent(userMessage: string, config: AgentConfig) {
  const stream = anthropic.messages.stream({ /* ... */ });

  // 1. 流式处理：不等待完整响应，边生成边显示
  stream.on('text', (text) => renderTokenByToken(text));

  // 2. 并发工具执行：多个工具调用并行运行
  const toolCalls = await collectToolCallsFromStream(stream);
  const results = await Promise.allSettled(
    toolCalls.map(call => circuitBreaker.execute(() =>
      executeToolWithTimeout(call, config.toolTimeoutMs)
    ))
  );

  // 3. 熔断器保护：连续失败 3 次断路
  // 4. Context 预算检查：超过 80% 时触发压缩
  if (tokenCounter.getUsage() > config.maxTokens * 0.8) {
    await compressContext(messages);
  }

  // 5. 权限验证：每个工具调用前检查
  // 6. 成本追踪：每轮 API 调用后记录
  // 7. 错误分类：区分可重试错误和致命错误
  // ...
}
```

功能差距不在 LLM 调用本身——那一行是一样的。差距在于 LLM 调用前后的处理。

## Claude Code 脚手架的五个教训

**教训 1：始终流式传输**

不要等待完整响应。流式传输有两个好处：用户立即看到输出（感知延迟降低），以及你可以在生成过程中提前识别工具调用并开始准备执行。

```typescript
// 等待完整响应：用户盯着空屏幕 3-8 秒
const response = await client.messages.create({ stream: false });

// 流式处理：第一个 token 通常在 200-500ms 内出现
const stream = client.messages.stream({ stream: true });
stream.on('text', chunk => process.stdout.write(chunk));
```

**教训 2：工具应声明并发安全性**

并发执行工具会加速任务，但不是所有工具都是并发安全的。读文件是安全的，但"读后写"序列如果并发执行会产生竞争条件：

```typescript
interface Tool {
  name: string;
  concurrencySafe: boolean;  // Claude Code 有这个字段
  execute: (params: any) => Promise<any>;
}

// 调度逻辑：并发安全的工具可以并行，否则串行
const [safe, unsafe] = partition(toolCalls, t => t.tool.concurrencySafe);
await Promise.all(safe.map(execute));        // 并行
for (const call of unsafe) await execute(call); // 串行
```

**教训 3：显式管理 Token 预算**

永远不要假设"上下文足够用"。在每次 API 调用前检查剩余预算，在使用率达到阈值时主动压缩：

```typescript
const COMPRESSION_THRESHOLD = 0.8;  // 80% 使用率时触发

if (currentTokens / maxTokens > COMPRESSION_THRESHOLD) {
  messages = await compressOldMessages(messages);
}
```

**教训 4：在 LLM 调用层熔断**

熔断器应该包裹整个 LLM 调用，而不只是工具执行。工具失败可以重试，但 LLM API 持续不可用时，你需要在更高层断路。

**教训 5：错误分类比错误处理更重要**

不是所有错误都应该重试。速率限制（429）→ 退避重试；认证失败（401）→ 立即终止；上下文过长（400）→ 压缩后重试；网络超时 → 短暂退避。

```typescript
function classifyError(err: APIError): ErrorStrategy {
  if (err.status === 429) return { action: 'retry', delay: 'exponential' };
  if (err.status === 401) return { action: 'abort', message: '认证失败' };
  if (err.status === 400 && err.type === 'context_length') return { action: 'compress' };
  return { action: 'retry', delay: 'linear', maxAttempts: 3 };
}
```

## 框架 vs 自建脚手架

LangChain、LlamaIndex 等框架提供了脚手架的现成实现。为什么 Claude Code 选择自建？

几个观察：
- 框架的抽象层会遮蔽底层行为，难以精细控制
- 框架的版本依赖会成为安全更新的阻力
- 自建代码更容易根据具体需求优化

这不是说框架不好——对于中小型项目，框架的开发速度优势显著。但对于 Claude Code 这种需要精细控制每个 token、每次重试、每个并发决策的产品，自建脚手架是合理选择。

## 关键收获

- 脚手架（围绕 LLM 调用的框架代码）通常比提示词本身对生产质量的影响更大，因为它决定了模型的"想法"能否可靠执行。
- 朴素 agent 和生产 agent 的功能差距几乎全部来自脚手架层：流式处理、并发工具执行、错误恢复、token 预算管理等。
- Claude Code 的五个核心脚手架教训：始终流式传输以降低感知延迟、让工具声明自身并发安全性、显式管理 token 预算并在阈值时压缩、在 LLM 调用层而非工具层设置熔断器、错误分类（区分可重试与致命错误）优先于笼统的错误处理。
- 框架（LangChain 等）提供开发速度，自建脚手架提供精细控制——选择取决于你对底层行为的控制需求和项目规模。
- 生产 agent 的本质是 20% 的 LLM 调用逻辑加上 80% 让这些调用可靠执行的管道代码，Claude Code 的 512K 行源码正是这一比例的佐证。
