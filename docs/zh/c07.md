# 记忆与梦境：跨会话的持久化

> Auto-Dream 运行四个阶段（定向→收集→整合→剪枝），使用只读 bash 子 agent。

## 概述

每个 LLM 会话本质上是健忘的——关闭终端，下次重新打开，模型对你是谁、你在做什么一无所知。Claude Code 通过两种机制解决这个问题：基于文件的持久记忆（通过 `CLAUDE.md`），以及一个叫做 Auto-Dream 的自动记忆整合过程。

理解这套机制，对构建有状态的 agent 系统至关重要。跨会话的持久化不只是"保存一些文本"——它需要解决合并冲突、去除冗余、识别过时信息等一系列问题。Claude Code 的 Dream 机制提供了一个经过验证的参考方案。

## CLAUDE.md：磁盘上的记忆

记忆的底层机制非常简单：写文件。当 Claude 需要"记住"某件事时，它调用工具把信息写入 `~/.claude/memory/` 下的 Markdown 文件：

```typescript
// 写入记忆的工具调用（概念性重现）
await writeFileTool.invoke({
  path: '~/.claude/memory/MEMORY.md',
  content: appendMemory(existingContent, {
    timestamp: new Date().toISOString(),
    topic: 'User preference',
    content: 'User prefers TypeScript strict mode',
  }),
});
```

记忆文件以结构化 Markdown 写入，包含时间戳：

```markdown
## User Preferences

### 2026-03-15T14:30:00Z — 代码风格
- TypeScript strict mode 偏好
- 函数式风格优于面向对象
- 使用 Zod 进行运行时验证

### 2026-03-20T09:15:00Z — 项目约定
- 使用 pnpm 而非 npm
- 测试框架：vitest
```

每次会话启动时，这些记忆文件被加载到系统提示词的动态部分（参见第五章），成为模型上下文的一部分。

## Auto-Dream：四阶段记忆整合

随着时间推移，记忆文件会变得越来越大、越来越冗余。早期记录的偏好可能已经改变；同一个事实可能被记录了三次；有些信息已经过时。Auto-Dream 是处理这个问题的机制。

当用户（或系统）触发"做梦"模式时，Claude 启动一个特殊的子 agent，执行四个阶段：

```
Phase 1: Orient（定向）
    ↓ 读取所有现有记忆文件
    ↓ 建立记忆的全局视图

Phase 2: Gather（收集）
    ↓ 扫描当前对话历史
    ↓ 识别值得持久化的新事实

Phase 3: Consolidate（整合）
    ↓ 合并新事实与现有记忆
    ↓ 去重，解决矛盾

Phase 4: Prune（剪枝）
    ↓ 删除过时/不相关的记忆
    ↓ 写回更新后的记忆文件
```

### Phase 1：Orient

```typescript
// Orient 阶段：建立记忆全局视图
async function orient(agent: DreamAgent): Promise<MemorySnapshot> {
  const memoryFiles = await agent.bash('ls ~/.claude/memory/');
  const contents = await Promise.all(
    memoryFiles.map(f => agent.readFile(f))
  );

  return {
    files: memoryFiles,
    totalSize: contents.reduce((acc, c) => acc + c.length, 0),
    topics: extractTopics(contents),
    snapshot: contents,
  };
}
```

### Phase 2：Gather

```typescript
// Gather 阶段：从对话中提取值得记住的事实
async function gather(
  agent: DreamAgent,
  conversation: Message[]
): Promise<NewFact[]> {
  // Dream agent 使用 AI 调用分析对话
  const analysis = await agent.analyze({
    task: '从以下对话中识别值得长期记忆的信息：用户偏好、项目约定、重要决策、联系人信息等',
    conversation,
  });

  return analysis.facts.filter(f => f.importance > 0.7);
}
```

### Phase 3：Consolidate

这是最复杂的阶段——需要合并可能相互矛盾的信息：

```typescript
async function consolidate(
  existing: MemorySnapshot,
  newFacts: NewFact[]
): Promise<UpdatedMemory> {
  const conflicts = findConflicts(existing, newFacts);

  // 对于冲突，使用更新的事实（带时间戳）
  const resolved = resolveConflicts(conflicts, { preferNewer: true });

  // 去重：语义相似的事实合并为一条
  const deduplicated = await semanticDeduplicate([
    ...existing.snapshot,
    ...newFacts.map(f => f.content),
  ]);

  return { content: deduplicated, resolved };
}
```

### Phase 4：Prune

```typescript
async function prune(memory: UpdatedMemory): Promise<void> {
  const pruned = memory.content.filter(fact => {
    // 删除超过 180 天未被引用的事实
    if (fact.lastReferenced < daysAgo(180)) return false;
    // 删除已被明确否定的事实
    if (fact.negated) return false;
    return true;
  });

  await writeMemoryFiles(pruned);
}
```

## 只读 bash 子 agent

Dream 过程的一个关键安全设计：Dream agent 是**只读的**。它可以读取文件、执行 `ls`、`cat` 等命令，但不能执行任何写操作——除了最终写回整合后的记忆文件。

```typescript
// Dream agent 的权限配置
const dreamAgent = createSubAgent({
  permissions: {
    bash: {
      allowedCommands: ['ls', 'cat', 'find', 'grep', 'wc'], // 只读命令
      denied: ['rm', 'mv', 'cp', 'write', 'chmod'],         // 禁止写操作
    },
    fileSystem: {
      read: true,
      write: false,  // 只有 Dream 的最终写入步骤例外
    },
  },
});
```

这个设计防止了 Dream 过程意外修改代码或配置文件——它只能读取和分析，由顶层机制完成最终写入。

## 关键收获

- 持久记忆的底层机制是文件写入：将结构化 Markdown 存储在 `~/.claude/memory/` 目录下，简单可靠且支持版本控制。
- 记忆文件在每次会话启动时加载到系统提示词的动态部分，使模型能够"回忆"跨会话的用户偏好和项目约定。
- Auto-Dream 触发四阶段流程：Orient（读取现有记忆建立全局视图）→ Gather（从对话中提取新事实）→ Consolidate（合并去重并解决矛盾）→ Prune（删除过时内容）。
- Dream 子 agent 被限制为只读权限，防止记忆整合过程意外修改代码或配置文件，只有最终写入步骤例外。
- 冲突解决策略采用"新优先"原则：更新的时间戳覆盖旧记录，语义重复的条目合并为一条以控制记忆文件体积。
