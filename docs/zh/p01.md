# Context Engineering：2026 范式转移

> 源码中每一个 token 阈值常量，都编码了一个关于注意力衰减和缓存成本的工程决策。

## 概述

2024 年的 AI 应用开发范式是"提示词工程"：写更清晰的指令、更好的角色设定、更精确的约束。到了 2026 年，决定 agent 输出质量的关键因素已经转移：从"你写了什么静态提示词"，转向"模型在每次调用时的 context window 里装了什么"。这个范式转移叫 **Context Engineering**。

Claude Code 的源码是这个范式最完整的工业实现。但社区讨论通常止步于"它有压缩功能"。本章从源码中的三个数字常量出发，揭示 Context Engineering 的真正工程权衡。

## 源码洞察 1：13,000 token 缓冲区 -- 为什么不是 5,000 或 30,000？

`autoCompact.ts:62-65` 定义了四个关键常量：

```typescript
// autoCompact.ts:62-65
export const AUTOCOMPACT_BUFFER_TOKENS = 13_000
export const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000
export const ERROR_THRESHOLD_BUFFER_TOKENS = 20_000
export const MANUAL_COMPACT_BUFFER_TOKENS = 3_000
```

`AUTOCOMPACT_BUFFER_TOKENS = 13_000` 的含义是：在 200K context window 中，当 token 使用量达到 `effectiveWindow - 13,000` 时触发自动压缩（`autoCompact.ts:72-91`）。这个数字编码了一个精确的权衡：

**为什么不是 5,000？** 5K 缓冲区意味着只有约 2.5% 的余量。一个工具调用的结果（比如 `Read` 读取一个大文件）可能消耗 3,000-8,000 tokens。如果在压缩触发和实际执行之间有一轮工具调用，5K 缓冲区很容易被突破，导致 API 返回 `prompt_too_long` 错误。

**为什么不是 30,000？** 30K 缓冲区（15% 余量）意味着每次压缩都过早发生，丢弃了可能仍然有用的对话历史。还要注意 `autoCompact.ts:29-30` 的注释：

```typescript
// autoCompact.ts:29-30
// Reserve this many tokens for output during compaction
// Based on p99.99 of compact summary output being 17,387 tokens.
const MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20_000
```

压缩操作本身需要预留 20K output tokens（基于 p99.99 统计数据）。`effectiveWindow = contextWindow - 20,000`。所以实际的自动压缩触发点是 `200,000 - 20,000 - 13,000 = 167,000` tokens -- 约占 context window 的 83.5%。

13K 是统计学和工程妥协的产物：足以覆盖单轮工具调用的尖峰，又不至于浪费太多可用上下文。

### 为什么不用百分比阈值？

社区建议经常是"达到 80% 就压缩"。但百分比忽略了一个关键事实：不同 context window 尺寸（200K vs 1M）下，同样的百分比对应完全不同的绝对空间。13K 是一个绝对值 -- 无论 context window 多大，一次大文件读取消耗的 tokens 是固定的。源码用 `getEffectiveContextWindowSize()` 动态计算（`autoCompact.ts:33-49`），但缓冲区本身是固定的绝对值。

### 为什么不在 API 调用前检查而非用固定阈值？

因为压缩不是免费的。`compactConversation()` 本身是一次 API 调用，消耗时间和金钱。每次 API 调用前都检查会增加延迟。固定阈值配合熔断器（`autoCompact.ts:67-70`，连续失败 3 次后停止重试）是更高效的策略。

## 源码洞察 2：CLAUDE.md 在哪里结束 -- 缓存边界的成本含义

`prompts.ts:105-115` 定义了一个关键边界标记：

```typescript
// prompts.ts:113-115
export const SYSTEM_PROMPT_DYNAMIC_BOUNDARY =
  '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'
```

注释（`prompts.ts:105-112`）明确说明：这个标记将 system prompt 分为两部分 -- **静态部分**（跨组织可缓存，scope: 'global'）和**动态部分**（每会话独特，不可缓存）。

`prompts.ts:560-576` 展示了完整的组装逻辑：

```typescript
// prompts.ts:560-576
return [
  // --- Static content (cacheable) ---
  getSimpleIntroSection(outputStyleConfig),
  getSimpleSystemSection(),
  getSimpleDoingTasksSection(),
  getActionsSection(),
  getUsingYourToolsSection(enabledTools),
  getSimpleToneAndStyleSection(),
  getOutputEfficiencySection(),
  // === BOUNDARY MARKER - DO NOT MOVE OR REMOVE ===
  ...(shouldUseGlobalCacheScope() ? [SYSTEM_PROMPT_DYNAMIC_BOUNDARY] : []),
  // --- Dynamic content (registry-managed) ---
  ...resolvedDynamicSections,
].filter(s => s !== null)
```

**CLAUDE.md 内容在哪里？** 它是通过 `loadMemoryPrompt()` 加载的（`prompts.ts:495`），属于 `dynamicSections` 数组 -- 也就是说，CLAUDE.md 在边界标记之后，位于**动态/非缓存区域**。

这意味着：每个用户的 CLAUDE.md 内容不同，所以它不能被全局缓存。修改 CLAUDE.md 不会破坏静态部分的 prompt cache。但同时，CLAUDE.md 的每一个 token 在每次 API 调用中都是"冷"token -- 全价计费。

### systemPromptSection vs DANGEROUS_uncachedSystemPromptSection

`systemPromptSections.ts:20-38` 定义了两种 section 类型：

```typescript
// systemPromptSections.ts:20-25
export function systemPromptSection(
  name: string, compute: ComputeFn,
): SystemPromptSection {
  return { name, compute, cacheBreak: false }
}

// systemPromptSections.ts:32-38
export function DANGEROUS_uncachedSystemPromptSection(
  name: string, compute: ComputeFn, _reason: string,
): SystemPromptSection {
  return { name, compute, cacheBreak: true }
}
```

`systemPromptSection` 计算一次后缓存直到 `/clear` 或 `/compact`。`DANGEROUS_uncachedSystemPromptSection` 每轮重新计算 -- 会破坏 prompt cache。源码中只有一处使用了 `DANGEROUS_`：MCP server instructions（`prompts.ts:513-519`），因为 MCP servers 在 turns 之间可能连接/断开。

### 为什么不把所有内容都放在静态区？

因为 prompt cache 是按前缀匹配的。一旦前缀中有任何动态字节变化（用户名、环境路径、日期），整个缓存就会 miss。把稳定内容集中在前缀，动态内容推到后面，是最大化缓存命中率的策略。

## 源码洞察 3：180K / 40K -- microcompact 的安全边距设计

`apiMicrocompact.ts:14-17` 定义了 API-based microcompact 的阈值：

```typescript
// apiMicrocompact.ts:15-16
const DEFAULT_MAX_INPUT_TOKENS = 180_000  // Typical warning threshold
const DEFAULT_TARGET_INPUT_TOKENS = 40_000 // Keep last 40k tokens like client-side
```

180K 触发 / 40K 保留，在 200K context window 中意味着：

- **触发时机**：当 input tokens 达到 180K（占 context 的 90%）。这比自动压缩的 ~167K 更晚 -- microcompact 是 autocompact 的补充手段，不是替代。
- **保留目标**：只保留最近的 40K tokens（20%），清除 `180K - 40K = 140K` tokens 的工具结果。
- **安全边距**：200K - 180K = 20K tokens -- 这恰好等于 `MAX_OUTPUT_TOKENS_FOR_SUMMARY`（`autoCompact.ts:30`）。这不是巧合：系统需要为模型的输出预留空间。

`apiMicrocompact.ts:19-33` 定义了两类可清除的工具：

```typescript
// apiMicrocompact.ts:19-26
const TOOLS_CLEARABLE_RESULTS = [
  ...SHELL_TOOL_NAMES, GLOB_TOOL_NAME, GREP_TOOL_NAME,
  FILE_READ_TOOL_NAME, WEB_FETCH_TOOL_NAME, WEB_SEARCH_TOOL_NAME,
]

const TOOLS_CLEARABLE_USES = [
  FILE_EDIT_TOOL_NAME, FILE_WRITE_TOOL_NAME, NOTEBOOK_EDIT_TOOL_NAME,
]
```

读操作（Bash/Read/Grep/Glob）的结果可以安全清除 -- 模型已经从中提取了信息。写操作（Edit/Write）的结果也可以清除 -- 写入操作的返回值通常只是确认信息。

### 为什么不用 50K 保留目标？

40K 大约等于"最近 2-3 轮完整交互"。更多保留意味着清除更少的历史，每次 microcompact 的效果更弱，可能需要更频繁地触发。40K 在"保留足够上下文让模型继续工作"和"清除足够空间让对话继续进行"之间取得平衡。

## 跨章节关联：三个系统如何协同实现 Context Engineering

Context Engineering 在 Claude Code 中不是单一模块，而是至少三个子系统的协同：

**1. compact.ts（c06 章节）-- 对话级压缩**
当 token 使用量超过 `effectiveWindow - 13,000` 时，`autoCompactIfNeeded()` 触发。它先尝试 session memory compaction（轻量级，复用已有摘要），失败后回退到完整的 `compactConversation()`（fork 一个子 agent 来生成摘要）。熔断器确保连续失败不超过 3 次（`autoCompact.ts:70`，注释引用 BQ 数据：1,279 个 session 曾有 50+ 次连续失败，每天浪费约 250K 次 API 调用）。

**2. systemPromptSections.ts（c05 章节）-- 提示词级缓存管理**
缓存边界将 system prompt 分为静态前缀和动态后缀。静态前缀（7 个 section，约 5,000-8,000 tokens）在全局范围内被缓存。动态后缀（CLAUDE.md、MCP instructions、环境信息等）每次重新计算但大多数通过 `systemPromptSection` 的内部缓存避免重复计算。只有 MCP instructions 使用 `DANGEROUS_uncachedSystemPromptSection` -- 即便如此，源码也在通过 `isMcpInstructionsDeltaEnabled()` 将其迁移到增量 attachments 方式（`prompts.ts:510-511`），进一步减少 prompt cache 破坏。

**3. ToolSearchTool（c11 章节）-- 工具定义级延迟加载**
`ToolSearchTool/prompt.ts:62-108` 的 `isDeferredTool()` 函数决定哪些工具被延迟加载。MCP 工具默认全部延迟（`tool.isMcp === true`）。这直接减少了每次 API 调用中工具定义占用的 tokens -- 在 40+ 个 MCP 工具的场景中，可以节省数千 tokens。

三个系统的协同关系：ToolSearchTool 减少了每次调用的工具定义 tokens，systemPromptSections 减少了系统提示的缓存失效频率，compact 管理对话历史的总量。它们分别作用于 context window 的不同组成部分。

注意 `isDeferredTool()` 中有显式的豁免列表（`ToolSearchTool/prompt.ts:70-107`）：ToolSearchTool 自身永不延迟（否则无法加载其他工具），Agent tool 在 fork-subagent 模式下不延迟（第一轮就需要可用），Brief tool 在 KAIROS 模式下不延迟（它是主要通信通道）。这些豁免规则体现了一个核心原则：**延迟加载的边界由通信关键路径决定，而非简单的使用频率。**

同样重要的是，query.ts（p02 章节详述）中 autocompact 的执行时机也考虑了这三个系统的交互。`query.ts:440-447` 展示了 context collapse 在 autocompact **之前**运行 -- 如果 collapse 已经把 token 使用量降到阈值以下，autocompact 就不需要触发，从而保留更细粒度的上下文而非生成一个粗粒度摘要。这种顺序依赖是手写编排的产物，框架抽象无法自动发现。

```
Context Window 200K tokens
├── System Prompt (~8K)     ← systemPromptSections 管理缓存
├── Tool Definitions (~2-5K) ← ToolSearchTool 延迟加载减少占用
├── Conversation History     ← compact + microcompact 管理总量
└── Safety Buffer (13K)      ← autoCompact 的固定缓冲区
```

## 关键收获

- 13,000 token 缓冲区不是拍脑袋的数字：它是"单轮工具调用最大 token 消耗"的统计上界，配合 20K output 预留和 3 次连续失败熔断器，构成了一个基于生产数据（p99.99 = 17,387 tokens）的精确工程设计。
- CLAUDE.md 内容位于 prompt cache 边界之后的动态区域，这意味着它的每个 token 都是全价计费的 -- 保持 CLAUDE.md 精简不仅是最佳实践，更是直接的成本优化。
- 180K/40K microcompact 阈值与 autocompact 的 167K 阈值形成两层防线：autocompact 在 83.5% 触发（主动压缩），microcompact 在 90% 触发（清除工具结果），两者互补而非互斥。
- Context Engineering 在 Claude Code 中不是单一模块而是三个子系统的协同：compact（对话历史管理）、systemPromptSections（prompt cache 优化）、ToolSearchTool（工具定义延迟加载），分别作用于 context window 的不同组成部分。
- `DANGEROUS_uncachedSystemPromptSection` 的命名本身就是 Context Engineering 的文化产物：函数名中包含"DANGEROUS"是对"每轮重计算会破坏 prompt cache"这一成本后果的显式警告。
