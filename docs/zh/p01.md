# Context Engineering：2026 范式转移

> 为什么 80% 重要的是动态上下文，而非静态 prompt？

## 概述

2024 年的 AI 应用开发范式是"提示词工程"：写更清晰的指令、更好的角色设定、更精确的约束。这仍然重要，但在 2026 年，它已经不再是最重要的变量。

在实际生产 agent 中，决定输出质量的关键因素已经转移：从"你写了什么静态提示词"，转向"模型在每次调用时的 context window 里有什么"。这个范式转移有一个名字：**Context Engineering**。

Claude Code 的源码是这个范式最好的教材。它的 c06 章节描述的所有压缩策略——对话历史剪枝、工具结果摘要、递归压缩——都是 Context Engineering 的具体实现。只是在它们被写出来的时候，这个词可能还没有流行起来。

## 什么在 Context Window 里？

理解 Context Engineering，首先要理解现代 agent 的 context window 构成：

```
一次典型的 Claude Code API 调用的 context：

[系统提示]        ~15K tokens  (静态，每次相同)
[工具定义]         ~5K tokens  (静态，根据工具集变化)
[对话历史]        ~30K tokens  (动态，随对话增长)
[工具调用结果]    ~20K tokens  (动态，最近几轮的工具输出)
[检索到的文件]    ~25K tokens  (动态，当前任务相关的代码)
[记忆/摘要]        ~5K tokens  (半动态，压缩后的历史)
─────────────────────────────────────────────
总计：            ~100K tokens
```

注意比例：静态系统提示只占约 20%，其余 80% 是动态内容。**你能控制的最大影响力区间，在于那 80% 动态内容如何被管理。**

## 多轮退化：量化的性能损失

有基准测试数据表明，同一个任务在多轮对话（携带完整历史）与单次对话（只有任务描述）相比，模型性能下降约 **39%**。

为什么？三个原因：

**噪音积累**：早期轮次的工具输出、错误信息、中间状态会持续存在于 context 中。这些内容在当前轮次已经没有意义，却占用着注意力资源。

**注意力稀释**：Transformer 的注意力机制在长序列下表现退化——关键信息被淹没在大量相关性低的历史内容中。

**无关历史干扰**：第一轮讨论的内容（比如"帮我看下这个文件的结构"）与第十轮的任务（"现在重构认证模块"）几乎没有关系，但两者都在 context 里。

39% 不是一个小数字。对于一个 10 轮对话的 agent 任务，这意味着你在第 10 轮得到的答案，质量可能只有理想状态的 60%——不是因为模型变差了，而是因为 context 变脏了。

## Context Engineering 的核心策略

**激进剪枝（Aggressive Pruning）**

不要保留所有历史。Claude Code 的实现会在对话进行到一定长度后，主动删除超过 N 轮之前的内容：

```typescript
function pruneHistory(messages: Message[], maxTurns: number): Message[] {
  if (messages.length <= maxTurns * 2) return messages;

  // 保留系统消息、最近 N 轮、以及被标记为重要的内容
  const systemMessages = messages.filter(m => m.role === 'system');
  const recentMessages = messages.slice(-(maxTurns * 2));
  const pinnedMessages = messages.filter(m => m.pinned);

  return deduplicate([...systemMessages, ...pinnedMessages, ...recentMessages]);
}
```

**定期重锚（Periodic Re-anchoring）**

每隔几轮，注入一条"当前状态摘要"，重新建立任务上下文：

```
[重锚消息]: "当前进度：已完成文件读取和依赖分析。
             当前任务：重构 auth.ts 中的 JWT 验证逻辑。
             已知约束：不能修改对外 API 签名。"
```

这不是在重复历史——它是在重置"什么是重要的"这件事。

**选择性工具结果压缩**

工具结果是 context 污染的主要来源。一次 `Bash` 调用可能返回 5000 token 的文件内容，但下一轮的任务只需要其中 3 行。Claude Code 的压缩策略会将工具结果替换为摘要：

```
原始工具结果（3000 tokens）：
  [完整的 package.json 内容]

压缩后（50 tokens）：
  [已读取 package.json：项目名 my-app，依赖 React 18、Next.js 14，无类型错误]
```

## 这与 Claude Code 的关系

回到 c06：Claude Code 的"上下文压缩管道"现在有了理论框架。它的每个步骤都是 Context Engineering 的实践：

- 对话历史剪枝 → 激进剪枝
- 进度摘要注入 → 定期重锚
- 工具输出摘要 → 选择性压缩
- 递归压缩 → 防止 context window 耗尽的最后防线

Context Engineering 不是一个新工具，而是对一批已有实践的命名和系统化。

## 关键收获

- Context window 的 80% 是动态内容（对话历史、工具结果、检索文件等），这是影响模型输出质量的最大可控杠杆点。
- 多轮对话相比单次对话性能下降约 39%，根本原因是噪音积累、注意力稀释和无关历史干扰导致关键信息被淹没。
- Context Engineering 的三个核心策略：激进剪枝（删除旧轮次）、定期重锚（注入当前状态摘要）、选择性工具结果压缩（将大体量工具输出替换为精炼摘要）。
- Claude Code 的七策略压缩管道（c06 章节）是 Context Engineering 在生产环境中最完整的实现案例。
- 2026 的范式转移已经发生：从"写更好的静态 prompt"转向"工程化管理动态 context window 中的每一个 token"。
