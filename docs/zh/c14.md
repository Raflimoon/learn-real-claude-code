# 可观测性：遥测、成本与恢复

> 熔断器（最多 3 次连续失败）防止无限重试循环；11 步指数退避处理瞬时错误。

## 概述

一个在生产环境中稳定运行的 agent，需要在三个维度上做好准备：知道发生了什么（遥测）、知道花了多少（成本追踪）、以及知道失败后怎么办（错误恢复）。Claude Code 在这三个维度都有完整的工程实现，而不只是"出错了就重试"。

这章的内容技术密度较高，但值得仔细看。特别是错误恢复管道——它解决的问题（防止无限重试耗尽 context window）是几乎所有 agent 开发者都会踩的坑，而且通常要在踩坑之后才意识到需要这种保护。

## 双遥测通道：StatsD + Sentry

Claude Code 同时运行两条独立的遥测管道，各司其职：

**StatsD（指标）**：用于性能和业务指标的时序数据。

```typescript
// 典型的指标上报
statsd.increment('tool.call.count', { tool: 'Bash', status: 'success' });
statsd.timing('api.response.time', responseMs, { model: 'claude-3-5-sonnet' });
statsd.gauge('context.tokens.used', currentTokens, { session: sessionId });
```

StatsD 是"我想知道系统总体状态"的工具：每分钟多少次 API 调用、平均响应时间、token 消耗趋势。

**Sentry（错误）**：用于异常捕获和堆栈追踪。

```typescript
try {
  await executeToolCall(tool, params);
} catch (err) {
  Sentry.captureException(err, {
    extra: { tool: tool.name, params, sessionId },
    tags: { errorType: classifyError(err) }
  });
  throw err;
}
```

Sentry 是"我想知道哪里出错了、为什么"的工具：具体的异常类型、完整的调用栈、出错时的上下文状态。

两者分工明确，不互相替代。

## cost-tracker.ts：细粒度成本追踪

Claude Code 对 API 成本的追踪精细到每次调用：

```typescript
interface CostEntry {
  sessionId: string;
  model: string;
  inputTokens: number;
  outputTokens: number;
  cacheReadTokens: number;   // prompt cache 命中的 token
  cacheWriteTokens: number;  // 写入 prompt cache 的 token
  costUsd: number;
  timestamp: Date;
}
```

注意 `cacheReadTokens` 和 `cacheWriteTokens`——这不是普通的 input/output 分类，而是精确追踪了 prompt cache 的使用情况。Cache read 的单价远低于普通 input token（约 10% 的价格），追踪这个数字能准确反映 prompt cache 带来的实际节省。

`costHook.ts` 在每次 API 调用完成后触发，将成本数据写入追踪器，同时在终端右上角显示实时累计成本。

## 11 步指数退避：处理瞬时错误

API 调用失败并不总是"真正的失败"——网络抖动、速率限制、服务器过载都是可以自愈的瞬时状态。Claude Code 实现了一个 11 步指数退避策略：

```typescript
const RETRY_DELAYS_MS = [
  100, 200, 400, 800, 1600,
  3200, 6400, 12800, 25600, 51200, 102400
];
//   ↑                                ↑
// 第1次重试                         第11次重试（约102秒后）

async function callWithRetry(apiCall: () => Promise<Response>) {
  for (let attempt = 0; attempt <= 11; attempt++) {
    try {
      return await apiCall();
    } catch (err) {
      if (!isRetryable(err) || attempt === 11) throw err;
      await sleep(RETRY_DELAYS_MS[attempt] * (0.8 + Math.random() * 0.4)); // 加抖动
      // jitter 防止多个会话在同一时刻同步重试（thundering herd）
    }
  }
}
```

"加抖动"（jitter）这个细节值得注意：如果多个 Claude Code 实例同时遇到速率限制并在完全相同的时刻重试，会制造新一波的流量峰值。随机化重试间隔（±20%）能有效避免这种"雷群效应"（thundering herd）。

## 熔断器：防止 Context 耗尽

指数退避解决了瞬时错误，但无法处理"持续失败"的场景。如果 API 服务中断了 5 分钟，11 轮重试会消耗约 3 分钟时间，同时不断将错误信息塞入 context window：

```
[重试失败记录 1] → +500 tokens
[重试失败记录 2] → +500 tokens
...
[重试失败记录 11] → +500 tokens
总计：5500+ tokens 的错误噪音 → context 被垃圾填满
```

熔断器（Circuit Breaker）是解决方案：

```typescript
class CircuitBreaker {
  private consecutiveFailures = 0;
  private readonly maxFailures = 3;
  private state: 'CLOSED' | 'OPEN' = 'CLOSED';

  async execute<T>(fn: () => Promise<T>): Promise<T> {
    if (this.state === 'OPEN') {
      throw new CircuitOpenError('服务暂时不可用，已停止重试');
    }

    try {
      const result = await fn();
      this.consecutiveFailures = 0;  // 成功则重置计数器
      return result;
    } catch (err) {
      this.consecutiveFailures++;
      if (this.consecutiveFailures >= this.maxFailures) {
        this.state = 'OPEN';  // 断路！
      }
      throw err;
    }
  }
}
```

3 次连续失败后，熔断器进入 OPEN 状态，后续调用直接抛出 `CircuitOpenError`，不再尝试 API 调用，不再产生新的错误 token。

## 遥测、成本、恢复的协同

这三个系统不是孤立的：

```
API 调用失败
    ↓
指数退避（11步）→ StatsD: retry.count++
    ↓（3次失败）
熔断器触发    → StatsD: circuit.open++
                Sentry: captureException(CircuitOpenError)
    ↓
用户提示：服务暂时不可用 → cost-tracker: 本次调用成本 $0（未完成）
```

## 关键收获

- 双遥测通道各司其职：StatsD 处理性能和业务的时序指标（API 调用频率、响应时间、token 趋势），Sentry 处理异常捕获和堆栈追踪（具体出错原因和上下文）。
- `cost-tracker.ts` 精细追踪每次 API 调用的 cache read/write token 数量，准确反映 prompt cache 带来的实际成本节省（cache read 单价约为普通 input 的 10%）。
- 11 步指数退避策略加随机抖动（jitter ±20%），防止多实例同步重试导致的雷群效应，最长单次等待约 102 秒。
- 熔断器在 3 次连续失败后进入 OPEN 状态并断路，阻止后续重试不断将错误信息塞入 context window 导致垃圾填满。
- 三个系统形成闭环协同：遥测记录系统发生了什么，成本追踪记录花了多少钱，熔断器在持续故障时决定何时停止尝试。
